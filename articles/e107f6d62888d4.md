# そもそも git worktree とは
`git worktree` を使えば、1つのリポジトリに紐づく複数の作業ツリーを管理できるようになります。

```shell
development/
  ├── my-project/         ← メインの作業ツリー（feature-A をチェックアウト中）
  │   ├── .git/
  │   ├── src/
  │   └── ...
  ├── my-project-hotfix/   ← 追加された作業ツリー（hotfix をチェックアウト中）
  │   ├── .git → ../my-project/.git  （実体はメインと共有）
  │   ├── src/
  │   └── ...
```

それぞれ作業ツリーは独立したディレクトリなので、Aの作業を `stash` や `commit` せずにそのままにして、Bの作業ができるようになります。


:::details git worktree のコマンド一覧
```shell
git worktree add [-f] [--detach] [--checkout] [--lock [--reason <string>]]
                 [--orphan] [(-b | -B) <new-branch>] <path> [<commit-ish>]
git worktree list [-v | --porcelain [-z]]
git worktree lock [--reason <string>] <worktree>
git worktree move <worktree> <new-path>
git worktree prune [-n] [-v] [--expire <expire>]
git worktree remove [-f] <worktree>
git worktree repair [<path>...]
git worktree unlock <worktree>
```
:::

実際に打つとしたら以下のような感じですね。
```shell
  # 新しいブランチ「hotfix」を作成して、../hotfix に作業ツリーを作る
  git worktree add ../hotfix

  # 既存のブランチ「feature-A」を ../feature-a に展開する
  git worktree add ../feature-a feature-A
```

# worktrunk はこれを扱いやすい & 便利機能もあるのでお気に入り

冒頭でも書きましたが、この `git worktree` が長かったり、worktree 移動後にあれこれやりたいとかを思うと、ちょっと面倒でした。そこで worktrunk を使ってみたところ色々と便利だったので、worktrunk のお気に入り機能を紹介していきます！


## worktree の 作成や移動が簡単

`git worktree add hotfix` とすると、今いるフォルダの中に `hotfix` フォルダが作成されます。
ぼくはフォルダの中に worktree が作られると、どれが worktree のフォルダで どれが 普段使っているフォルダか分かりづらいとか 新しく作られた worktree も .gitignore に含めなきゃいけないとか、横にあるフォルダも普段と変わってしまうのが嫌なので、同じ階層に作られている方が好きです。

`git worktree add ../hotfix` とすると、同じ階層に作業ツリーが作成されます。

これを `worktrunk` であれば、 `wt switch hotfix` で実現できます。
しかも、 `zenn.hotfix-3` のような `リポジトリ名.worktree名` となるので、同じ階層でもわかりやすくて便利です。
横にあれば、main の worktree と同じように横にあるフォルダも参照ができます。

もうちょっと実運用的に使うオプションを含めて、 git worktree と wt のコマンドの比較をすると、`git worktree add -b hotfix ../repo.hotfix && cd ../repo.hotfix` だったものが `wt switch -c hotfix` でよくなったのです。

worktree の削除も `cd ../repo && git worktree remove ../repo.feat && git branch -d feat` が `wt remove` で実現できます。

この辺りはエイリアスでも実現できるかもですが、デフォルトで用意されている worktrunk 便利ですね。

## hooks で worktree 移動・作成・削除後の コマンドを設定できる

worktree で作業用のフォルダを作ったら、`.gitignore` にあるファイル群を生成やシンボリックリンクを貼ったりすることが多いと思います。
`npm install` や `bundle install` をしたり、`.env` に シンボリックリンクを貼ったりですね。

worktrunk の hooks 機能は、worktree ごとに毎回行う作業を事前に config に書いておけば、特定のイベントを hook してやってくれます。

hooksのイベント一覧は [こちら](https://worktrunk.dev/hook/#hook-types) から見ていただけます。

- switch（worktreeの切り替え時）
- start（worktreeの作成時）
- commit（commit時）
- merge（merge時）
- remove（remove時）

これらの pre- と post- が用意されています。

便利な使い方としては、

1. pre-start で `npm install` や `.env` にシンボリックリンクを貼り
2. `.env.local` に 起動したい port 番号を追記する
  - `echo 'PORT={{ branch | hash_port }}' > .env.local`
3. `npm run dev` で立ち上げ

ですね。

僕がよく使っているのを具体的に紹介すると `.config/wt.toml` に

```toml
[[post-start]]
  symlink-env = "ln -sfn {{ primary_worktree_path }}/.env .env"

[[post-start]]
  pnpm-install = "pnpm install --frozen-lockfile"
```

と書いていたりします。

どのブランチがどんな Status で どの程度差分があって、そのworktree 内容の Summary もAIが用意してくれて、 CI 進捗状況/結果もわかります。

Summary や CI 列は `wt list --full` で表示されます。
Summary は LLM によるブランチの変更要約で、diff が変わるまでキャッシュされます。

Status 列の主な記号はこんな感じです。

| 記号 | 意味 |
|---|---|
| `!` | 未ステージの変更あり |
| `?` | 未追跡ファイルあり |
| `✗` | mainにマージするとコンフリクト |
| `⊂` | mainに取り込み済み |
| `↑` `↓` | mainより先行 / 遅れ |
| `↕` | mainと分岐 |
| `⇡` `⇣` | リモートより先行 / 遅れ |

詳しくは公式ドキュメントを参照してください。

## PR を別 worktree でサッと確認できる

`wt switch pr:1234` のように `pr:` の後にプルリクエストの番号を書くと、その PR のブランチを Fetch した Worktree が作成されます。

普通に `gh pr checkout 1234` すると今の worktree が上書きされるので、作業中の変更を stash したり commit したりする必要があります。`wt switch pr:1234` なら別 worktree で開くので、今の作業はそのまま。PR のレビューや動作確認が終わったら `wt remove` で片付けるだけです。

フォークからの PR の場合も、参照を自動で fetch して `pushRemote` をフォーク URL に設定してくれます。

GitHub のブラウザでコードレビューが大変な時、ローカルにそのプルリクエストを持ってくることがあります。そんな時に非常に便利ですね。


## おわりに

普段は 以下のようなエイリアス と .config/wt.toml を リポジトリごとに設定しています。
```sh
alias ws='wt switch'
alias wsc='ws --create -x claude'
```

`-x claude` を使うと、Worktree が作られたら、そのターミナルでさらに claude を起動した状態になって便利なんですよね。

git worktree 管理CLI はいくつかあると思いますが、worktrunk イイぞっていう紹介でした。

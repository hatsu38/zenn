## はじめに
個人で運用しているRailsのアプリケーションがあるのですが、そのRailsで、CPUもメモリも問題ないのにサーバーが落ちてしまうことがありました。
原因は、コスト削減のために非同期Jobの[Solid Queue](https://github.com/rails/solid_queue) をアプリケーションサーバー上で動かしており、その非同期Job から OpenAI の APIを投げてその I/O 待ちが非常に長くなって、利用できるスレッドがなくなったのだと思われます。

実際、Solid Queue を別のサーバーに分けたら、問題なく動くようになりました。

今回、問題の原因が掴みづらかったり、事前に検知ができなかったのは、CloudWatch にも NewRelic にも I/O や スレッド利用数のモニタリングをしていなかったためです。

というわけで、今回はそれの導入と、その時に便利だった Puma::Plugin とかの話を書いていきます。

最終的に出来上がったメトリクスは、こんな感じです。

![puma-stats](https://static.zenn.studio/user-upload/e2cd914c2236-20260429.png)

ちなみに、この記事のコードは Ruby 4.0.0 / Puma 8.0.1 で確認しています。Puma のバージョンによって `Puma.stats_hash` で取れる値の名前や種類が変わることがあります。実際、最近出た Puma 8系と7系では取得できる値が違ったりします。

## Puma の Stats を確認する

Puma を使っているならば、 `Puma.stats` や `Puma.stats_hash` でさまざまな値を取得できます。

```ruby
[1] pry(Metrics::PumaCollector)> Puma.stats_hash
=> {started_at: "2026-04-29T03:42:54Z",
 backlog: 3,
 running: 3,
 pool_capacity: 3,
 busy_threads: 0,
 io_threads: 0,
 backlog_max: 0,
 max_threads: 3,
 requests_count: 125,
 reactor_max: 9,
 versions: {puma: "8.0.1", ruby: {engine: "ruby", version: "4.0.0", patchlevel: 0}}}
```

https://puma.io/puma/file.stats.html 
詳しくはこちらにも書いてありますが、簡単に紹介していきます。

1. `started_at`: そのままですが、Pumaが起動した時刻ですね
2. `backlog`: 利用可能なスレッドが利用可能になるのを待っているリクエストの数です。この値が頻繁に0を超える場合は、より多くの処理能力が必要です。
3. `running`: 生成されているスレッドの数
4. `pool_capacity`: Pumaがすぐにリクエストを受け取って開始できるスレッドの数
    - `pool_capacity == max_threads` → 全スレッドが暇している
    - `pool_capacity > 0` → まだ余裕あり。新規リクエストは即座にスレッドに渡される
    - `pool_capacity == 0` → 全枠埋まっている。新規リクエストは backlog に積まれる
5. `busy_threads`: 処理中のスレッド数 + backlog
      | 状況 | 処理中 | backlog | busy_threads |
      | --- | --- | --- | --- |
      | 余裕 | 1 | 0 | 1 |
      | ぴったり | 3 | 0 | 3 |
      | あふれ始め | 3 | 2 | 5 |
      | パンク | 3 | 50 | 53 |
6. `max_threads`: ワーカーの最大所有可能スレッド数
7. `requests_count`: 起動してから処理したリクエストの数
8. `versions`: Puma や Ruby のバージョンが載っています。


これらの値を使って、どのくらいスレッドに余裕があるかをログやNewRelicに送って、モニタリングを始めました。


## Puma の Stats を送信する

Puma のプラグインを作って、Puma が起動されたら、別スレッドで定期的に Puma の状態を観測して Log に出力し、NewRelic に送信する ことをしました。

もう少し具体的に書いていきます。

### Puma のプラグインを作る
自分でスレッドを作って実現することもできますが、せっかく公式が用意してくれているものがあるので、プラグインを使ってみました。

```ruby
  # lib/puma/plugin/metrics.rb
  require "puma/plugin"

  Puma::Plugin.create do
    def start(_)
      # Puma が起動完了したら、ここが呼ばれる
    end
  end   
```

プラグインのテンプレートは上記のような感じです。プラグインの説明は[こちら](https://github.com/puma/puma/blob/main/docs/plugins.md)にもあります。

このプラグインを作ったら、puma.rbに `plugin :metrics` と書いて、読み込ませます。

`start` は、サーバーが起動した時に実行されるフックです。

`start` の 中で `in_background do ~ end` の Blockを書いていきます。

```ruby
Puma::Plugin.create do
  def start(_)
    in_background do
      loop do
        sleep 30
        Metrics::PumaCollector.publish
        Metrics::SolidQueueCollector.publish
      end
    end
  end
end
```

`in_background` に書いた Block は、 puma だと こちらにコードが書いてあります。
https://github.com/puma/puma/blob/cee7e613c0d6e072b1ae9993c6dec63cbf259ec3/lib/puma/plugin.rb#L60-L72


```ruby
  def add_background(blk)
    @background << blk
  end

  def fire_background
    @background.each_with_index do |b, i|
      Thread.new do
        Puma.set_thread_name "plgn bg #{i}"
        b.call
      end
    end
  end
```

つまり、 Thread を作って、Thread に名前をつけて、 Block のコードを実行するというものです。

Block の中では、 30秒ごとに `Metrics::PumaCollector.publish` を呼び出しています。 `PumaCollector` は自作の Puma の stats を送信するクラスです。

### PumaCollector で puma の情報を送信する

上の方で書いたような `Puma.stats_hash` の値を取り出して、`Metrics::Publisher.gauge_batch` に渡しているだけのシンプルなクラスです。`PoolCapacity` と `BacklogMax` の2つを追っておけば、Pumaが詰まりかけているか / 詰まったかは大体分かります。


:::details Metrics::PumaCollector の中身
```ruby
class Metrics::PumaCollector
  def self.publish
    stats = Puma.stats_hash
    pool_capacity = stats[:pool_capacity].to_i
    max_threads = stats[:max_threads].to_i

    metrics = {
      "Custom/Puma/PoolCapacity" => pool_capacity,
      "Custom/Puma/Backlog" => stats[:backlog].to_i,
      "Custom/Puma/BacklogMax" => stats[:backlog_max].to_i,
      "Custom/Puma/Running" => stats[:running].to_i,
      "Custom/Puma/BusyThreads" => stats[:busy_threads].to_i,
      "Custom/Puma/ReactorMax" => stats[:reactor_max].to_i,
      "Custom/Puma/MaxThreads" => max_threads,
      "Custom/Puma/RequestsCount" => stats[:requests_count].to_i,
      "Custom/Puma/PoolUsageRatio" => 1.0 - (pool_capacity.to_f / max_threads)
    }
    metrics["Custom/Puma/Workers"] = stats[:workers] if stats[:workers]

    Metrics::Publisher.gauge_batch(metrics)
  end
end
```
:::

具体的にはこんな情報を送っています。
| メトリクス名 | 意味 |
| --- | --- |
| `Custom/Puma/PoolCapacity` | 即座に新規リクエストを受けられるスレッド枠の数 |
| `Custom/Puma/Backlog` | 待ち行列に積まれているリクエスト数 (瞬間値) |
| `Custom/Puma/BacklogMax` | 直近期間の `Backlog` の最大値 |
| `Custom/Puma/Running` | 起動済みスレッドの総数 (idle 含む) |
| `Custom/Puma/BusyThreads` | 処理中のスレッド数 + backlog (= 抱えている仕事の総量) |
| `Custom/Puma/ReactorMax` | reactor が直近期間に同時に保持していたリクエスト数の最大値 |
| `Custom/Puma/MaxThreads` | スレッド数の上限 |
| `Custom/Puma/RequestsCount` | 起動以来の累計処理リクエスト数 |
| `Custom/Puma/PoolUsageRatio` | `1 - PoolCapacity / MaxThreads` で計算する 0〜1 のプール使用率 |
| `Custom/Puma/Workers` | worker プロセス数 (cluster mode 時のみ) |


`gauge_batch` で受け取ったメトリクスを、NewRelic Custom Metrics と 1行JSON のログの両方に流します。

:::details Metrics::Publisher の中身
```ruby
class Metrics::Publisher
  def self.gauge(name, value)
    NewRelic::Agent.record_metric(name, value)
  end

  def self.gauge_batch(metrics)
    Rails.logger.info { metrics.merge(event: "metrics").to_json }
    metrics.each { |name, value| gauge(name, value) }
  end
end
```
:::

## 集めた情報を NewRelic Dashboard に 表示させる

集めたメトリクスを NewRelic に送るだけだと、確認が大変です。
せっかくならダッシュボードに表示しておきます。

私が普段見ているダッシュボードのクエリを載せておきます。

### Puma スレッドプールの状態

`PoolCapacity`、`Backlog`、`Running` を時系列で並べたパネル。「いまサーバーがどのくらい余裕があるか」を見るのに使います。

```sql
SELECT average(newrelic.timeslice.value)
FROM Metric
WHERE metricTimesliceName IN (
  'Custom/Puma/PoolCapacity',
  'Custom/Puma/Backlog',
  'Custom/Puma/Running'
)
SINCE 1 day ago
TIMESERIES
FACET metricTimesliceName
```
![](https://static.zenn.studio/user-upload/f37d0f82b94a-20260429.png)

`PoolCapacity` が 0 に張り付き、`Backlog` が積み上がっていく瞬間が、詰まりが発生しているってことですね。
average にしていますが、 min で見るのは、次に書いています。

### 最小 PoolCapacity (1日)

直近24時間で、Pool に余裕がなくなった瞬間があったかを一発で見るパネル。

```sql
SELECT min(newrelic.timeslice.value) AS '最小 PoolCapacity'
FROM Metric
WHERE metricTimesliceName = 'Custom/Puma/PoolCapacity'
SINCE 24 hours ago
TIMESERIES AUTO
```

![](https://static.zenn.studio/user-upload/f12007fddf70-20260429.png)

`min` が 0 まで下がっている時間帯があれば、「あの時間にスパイクが来ていた」と即座に気付けます。これは NewRelic Alerts の Condition のベースにもしやすいクエリで、例えば `min < 1` が5分続いたら Slack に通知、といった設定にしておけば、コード側に閾値判定を持たせずに枯渇を検知できます。

### BusyThreads と MaxThreads の比較

「サーバーが今どれくらい仕事を抱えているか」を `MaxThreads` ラインと比較して見るパネル。

```sql
SELECT max(newrelic.timeslice.value)
FROM Metric
WHERE metricTimesliceName IN (
  'Custom/Puma/BusyThreads',
  'Custom/Puma/MaxThreads'
)
SINCE 1 day ago
TIMESERIES
FACET metricTimesliceName
```

![](https://static.zenn.studio/user-upload/1a5b01388224-20260429.png)

`MaxThreads` は設定値なので、ほぼ水平な基準ラインとして引かれます。一方 `BusyThreads` は「処理中スレッド数 + backlog」なので、サーバーが忙しい時は `MaxThreads` のラインを上に突き抜けていきますね。


## おわりに

スレッドの枯渇で困ったことがあったので、こんなモニタリングを始めてみましたが、 `Puma::Plugin` とか　`Puma.stats` とかを知ることができてとても良い機会でした。

NewRelic のダッシュボードも使う機会がちゃんと出てきてよかったなと。

実はこれ以外にも `gvl_metrics_middleware` gem も入れているのですが、その話はまた長くなりそうなので、別の記事に書けたらなと。

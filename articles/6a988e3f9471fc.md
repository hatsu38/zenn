Ruby のイメージが大きくなっているので、slimを使って減らすことにします。
ちなみに、`alpine` を使うか `slim` を使うかが候補にありましたが、さまざまな記事を見て slim にすることにしました。

alpineとslimの違いは、[Dockerイメージ alpine,slim,stretch,buster,jessie等の違いと使い分け](https://www.ted027.com/post/docker-debian-difference/)の記事を参考にしています。
[軽量Dockerイメージに安易にAlpineを使うのはやめたほうがいいという話](https://blog.inductor.me/entry/alpine-not-recommended)の記事も見て、軽量化したOSを使うほど軽量化する必要性もなかったので、Debianベースの状態で軽量化されている [slim](https://hub.docker.com/layers/library/ruby/4.0.0-slim/images/sha256-4cad849f6d6f691b3b1ad380ae3d4110bc89330219454f9f42deada5be53538a) を採用しました。

## slim を使うときは apt-get cleanも実行する

apt がダウンロードした `/var/cache/apt/archives/` 内のキャッシュされたdebファイルをすべて削除します。

apt-getの末尾に以下を追加します。
```Dockerfile
RUN apt-get update -qq && \
    apt-get install -y --no-install-recommends \
    build-essential \
    libpq-dev \
    libyaml-dev \
    libssl-dev \
    libvips-dev \
    postgresql-client \
    libvips42 \
    locales \
    curl \
    tzdata && \
    apt-get clean && \         # 追加
    rm -rf /var/lib/apt/lists/* # 追加
```

すると 1.22GB→1.20GB と、0.02GB 減りました。

## コンテナのホストユーザーをrootから変更する

> デフォルトでは、 USER が指定されていない場合、Dockerはrootユーザーとしてコマンドを実行するため、重大なセキュリティリスクが生じる可能性があります。
Dockerコンテナが侵害された場合、攻撃者はコンテナに割り当てられたすべてのリソースに対してホストレベルのルートアクセス権を持つことになります。 

https://www.docker.com/ja-jp/blog/understanding-the-docker-user-instruction/

つまり、コンテナ内のrootユーザーが作成したファイルは、ホストOSに共有した際に、ホストOS側の所有者もrootユーザーになります。

ということで、非root化は侵害後の権限を最小化する対策になるので、コンテナを実行するユーザを指定するようにします。


```Dockerfile
# 省略
RUN groupadd --system --gid 1000 rails && \
    useradd rails --uid 1000 --gid 1000 --create-home --shell /bin/bash # 追加
USER rails:rails # 追加

COPY --from=builder --chown=rails:rails /app/vendor/bundle /app/vendor/bundle # 変更
COPY --chown=rails:rails . /app # 変更

# assets:precompileを実行するためにRAILS_MASTER_KEYを環境変数にセット
ARG RAILS_MASTER_KEY
ENV RAILS_MASTER_KEY=${RAILS_MASTER_KEY}
# mission_control-jobs gemを本番で使うために、assets:precompileを実行する
RUN SECRET_KEY_BASE_DUMMY=1 ./bin/rails assets:precompile

EXPOSE 3000

CMD ["/app/bin/entrypoint"]
```


## 機密な値は ARG / ENV ではなく --mount=type=secret を使う

環境変数を ARGやENVを使って渡すことがあると思います。
しかし、秘匿な変数の場合はその手段を使うのは良くないです。

今は下記のように、RAILS_MASTER_KEYをBuild時に渡すようになってしまっています。
```Dockerfile
# assets:precompileを実行するためにRAILS_MASTER_KEYを環境変数にセット
ARG RAILS_MASTER_KEY
ENV RAILS_MASTER_KEY=${RAILS_MASTER_KEY}
# mission_control-jobs gemを本番で使うために、assets:precompileを実行する
RUN SECRET_KEY_BASE_DUMMY=1 ./bin/rails assets:precompile
```
これはよくないです。Dockerのサイトには以下のようなことが書かれています。

> While it is common to pass secrets to running processes through environment variables during local development, setting secrets in a Dockerfile using ENV or ARG is insecure because they persist in the final image. This rule reports violations where ENV and ARG keys indicate that they contain sensitive data.

> ローカル開発では、実行中のプロセスに環境変数を介してシークレットを渡すことが一般的ですが、Dockerfile 内で ENV または ARG を使用してシークレットを設定すると、最終イメージにシークレットが残るため安全ではありません。このルールは、ENV キーと ARG キーに機密データが含まれていることを示す違反を報告します。

参考: https://docs.docker.com/reference/build-checks/secrets-used-in-arg-or-env/

つまり、ARG などを使って機密な値を渡していると、最終イメージに機密な値が残ってしまいます。

これを防ぐためには、[--secret flag](https://docs.docker.com/build/building/secrets/) を使うといいです。
`--mount=type=secret` は、ビルド中のそのRUN命令の間だけコンテナ内に露出し、通常はイメージに残りません。

secretを使った場合は以下のようなDockerfileになります。
```diff Dockerfile
- # assets:precompileを実行するためにRAILS_MASTER_KEYを環境変数にセット
- ARG RAILS_MASTER_KEY
- ENV RAILS_MASTER_KEY=${RAILS_MASTER_KEY}
- # mission_control-jobs gemを本番で使うために、assets:precompileを実行する
- RUN SECRET_KEY_BASE_DUMMY=1 ./bin/rails assets:precompile

+ RUN --mount=type=secret,id=RAILS_MASTER_KEY,env=RAILS_MASTER_KEY \
+   ./bin/rails assets:precompile
```
のようにします。`docker build` 時に `docker build --secret id=RAILS_MASTER_KEY,env=RAILS_MASTER_KEY` のようにするとよいです。

:::details 最終的なDockerfile
```Dockerfile
FROM ruby:4.0.0-slim AS base

WORKDIR /app

ENV LANG=C.UTF-8 \
    RAILS_ENV=production \
    RACK_ENV=production \
    BUNDLE_PATH=vendor/bundle \
    BUNDLE_DEPLOYMENT=true \
    BUNDLE_WITHOUT=development:test

RUN gem update --system && gem install bundler:4.0.3

# builder
FROM base AS builder

RUN apt-get update -qq && \
    apt-get install -y --no-install-recommends \
    build-essential \
    libpq-dev \
    libyaml-dev \
    libssl-dev \
    libvips-dev \
    postgresql-client \
    libvips42 \
    locales \
    curl \
    tzdata && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

COPY Gemfile Gemfile.lock /app/
RUN bundle install --jobs=4 --retry=3

# main
FROM base AS main

RUN apt-get update -qq && \
    apt-get install -y --no-install-recommends \
    libpq5 \
    libvips42 \
    curl \
    tzdata \
    locales && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
```

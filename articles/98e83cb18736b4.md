こんにちは、hatsu です。個人で作っている ECS に載せている Rails のアプリケーションのデプロイ方法を変えて、少し速くしたのでその紹介です。

ぼくが関わっているプロダクトでは、Ruby on Rails + Nginx の構成で Amazon ECS に載せて運用しています。
以前はリリースのたびに GitHub Actions で Deploy のワークフローが走って Docker イメージのビルドからデプロイまでを一気通貫で実行していたのですが、9分弱くらいかかっていました。

![](https://storage.googleapis.com/zenn-user-upload/16f25bcd8aae-20260322.png)

このリリースが、ちょっと長いなと思っていて改善方法を悩んでいました。

そこで最近 **DockerのBuild と ECR の Push と Deploy を別の GitHub Actions ワークフローに分離** したところ、デプロイは ECSタスクの更新だけになり、DockerのBuild待ちの時間が発生しなくなり、リリースまでの時間の短縮ができました。

その変更について、具体的なことを書いていきます。

## TL;DR
- デプロイをしたタイミングで、Docker Build -> ECR に Push -> ECS Task の更新 をしていたものを変えた
- デプロイする前に main ブランチにマージされたらどんどん Build と Pushまではする
- デプロイをするときは、ECS Task の更新だけにする

という話です。(事前に Build をしておくって話なので、デプロイの時間は短くなったが、Build ~ Pushのトータルの時間が短くなったわけではないです 🙏)


## 変更前
元々の運用フローは、mainブランチにマージをすると、Draft の Tag が作成されます。
![](https://storage.googleapis.com/zenn-user-upload/eee36d308219-20260322.png)

その後も main ブランチにマージしていくと、この Tag に PRが溜まっていきます。
ちなみにこの Tag の運用は [release-drafter](https://github.com/release-drafter/release-drafter) を使っています。

リリースをしたくなったら、この Tag を Draft から Publish に変更します。
それを Hook にして 本番のリリースが走ります。
以下のようなものですね。
```yaml
name: 【Production】Deploy

on:
  release:
    types: [published]
```

リリースが走ると、
1. ECR に ログイン: 1秒
2. NginxとRails をそれぞれ Docker Build & タグの設定 (docker build -t): 3分
3. NginxとRails をそれぞれ ECR へ Docker Push: 1分弱
4. [aws-actions/amazon-ecs-deploy-task-definition](https://github.com/aws-actions/amazon-ecs-deploy-task-definition)を使って ECS Taskの更新: 4分くらい
5. Slackにデプロイ成功/失敗通知

おおよそ上記の時間配分で、Build から Push までで4分ほどかかっているのがボトルネックです。
一方で、mainブランチにマージされてからすぐにリリースするわけではなく、動作確認の時間があります。
であれば、その間に Build から Push まで済ませておけば、デプロイ時にはビルド待ちがなくなるよね、というのが今回の変更点です。

## 変更後
Build & Push と Deploy を2つのワークフローに分けました。

まず Build & Push のワークフローは、main ブランチへの push をトリガーにしています。
```yaml
name: 【Production】Build & Push

on:
  push:
    branches: [main]
```

mainブランチへの push が走ったら
1. ECR に ログイン: 1秒
2. NginxとRails をそれぞれ Docker Build & タグの設定 (docker build -t): 3分
3. NginxとRails をそれぞれ ECR へ Docker Push: 1分弱
4. SlackにBuild ~ Pushの成功/失敗通知

ここまでは毎回やります。で、Tag を Publish にしてリリースするとき(=デプロイ時)には、以下のステップだけを走らせます。

1. ECR のイメージ存在チェック(aws ecr describe-images): 3秒
2. [aws-actions/amazon-ecs-deploy-task-definition](https://github.com/aws-actions/amazon-ecs-deploy-task-definition)を使って ECS Taskの更新: 4分くらい

これだけです。9分かかっていたのが、4分程度で終わるようになりました。

![](https://storage.googleapis.com/zenn-user-upload/d6bb89e98601-20260322.png)


### かかるコスト

mainブランチにpushするたびに ECR に Pushしているので、不要なImageはECRに多く溜まると思います。さらに CI の時間も Buildする回数が多いので、増えますね。
ECRはライフサイクルポリシーを設定しています。特定の個数溜まったら古いImageから消えるので、コストも変わらないです。
CI の時間はかかってしまうコストですが、デプロイが早くなるのに比べたら許容できるコストとして、受け入れました。

## 終わりに
いやぁ9分弱かかっていたものが、4分程度と速くなりましたね。
3年前に作ったこのCDを見直していたら、こうやって速くすることができたわけですが、今回紹介したもの以外にも、そもそも `docker build -t ` とか使わないで、[docker/build-push-action](https://github.com/docker/build-push-action) を使うようにしたら速くなったのも実は嬉しいことでした。

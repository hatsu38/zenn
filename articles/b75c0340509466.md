## gvl_metrics_middleware で計測できるもの

`gvl_metrics_middleware` は、リクエスト1件ごとに次の4つの値を取れます。

| 名前 | 意味 |
| --- | --- |
| `total` | リクエストの総時間 = `running + io_wait + gvl_wait` |
| `running` | Ruby のコードを実行していた時間 (CPU 使用時間) |
| `io_wait` | GVL を解放したあと、I/O 待ちで止まっていた時間 (DB応答、外部API、ファイル I/O 等) |
| `gvl_wait` | GVL の取得を待たされていた時間 (= 別スレッドが GVL を保持中) |


ちなみにここで出てくる `GVL (Global VM Lock)` は、
ざっくり言うと、複数のスレッドにわたるRubyコードの並列実行を防止するものです。並行処理は可能です。
つまり、あるスレッドが Ruby のコードを動かしている間、他のスレッドは(別のメソッドや別のコードを動かしたくても) 順番待ちになります。この「待っていた時間」が `gvl_wait` です。

RubyKaigi2023のこの発表が、GVL の雰囲気を掴むのによかったです。

https://rubykaigi.org/2023/presentations/KnuX.html#day1

例として、200msかかっているリクエストをタイムラインぽく並べて、こんな感じのリクエストだとします。

```
1件のリクエスト処理 (合計 200ms) の中身:

[ Ruby実行 ][      I/O 待ち      ][ Ruby実行 ][ GVL待ち ][   Ruby実行   ]
    30ms          50ms                40ms       20ms          60ms

= running 130ms + io_wait 50ms + gvl_wait 20ms = total 200ms
```

スレッドが時間軸の中で「Ruby を実行する瞬間」「I/O 待ちで止まっている瞬間」「GVL を取れずに待たされている瞬間」を行ったり来たりしています。それらを足し合わせたのが `total` です。


### 内訳によって対処方針が変わる

「リクエストに200msかかった」っていう事実だけだと、何を直せばいいか分からないんですよね。でも、内訳が見えると対処方針がはっきりしてきます。

例えばこんな感じ。

**`running` が長い場合 (CPU bound)**

```
[              Ruby実行              ][I/O][Ruby実行]
              150ms                    20ms   30ms
```

  200ms のうち `running` が 150ms (75%)。Ruby のコード自体が重いってことになります。JSON のシリアライズが重いとか、テンプレートレンダリングが遅いとかですね。

**`io_wait` が長い場合 (I/O bound)**

```
[Ruby実行][          I/O 待ち          ][Ruby実行]
  20ms             150ms                  30ms
```

`io_wait` が 150ms (75%)。外部 API の応答が遅いか、回数が多すぎるか。あとは DB クエリが遅いとか、そんな話になります。

**`gvl_wait` が長い場合**

```
[ GVL待ち ][Ruby実行][ GVL待ち ][Ruby実行][ GVL待ち ]
   50ms      30ms       40ms     30ms      50ms
```

`gvl_wait` が 140ms (70%)。スレッドが多すぎてプロセス内で GVL を奪い合ってる可能性が高いです。Puma のスレッド数を減らすとか、プロセスを分けるとか。

同じ「遅い200ms」でも、原因が何かによってやることが違ってきます。

こうしたことを見て、I/O待ちが長すぎないか、最適なスレッド数は幾つかとかが考察できるようになってきます。


## 使い方

書こうと思いましたが、README に書いてあることで十分だし、難しくないので、基本省略します。Gemを入れて、以下のように書けば良いです。

```ruby
# config/initializers/gvl_metrics_middleware.rb
GvlMetricsMiddleware.configure do |config|
  config.sampling_rate = 0.1

  config.rack do |total, running, io_wait, gvl_wait|
    NewRelic::Agent.record_metric("Custom/Rack/GVL/total", total)
    NewRelic::Agent.record_metric("Custom/Rack/GVL/running", running)
    NewRelic::Agent.record_metric("Custom/Rack/GVL/io_wait", io_wait)
    NewRelic::Agent.record_metric("Custom/Rack/GVL/gvl_wait", gvl_wait)
  end
end
```

ぼくはもう少し改造して 割合を入れたり、別クラスに切り出すとかもしていたりします。

## NewRelic ダッシュボードに表示する

### 実数値の時系列

```sql
SELECT
  filter(average(newrelic.timeslice.value), WHERE metricTimesliceName = 'Custom/Rack/GVL/running')  AS 'CPU 実行中',
  filter(average(newrelic.timeslice.value), WHERE metricTimesliceName = 'Custom/Rack/GVL/io_wait')  AS 'I/O 待ち',
  filter(average(newrelic.timeslice.value), WHERE metricTimesliceName = 'Custom/Rack/GVL/gvl_wait') AS 'GVL 競合'
FROM Metric
SINCE 24 hours ago
TIMESERIES
```

![](https://static.zenn.studio/user-upload/6ded190820cf-20260429.png)


絶対値で「どこに何ナノ秒使っているか」が見えます。

### 比率の100%積み上げ

```sql
SELECT average(newrelic.timeslice.value)
FROM Metric
WHERE metricTimesliceName IN (
  'Custom/Rack/GVL/io_wait_ratio',
  'Custom/Rack/GVL/gvl_wait_ratio',
  'Custom/Rack/GVL/running_ratio'
)
SINCE 24 hours ago
TIMESERIES
FACET metricTimesliceName
```

![](https://static.zenn.studio/user-upload/748a181195dc-20260429.png)

こう見ると、CPU 比率がダントツで多いですね。

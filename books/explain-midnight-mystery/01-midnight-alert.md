## 1

午前2時14分、Slack が鳴った。

枕元のスマホが青白く光る。オンコール当番の通知音は、他のどの通知よりも嫌な音に聞こえるよう設定してある。効果は抜群だった。心臓に悪いという意味で。

『[alert] /articles のレスポンスタイムが 12 秒を超えています』

僕──蒼井湊、Web エンジニア 2 年目──は、ベッドから転がり出てノート PC を開いた。

僕らが運営しているのは、技術記事の投稿サービスだ。記事が 10 万件、コメントが 100 万件。決して巨大ではないが、深夜 2 時にユーザーがゼロになるほど小さくもない。ダッシュボードを開くと、記事一覧 API のレイテンシのグラフが、崖のように立ち上がっていた。

12.4 秒。

Web の世界で 12 秒は「遅い」ではない。「死んでいる」だ。

スロークエリログを開く。犯行現場には、一本の SQL が残されていた。長い。JOIN が 3 つ、サブクエリが 1 つ、ORDER BY。読める。読めるが──**なぜこれが遅いのかは、まったく分からない**。

僕はログを睨んだまま固まった。インデックスを足せばいいのか？ どのカラムに？ そもそもインデックスは使われているのか？ いないのか？ それを、どうやって確かめる？

震える指で、もうひとつの通知を開いた。オンコールのエスカレーション先。桐生さん。インフラもアプリも DB もひととおり潜ってきた、無口な先輩だ。

深夜に起こすことへの罪悪感より、12 秒のほうが重かった。通話ボタンを押す。

ワンコールで出た。

「……見てる」

起きていたらしい。桐生さんの背後で、かすかにキーボードの音がした。

「すみません、クエリは特定できたんですが、原因が分からなくて。インデックスを足そうかと」

「どこに」

「え……author_id、とか」

「根拠は」

沈黙。僕には、根拠がなかった。あるのは「インデックスを足せば速くなるらしい」という、どこかで読んだ記憶だけだ。

桐生さんは小さく息を吐いた。呆れたのではなく、切り替えた音だった。

「湊。推測でスキーマを触るな。まず容疑者を全員並べろ。**EXPLAIN を打て。この状況で嘘をつかないのは実行計画だけだ**」

「……EXPLAIN、ちゃんと読めたことがないんです。cost とか、数字がいっぱい出てくるやつですよね。あれ、何の数字なのか分からなくて」

正直に言った。深夜 2 時は人を正直にする。

また沈黙。ただし今度は短かった。

「よし。なら 15 分やる。読めるようにしてやる」

「今からですか？ 障害対応中に？」

「読めない捜査資料を睨んでても犯人は挙がらない。それに──」キーボードの音が止まった。「レプリカのグラフを見ろ。12 秒を叩いてるのは特定の 1 ページだけだ。CDN のキャッシュが延命してる。完全に死ぬまで、まだ猶予がある」

画面を見た。たしかに、エラー率はまだ跳ねていない。瀕死だが、死んではいない。

「サンドボックスに繋げ。本番と同じスキーマ、同じ規模のデータが入ってるやつだ。稽古は本番でやるな」

## 2

言われるがまま、手元の Docker で立ち上がるサンドボックス DB に psql で繋いだ。articles 10 万行、comments 100 万行。本番の縮図だ。

「まず、事件のクエリは忘れろ。JOIN も WHERE もない、世界一単純な SELECT から始める。打て」

```sql
EXPLAIN SELECT * FROM articles;
```

「打ちました。……えっ、一瞬で返ってきましたけど。articles って 10 万行ありますよ？」

「それが EXPLAIN の第一の性質だ。**EXPLAIN はクエリを実行しない**。パースして、プランナが実行計画を立てて、そこで止まる。データは 1 行も読まない。5 億行のテーブルに打っても一瞬だ」

「実行しない……じゃあ、この出力は何なんですか」

「**計画書**だ。旅に出る前の地図。『この道を通るつもりだ』という宣言であって、歩いた記録じゃない。だから本番 DB に打っても安全だ──これは覚えておけ。今夜、本番で使う」

![SQL はパース→プラン→実行の 3 段階を通る。EXPLAIN は「プラン」で止めて計画書だけを返す](/images/postgres-explain-internals/ch01/01-sql-three-phases.png)
*捜査資料: SQL は「パース → プラン → 実行」の 3 段階。EXPLAIN は 2 段目で止まり、現場（実行）には踏み込まない*

画面には、たった 1 行の答えが出ていた。

```
                            QUERY PLAN
------------------------------------------------------------------
 Seq Scan on articles  (cost=0.00..12181.00 rows=100000 width=852)
(1 row)
```

「読め」

「Seq Scan on articles……コスト、0 から 12181……rows は 10 万で、width が 852、です」

「読み上げただけだな。いいか、この 1 行には情報が 5 つ入っている。分解するぞ」

| 部分 | 意味 |
| --- | --- |
| `Seq Scan` | **ノードの種類**。Sequential Scan ─ テーブルを先頭から順に全部読む |
| `on articles` | **対象テーブル** |
| `cost=0.00..12181.00` | **プランナのコスト見積り**（`スタートアップ..トータル`） |
| `rows=100000` | このノードが返すと**予想される**行数 |
| `width=852` | 1 行あたりの**推定バイト数** |

「Seq Scan は分かるか」

「インデックスを使わずに、全部読むやつ……ですよね。インデックスがないから Seq Scan になってる、と」

「半分間違いだ」

「えっ」

「このクエリには WHERE がない。**絞り込む条件がそもそもないから、全行読む以外にやりようがない**。だったらテーブルを先頭から一本道で読むのが最速だ、とプランナが判断した。『仕方なく』じゃない。『それが最善だから』選ばれてる。インデックスがあっても、このクエリなら Seq Scan になる」

容疑者だと思っていた Seq Scan に、アリバイがあった。そういう感覚だった。

## 3

「次。cost=0.00..12181.00。数字が 2 つあるのに気づいたか」

「0.00 と 12181.00……範囲、ですか？」

「違う。**別々の 2 つの数字**だ。左がスタートアップコスト──*最初の 1 行*を返すまでの作業量。右がトータルコスト──*全行*を返し切るまでの作業量。Seq Scan は先頭ページを開けば即 1 行目を返せるから、スタートアップは 0.00 になる」

「なるほど……で、右の 12181 は。これ、ミリ秒ですか？」

「**時間じゃない**。これが一番多い誤解だ。秒でもミリ秒でもない。PostgreSQL が独自に決めた無次元の相対値──『ディスクから 1 ページを連続読みする作業』を 1.0 と置いた、ただの数字だ」

「ただの数字……そんな曖昧なものを頼りに捜査するんですか」

「曖昧？」

通話越しに、桐生さんが少し笑った気がした。

「湊。その 12181 は、**手計算で 1 の位まで再現できる**」

「……は？」

「プランナは魔法じゃない。単価表を持った見積り屋だ。公式ドキュメントに式が書いてある」

> 推定コストは（ディスクページ読み取り × `seq_page_cost`）+（スキャンした行 × `cpu_tuple_cost`）と計算されます。
> ─ [PostgreSQL 17 文書 14.1.1 EXPLAIN の基本](https://www.postgresql.jp/document/17/html/using-explain.html)

```
total_cost = seq_page_cost × ページ数 + cpu_tuple_cost × 行数
```

「単価は 2 つ。`seq_page_cost` は 1 ページを連続読みする単価で、デフォルト **1.0**。これが基準だ。`cpu_tuple_cost` は 1 行を CPU でさばく単価で、デフォルト **0.01**。ページ読みの 100 分の 1。信じなくていい。`SHOW` で見ろ」

```sql
SHOW seq_page_cost;    -- 1.0
SHOW cpu_tuple_cost;   -- 0.01
```

「あとは材料だ。articles が何ページに格納されているか。プランナが行数をいくつだと思っているか。どっちもシステムカタログの `pg_class` が持ってる」

```sql
SELECT
  reltuples::bigint AS planner_rows,
  relpages,
  pg_size_pretty(relpages * 8192::bigint) AS table_size
FROM pg_class
WHERE relname = 'articles';
```

```
 planner_rows | relpages | table_size
--------------+----------+------------
       100000 |    11181 | 87 MB
```

「PostgreSQL はテーブルを 8KB の**ページ**という単位で持ってる。articles は 11,181 ページ、約 87MB。プランナが思っている行数は 100,000。──さあ、式に入れろ。電卓を出せ」

![ヒープページ俯瞰と 1 ページ拡大。約 9 タプル × 11,181 ページ ≈ 100,000 行](/images/postgres-explain-internals/ch01/03-heap-page-tuple-structure.png)
*捜査資料: articles の現場見取り図。8KB のページが 11,181 枚並び、1 ページに約 9 行が詰まって計 10 万行*

僕は言われたとおり、深夜 2 時半に電卓を叩いた。

```
total_cost = 1.0 × 11181 + 0.01 × 100000
           = 11181 + 1000
           = 12181.00
```

12181.00。

EXPLAIN の出力を見返す。`cost=0.00..12181.00`。

「……合った。ぴったり合いました。1 の位まで」

鳥肌が立った。画面に出ていた意味不明の数字が、たった今、自分の電卓から出てきた数字になった。

「それがコストの正体だ。I/O の作業量 11,181 と、CPU の作業量 1,000 の足し算。難解な統計モデルなんて裏にいない。**出どころが分かる数字は、もう怖くない**」

![Seq Scan のコスト構造。I/O コスト (11,181) と CPU コスト (1,000) の足し算で 12,181.00](/images/postgres-explain-internals/ch01/04-seq-scan-cost-structure.png)
*捜査資料: cost=12181.00 の解剖図。I/O の 11,181 と CPU の 1,000 の単純な足し算*

「rows=100000 も pg_class の reltuples そのままなんですね。じゃあ width=852 は？」

「1 行の推定バイト数。可変長カラムの平均長を足し上げた値だ。`SELECT *` で body みたいな TEXT まで全部引きずり出すから太くなる。──width は今夜の事件でも効いてくるが、それは後の話だ」

## 4

「仕上げだ。式がまぐれじゃないことを、別のテーブルで確かめろ。comments。100 万行ある」

```sql
EXPLAIN SELECT * FROM comments;
SELECT relpages, reltuples::bigint FROM pg_class WHERE relname = 'comments';
```

`relpages=18182`、`reltuples=1000000`。式に入れる。

```
total_cost = 1.0 × 18182 + 0.01 × 1000000
           = 18182 + 10000
           = 28182.00
```

EXPLAIN の出力は `cost=0.00..28182.00`。また、ぴったり合った。

「合いました。……あれ、でも変ですね。行数は articles の 10 倍なのに、コストは 2.3 倍くらいにしかなってない」

「いい所に気づいた。なぜだと思う」

出力を見比べる。comments の width は 114。articles の 852 より、ずっと細い。

「……1 行が細いから、1 ページにたくさん詰まる。だからページ数が行数ほどは増えない。コストの大半は relpages × 1.0 の I/O だから──**行数より、行の太さがコストを決めてる**？」

「合格だ」

短い一言だったが、深夜 2 時台にもらう「合格」は、昼の会議室でもらうどんな賞賛より効いた。

## 5

「さて」桐生さんの声のトーンが変わった。稽古の終わりの合図だった。「本題に戻るぞ。本番の 12 秒のクエリだ」

「はい。EXPLAIN、本番に打ってきます。実行されないから安全、なんですよね」

「そうだ。ただし──」

キーボードの音が、また止まる。

「EXPLAIN が見せるのはあくまで*計画*だ。プランナが『12181 のはずだ』と見積もっても、現場で本当にその通りだったかは分からない。見積りと現実がズレたとき、その**ズレこそが犯人の足跡**になる」

「ズレ……どうやって測るんですか」

「`EXPLAIN ANALYZE`。計画を立てた上で、**実際にクエリを実行して**、予想と実測を並べて見せてくれる。actual time、実際の rows、loops。現行犯逮捕の道具だ」

「じゃあそれを本番に──」

「**待て**」

低い声だった。

「ANALYZE 付きは*本当に実行される*。SELECT ならまだいい。だが UPDATE や DELETE に付ければ、**本当に更新され、本当に消える**。EXPLAIN は安全、EXPLAIN ANALYZE は刃物だ。抜き方は次に教える」

時計は午前 2 時 47 分。ダッシュボードの上で、12 秒のグラフはまだ、静かに崖を描き続けていた。

犯人はまだ挙がっていない。だが僕の手の中には、初めて読める捜査資料があった。

（第2話「現行犯逮捕 ─ EXPLAIN ANALYZE」につづく）

## 今夜の捜査メモ

物語に出てきた事実の整理です。すべてサンプル DB で再現できます。

- **EXPLAIN はクエリを実行しない**。プランナの計画書だけを返す。本番 DB でも安全に打てる
- 出力 1 行 `Seq Scan on articles (cost=0.00..12181.00 rows=100000 width=852)` には **5 つの情報**がある: ノード種類 / 対象テーブル / cost（スタートアップ..トータル）/ 推定行数 / 推定行サイズ
- WHERE がないクエリで Seq Scan になるのは「インデックスがないから」ではなく「**全行読むなら一本道が最速だから**」
- cost は時間ではなく無次元の相対値。手計算で再現できる:
  `total_cost = seq_page_cost(1.0) × relpages + cpu_tuple_cost(0.01) × reltuples`
  articles: `1.0 × 11181 + 0.01 × 100000 = 12181.00`
- `rows` は `pg_class.reltuples` 由来の**推定値**。実測ではない
- I/O コストが支配的なテーブルでは、**行数より width（行の太さ）がコストを決める**
- `EXPLAIN ANALYZE` は**実際に実行される**。UPDATE / DELETE に付けるときは要注意（→ 第2話）

## 1

午前3時41分。

「Index Only Scan の話をする前に、確認だ。第 3 話で 1 行ヒットのクエリを撃ったとき、コストは 8.44 だった。内訳を分解するとこうなる」

```
0.42  (B-tree を根から葉まで降りる)
+ 4.0 (インデックス葉ページのランダム I/O)
+ 4.0 (ヒープページのランダム I/O)      ← ここに注目
+ 0.01 + 0.005 (CPU)
≒ 8.44
```

「後半の 4.0 が、インデックスで見つけたあと**ヒープへ実データを取りに飛ぶ**ランダム I/O だ。──では湊、このクエリの `SELECT *` を `SELECT id` に変えたら、何が起きると思う」

「返すカラムが減るだけで……プランは同じじゃないんですか？」

「撃て」

```sql
EXPLAIN ANALYZE SELECT id FROM articles WHERE id = '0b3f973e-20c5-4459-8010-2bc0882d1102';
```

```
 Index Only Scan using articles_pkey on articles  (cost=0.42..4.44 rows=1 width=16)
                                                  (actual time=... rows=1 loops=1)
   Index Cond: (id = '0b3f973e-20c5-4459-8010-2bc0882d1102'::uuid)
   Heap Fetches: 0
```

「……ノードの名前が変わった。Index *Only* Scan。コストが 8.44 から **4.44**──あっ、さっきの内訳の、ヒープの 4.0 が**まるごと消えてる**」

「そうだ。理由は単純で、インデックスタプルは『**キーの値** + ヒープへのポインタ』を持ってる。このクエリが返すべきものは id の値だけ。**答えはもう、インデックスの中にある**。ならヒープに飛ぶ必要がない──現場に一歩も立ち入らず、事件を解決する。完全犯罪だ」

「`Heap Fetches: 0` は、『ヒープに触った回数ゼロ』ってことですか」

「そうだ。そして本番のプランで見ただろう──**Heap Fetches に万単位の数字が付いた Index Only Scan** を。完全犯罪のはずが、何万回も現場に舞い戻ってる。なぜだと思う」

「……分からないです。答えがインデックスにあるなら、戻る理由がない」

「ある。**インデックスには決定的に欠けている情報**があるんだ。ここからが今夜いちばん深い話になる。PostgreSQL が行を*消さない*という話だ」

## 2

「湊。PostgreSQL で UPDATE が走ったとき、内部で何が起きてると思う」

「行が書き換わる、でしょう。普通に」

「**書き換わらない**。PostgreSQL は行を上書きしない。**古い行をそのまま残して、新しい行を別の場所に追記する**」

「え……じゃあ UPDATE のたびに、同じ id の行が 2 つに増えるんですか」

「一時的にはな。各行には 2 つの隠しメタ情報が刻まれてる。`xmin`──その行を**作った**トランザクションの ID。行の生年だ。`xmax`──その行を**消した**トランザクションの ID。享年だ。まだ生きてる行は xmax=0。UPDATE はこう動く」

```
UPDATE 前
  古い行: xmin=T50,  xmax=0      ← T50 に生まれ、まだ生きている

T100 が UPDATE すると
  古い行: xmin=T50,  xmax=T100   ← T100 が享年を刻む（物理的には残る）
  新しい行: xmin=T100, xmax=0    ← T100 が新しく産む
```

「なんでそんな回りくどいことを……」

「同時並行のためだ。お前が UPDATE してる*まさにその瞬間*、別のトランザクションが同じ行を SELECT してたらどうする。ロックで待たせるか？ PostgreSQL の答えは『**待たせない。古い行を見せる**』だ。各トランザクションは自分が始まった時点のスナップショットを見る。読み手と書き手が互いを待たない──この仕組みを **MVCC** と呼ぶ。多版型同時実行制御。そしてこの設計の代償が、**享年を刻まれただけで物理的には残り続ける行**──通称 **dead tuple**、死んだ行だ」

死んだ行。テーブルの中に、見えない死体が積まれていく絵が浮かんだ。

「dead tuple の話は、今夜もう一度、もっと嫌な形で出てくる。今は Index Only Scan に戻るぞ。──いいか、行が見えるかどうかの判定には xmin と xmax が要る。そしてこの 2 つは**ヒープタプルだけが持ってる**。インデックスタプルには無い」

「あっ……そうか。だからインデックスで『キー A のエントリが見つかった』としても、そのエントリが指してる行が**生きてるのか死んでるのか、インデックスだけじゃ分からない**」

「そのとおり。指した先が dead tuple かもしれない。未コミットの行かもしれない。確かめるには結局ヒープを開いて xmin/xmax を見るしかない。**本来、完全犯罪は不可能**なんだ」

「でも……現に `Heap Fetches: 0` が出てますよ。ヒープを開かずに答えてる」

「そこに、抜け道がある」

## 3

「PostgreSQL はヒープテーブルごとに、**visibility map**──可視性の地図──という小さな補助ファイルを持ってる。ヒープページ 1 枚につき 2 ビット。Index Only Scan が見るのは **all-visible ビット**だ。意味は『**このページの全行は、すべてのトランザクションから見える**』。死んだ行も、未コミットの行も、1 つも無いページだという保証だ」

「ページ単位の……保証書」

「そうだ。判定はこう流れる」

1. インデックスから条件に合うエントリを引く
2. エントリが指す**ヒープページの番号**を取る
3. そのページの **all-visible ビット**を見る
4. **1 なら**、ヒープを開かず「見える」と断定 → `Heap Fetches` は増えない
5. **0 なら**、ヒープを開いて xmin/xmax を確認 → `Heap Fetches += 1`

「サイズ感がミソだ。articles のヒープは 11,181 ページ、約 87MB。visibility map は 1 ページ 2 ビットだから、`11181 × 2bit ÷ 8` で**約 3KB**。87MB の現場検証を、3KB の地図の照会で済ませてる。だから Index Only Scan は速い」

「じゃあ、そのビットは誰が立てるんですか」

「いい質問だ。公式ドキュメントの言い回しをそのまま覚えろ──**ビットを立てられるのは VACUUM だけ。データ編集は必ずビットを倒す**。VACUUM が『このページはもう全行可視だ』と確認して 1 を立てる。INSERT / UPDATE / DELETE がそのページに触れた瞬間、0 に戻る。**立てる者と倒す者の、終わらない綱引き**だ」

## 4

言われて、僕は本番のプランをもう一度開いた。Index Only Scan の行。その下の `Heap Fetches:` の万単位の数字。

「……読めました。この Index Only Scan は、**名前だけ**なんだ。visibility map のビットがほとんど倒れてるから、結局ほぼ毎回ヒープに飛んでる。実態はただの Index Scan と変わらない」

「そうだ。『Index Only Scan が選ばれている』と『実機で速い』は**別の話**。Heap Fetches まで見て初めて判定できる。──さて湊、ここで捜査の質問だ。**ビットは誰が倒した？**」

「データを書いた奴……INSERT か UPDATE か DELETE。しかも Heap Fetches がこの規模ってことは、articles のページの大部分でビットが倒れてる。つまり──**最近、articles にとんでもない量の書き込みをした奴がいる**」

言いながら、心当たりが浮かんでいた。デプロイカレンダー。昨日の夕方。

「桐生さん。昨日 18 時に、記事の全件再集計バッチが走ってます。view_count の洗い替えで、articles を**ほぼ全行 UPDATE** してる……」

「ほう」

キーボードの音が、少しだけ速くなった。

「全行 UPDATE なら、visibility map は全滅。dead tuple も 10 万行ぶん積まれてる。そして──第 2 話で数えた典型 4 パターンを思い出せ。**大量更新の直後、統計情報が追いついていなければ、プランナの rows 推定も狂う**。visibility map、dead tuple、狂った推定。全部、同じ夜の同じバッチに繋がる」

「じゃあ、犯人はそのバッチ……！」

「まだだ」

桐生さんの声は、逸る僕を静かに押さえた。

「バッチは*容疑者*だ。逮捕には物証が要る。統計情報がいつ取られたか、autovacuum が走ったか、dead tuple が実際に何行あるか──全部、システムカタログに記録が残ってる。それは後で順に挙げていく。──その前に、プランにはまだ読んでない大物がいる。真ん中に鎮座してる **Sort** だ。ORDER BY created_at、LIMIT 20。並べ替えには代償が付く。work_mem という名の」

時計は午前4時2分。窓の外は、まだ夜だった。だが捜査線上には、初めて具体的な容疑者が浮かんでいた。

（第5話「並べ替えの代償 ─ Sort と work_mem」につづく）

## 今夜の捜査メモ

- `SELECT *` → `SELECT id` に変えるだけで **Index Scan → Index Only Scan** に変わり、コストからヒープのランダム I/O `4.0` が消える（8.44 → 4.44）
- PostgreSQL の UPDATE は上書きではなく**追記**: 古い行に `xmax`（享年）を刻んで残し、新しい行を追加する。これが **MVCC**。読み手と書き手が互いを待たない設計の代償が **dead tuple**
- 行の可視性判定に必要な `xmin` / `xmax` は**ヒープタプルだけ**が持つ。インデックスには無い → 本来はヒープ訪問が必須
- **visibility map** がその抜け道: ヒープページごとの all-visible ビット（テーブル 87MB に対して地図は約 3KB）。ビットが 1 のページはヒープを開かずに可視と断定できる
- `Heap Fetches: 0` が Index Only Scan の本領。**大きな値なら「名前だけ」**で、実態は Index Scan と変わらない
- ビットは **VACUUM だけが立て、INSERT / UPDATE / DELETE が倒す**。大量更新の直後は Index Only Scan が効かなくなる
- 補足: インデックスのキー以外のカラムも返したいなら `CREATE INDEX ... INCLUDE (col)` の**カバリングインデックス**で Index Only Scan 化できる（詰めすぎると肥大化に跳ね返る）

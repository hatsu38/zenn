## 1

午前3時16分。

レプリカから持ち帰った 30 行のプランを、僕はテキストエディタに貼って睨んでいた。推定と実測が 4 桁ズレた Nested Loop。異常な loops。足跡は見つかった。なのに、分からないことが増えていた。

「桐生さん。このプラン、comments を読むところが Seq Scan なんです。comments には article_id のインデックスがあるのに。これ、プランナのバグじゃ……」

「バグじゃない。**入札**だ」

「入札？」

「プランナは 1 つのテーブルを読むとき、候補を全員呼んで見積もりを出させる。Seq Scan、Index Scan、Bitmap Scan──全員がコストで入札して、**最安値が落札する**。インデックスがあっても、そいつの入札額が高ければ使われない。それだけの話だ」

「インデックスを使うほうが高くつくことなんて、あるんですか」

「ある。しかも珍しくない。──いいか、湊。それを体で覚えてないと、お前は今夜、無意味なインデックスを本番に張って『直った気になる』ことになる。サンドボックスに戻れ。**同じテーブルに、ヒット件数だけ変えた WHERE を 5 発撃つ**。階段を作るんだ」

## 2

「先に道具の名前だけ揃える。第 1 話でテーブルは 8KB の**ページ**の並びだと言った。テーブル本体のページを**ヒープページ**と呼ぶ。インデックスは別の場所にいて、持っているのは『キーの値と、**ヒープ上の行を指すポインタ**』だけだ。つまりインデックスで見つけても、実データを取るには**ポインタを辿ってヒープに飛ぶ**。二段構えだ」

「巻末の索引で『142 ページ』と引いてから、本文の 142 ページを開く、みたいな」

![ヒープページとインデックスページの対応。インデックスタプルの ptr が、対応するヒープタプルを指す](/images/cddb89f9abfaca/ch03/01-heap-index-pages-tuples.png)
*捜査資料: インデックスタプルは「キー + ヒープへのポインタ」だけ。実データはポインタの先のヒープにいる*

「そうだ。そして飛ぶという行為には値段が付く。ページを先頭から順に読む**連続 I/O** は単価 1.0──第 1 話の `seq_page_cost` だ。飛び飛びに読む**ランダム I/O** は単価 4.0、`random_page_cost`。HDD 時代、ヘッドの移動は高くついた。その名残の相場だ。──さあ、撃て。階段の一段目は 1 行ヒットだ」

```sql
EXPLAIN SELECT * FROM articles WHERE id = '0b3f973e-20c5-4459-8010-2bc0882d1102';
```

```
 Index Scan using articles_pkey on articles  (cost=0.42..8.44 rows=1 width=852)
   Index Cond: (id = '0b3f973e-20c5-4459-8010-2bc0882d1102'::uuid)
```

「Index Scan。コスト 8.44──第 1 話の Seq Scan は 12181 でしたから、圧倒的に安い」

![Index Scan の動き。インデックスから id の一致を引き、ポインタが指すヒープページにランダム I/O で飛んで 1 行取り出す](/images/cddb89f9abfaca/ch03/04-index-scan-pinpoint.png)
*捜査資料: Index Scan の急襲ルート。索引で居場所を特定し、ヒープの該当ページに一直線*

「1 行取るのに 11,181 ページ読む奴はいない。次、数十行ヒット」

```sql
EXPLAIN SELECT * FROM articles WHERE author_id = 1;
```

```
 Bitmap Heap Scan on articles  (cost=4.67..191.55 rows=49 width=852)
   Recheck Cond: (author_id = 1)
   ->  Bitmap Index Scan on index_articles_on_author_id  (cost=0.00..4.66 rows=49 width=0)
         Index Cond: (author_id = 1)
```

「……Index Scan じゃない。Bitmap、なんとかが出ました。2 段になってる」

「第 2 話の本番プランにもいた顔だ。動きはこうだ。下の **Bitmap Index Scan** がインデックスから該当行の*居場所*だけを先に全部集めて、『どのヒープページに該当行があるか』の 0/1 の表──**ビットマップ**──を作る。上の **Bitmap Heap Scan** がそれを**ページ番号順に並べ直してから**ヒープを読む」

「先に住所リストを作って、回る順番を最適化してから配達に行く……」

![Bitmap Heap Scan の 3 段階。① Bitmap Index Scan が候補のページ位置を集め、② ビットマップを作り、③ ページ順にヒープを読む](/images/cddb89f9abfaca/ch03/05-bitmap-heap-scan-three-steps.png)
*捜査資料: Bitmap の 3 段作戦。候補収集 → 0/1 の地図作成 → ページ番号順の一斉家宅捜索*

「そうだ。49 行を 1 行ずつ Index Scan で引けばランダム I/O が 49 発、`4.0 × 49` でざっと 196。ビットマップ経由なら並べ直しの前処理は掛かるが、ヒープをページ順にまとめ読みできて 191.55。**数点差で Bitmap が落札**した。──次。ヒットを 100 倍にしろ」

```sql
EXPLAIN SELECT * FROM articles WHERE author_id BETWEEN 1 AND 100;
```

```
 Bitmap Heap Scan on articles  (cost=72.46..8811.42 rows=4699 width=852)
   ...
```

「4,699 行ヒット。まだ Bitmap ですね。コストは 100 倍近くになったけど、Seq Scan の 12181 はまだ下回ってる」

「Bitmap は意外に広いレンジを守る。──では最後だ。75% ヒットさせろ」

```sql
EXPLAIN SELECT * FROM articles WHERE author_id BETWEEN 1 AND 1500;
EXPLAIN SELECT * FROM articles WHERE published = true;
```

```
 Seq Scan on articles  (cost=0.00..12681.00 rows=74647 width=852)
   Filter: ((author_id >= 1) AND (author_id <= 1500))

 Seq Scan on articles  (cost=0.00..12181.00 rows=74643 width=852)
   Filter: published
```

「Seq Scan に戻った……！ しかも 2 本目、コストが 12181.00。これ、第 1 話の **WHERE なしと同じ値**です」

「気づいたか。Seq Scan は WHERE があろうがなかろうが**全ページ読む**。Filter は読んだあとに 1 行ずつ『お前は該当か』と尋問するだけで、読む量は 1 ページも減らない。75% も取るならビットマップを作る前処理のほうが高くつく。だから最初から全部読む──それが最安の入札だ」

僕は 5 発の結果を表に並べた。階段が、はっきり見えた。

| クエリ | ヒット件数 | 落札プラン | total_cost |
|---|---:|---|---:|
| `WHERE id = 'uuid'` | 1 | Index Scan | 8.44 |
| `WHERE author_id = 1` | 49 | Bitmap Heap Scan | 191.55 |
| `WHERE author_id BETWEEN 1 AND 100` | 4,699 | Bitmap Heap Scan | 8,811.42 |
| `WHERE author_id BETWEEN 1 AND 1500` | 74,647 | Seq Scan | 12,681.00 |
| `WHERE published = true` | 74,643 | Seq Scan | 12,181.00 |

「**少なければ飛ぶ。中くらいならまとめ読み。多ければ全部読む**。そして 4 本目と 5 本目を見ろ──カラムが違ってもヒット件数が同じなら、選ばれるプランも同じだ。プラン選択を決めるのは『どのカラムか』じゃない。**『何行ヒットしそうか』**だ」

![プラン選択は選択率で決まる。実機データ点を選択率軸に並べると、Index Scan / Bitmap Heap Scan / Seq Scan の 3 ゾーンに分かれる](/images/cddb89f9abfaca/ch03/06-plan-selection-by-selectivity.png)
*捜査資料: 入札結果の勢力図。ヒット率の軸上に、3 つのスキャンの縄張りがはっきり分かれる*

「何行ヒットしそうか……って、つまり rows の*推定*ですよね。第 2 話の」

「そうだ。話が繋がってきたな。だがその前に、もうひとつ軸がある。お前が今夜追ってる事件に、直接関係する軸だ」

## 3

「湊。遅くなってる本番のページ、記事一覧だったな。**LIMIT は付いてるか**」

スロークエリログを見返す。付いていた。ページネーションの、ありふれた LIMIT。

「付いてます。それが何か……」

「ヒット件数が*同じでも*、LIMIT を付けるだけでプランは変わる。さっき Bitmap が落札した `author_id = 1` に、LIMIT 5 を付けて撃ってみろ」

```sql
EXPLAIN SELECT * FROM articles WHERE author_id = 1 LIMIT 5;
```

```
 Limit  (cost=0.29..20.79 rows=5 width=852)
   ->  Index Scan using index_articles_on_author_id on articles  (cost=0.29..201.14 rows=49 width=852)
         Index Cond: (author_id = 1)
```

「……Index Scan に変わった。WHERE は同じなのに」

「理由は 2 つのノードの*性格*の差だ。Index Scan は 1 件ずつ引く読み方だから、**5 行取れたら途中でやめられる**。Bitmap は全行ぶんのビットマップを作ってからヒープに向かうから、LIMIT があっても前処理は 1 ミリも減らない。第 2 話でやった A ≪ B の話だ──スタートアップの安い奴は、LIMIT に強い」

「Limit ノードのコスト 20.79 って、どこから来てるんですか。下の Index Scan は 201.14 なのに」

「按分だ。49 行の見積もりのうち 5 行しか取らないから、`0.29 + (201.14 - 0.29) × 5/49 = 20.79`。──手を動かした甲斐があったな、もう自分で検算できるだろう。じゃあ湊、ここで問題だ。**LIMIT をいくつまで増やしたら、Bitmap に戻る？**」

僕は LIMIT を 1 ずつ増やしながら EXPLAIN を撃ち続けた。午前 3 時半、無心の砲撃だった。

| LIMIT | 落札プラン | Limit total_cost |
|---:|---|---:|
| 15 | Index Scan | 61.78 |
| 16 | Index Scan | 65.87 |
| 17 | Index Scan | 69.97 |
| **18** | **Bitmap Heap Scan** | **73.32** |
| 19 | Bitmap Heap Scan | 77.14 |

「**17 と 18 の間**です。境目、ピンポイントでありました」

「では検算しろ。Index と Bitmap、それぞれの按分式が交わる点を解け」

```
Index Scan:        0.29 + (201.14 - 0.29) × L/49
Bitmap Heap Scan:  4.67 + (191.55 - 4.67) × L/49

交点: L = 4.38 × 49 / 13.97 ≈ 15.36
```

「……あれ。理論では **15.36** で交わる。L=16 からは Bitmap のほうが安いはずなのに、実測は 17 まで Index Scan が居座ってます。コスト計算が合ってるのは確認しました。**プランナが、最安値を選んでない**」

言ってから、ぞっとした。入札のルールが破られている。

「いい顔になってきたな」桐生さんは愉しそうだった。「それは不正じゃない。*条文*があるんだ。PostgreSQL のソース、`src/backend/optimizer/util/pathnode.c` を開け」

```c
#define STD_FUZZ_FACTOR 1.01
```

「プランナは 2 つのプランのコスト差が**約 1% 以内なら同点とみなす**。同点のときは別の基準──スタートアップの安さなどでタイブレークする。誤差の範囲で乗り換えてもプルプル震えるだけだからな。L=17 ではコスト差 0.66% で同点、Index が残留。L=18 で差が 1.01% になって、初めて Bitmap が有意に安いと認められた。理論交点 15.36 と実測 18 のズレ 1.5 は、この 1 行の定数で全部説明がつく」

EXPLAIN の数字の動きから、ソースコードの 1 行まで、一本の線が通った。深夜 3 時半の脳に、それは少し感動的すぎた。

## 4

「──さて、事件に戻るぞ」

桐生さんの声が、稽古の温度から捜査の温度に戻った。

「今夜の本番プランを見直せ。入札のしくみが分かった目で」

僕はエディタに戻った。Seq Scan on comments。さっきまで「バグでは」と思っていた行。今は違うものが見える。

「……プランナは、comments から*大量に*行が返ると見積もったから、Seq Scan に入札させた。それ自体は正しい判断です。でも」

「でも？」

「入札の**根拠が rows の推定**なんですよね。そして第 2 話で見つけたとおり、このプランは推定が 4 桁外れてる。つまり──**入札が、狂った見積書で行われてる**。プランナは『コスト最小』を選んだつもりで、実時間最大のプランを掴まされてる」

「それが今夜の事件の構図だ」

キーボードの音が止まった。

「コスト最小と実時間最小は、推定が正しいときにしか一致しない。プランナを責めるな。**奴に嘘の統計を読ませた何かがいる**。それが真犯人だ。──だがその前に、ひとつ気になる行がある。プランの上のほうだ。読んでみろ」

言われた行を見た。

「`Index Only Scan`……インデックス*だけ*スキャン？ そんなのもあるんですか。あ、でもその下に……`Heap Fetches:` って、なんか大きい数字が付いてます」

「Index **Only** Scan は、ヒープに一切触らずインデックスだけで答えを返す──いわば*完全犯罪*だ。痕跡を残さず、最速で終わる。だが `Heap Fetches` が付いてるということは、**完全犯罪のはずが、足がついてる**。ヒープに何万回も触りに戻ってる」

「なんで触りに戻るんですか？ インデックスだけで済むんじゃ」

「それを知るには、PostgreSQL が『この行は*誰から*見えるのか』をどう管理してるかを知る必要がある。visibility map──**可視性の地図**だ。次はそこへ潜る」

午前3時41分。容疑者リストに、新しい名前が増えた。

（第4話「完全犯罪の条件 ─ Index Only Scan」につづく）

## 今夜の捜査メモ

- プランナは**入札制**: Seq / Index / Bitmap にコストを見積もらせ、最安値を選ぶ。「インデックスがあるのに使われない」のはたいてい正しい判断
- インデックスは「キー + ヒープへのポインタ」だけを持つ。実データはヒープにあり、**連続 I/O（単価 1.0）とランダム I/O（単価 4.0）**の差が入札額を分ける
- ヒット件数の階段: **少なければ Index Scan（飛ぶ）、中くらいなら Bitmap（まとめ読み）、多ければ Seq（全部読む）**。決め手は「どのカラムか」ではなく「何行ヒットしそうか」
- Seq Scan は WHERE があっても読む量が減らない。`Filter` は読んだ後の尋問。だから `published = true`（75%）のコストは WHERE なしと同じ 12181.00
- **LIMIT はプランを曲げる**: 途中でやめられる Index Scan が有利になる。Limit ノードのコストは按分式 `startup + (total - startup) × L/rows`
- 切替点は理論交点どおりには来ない: コスト差 1% 以内は同点とみなす **`STD_FUZZ_FACTOR = 1.01`**（`pathnode.c`）がタイブレークする
- **コスト最小 = 実時間最小は、rows の推定が正しいときだけ**。推定が狂えば入札ごと狂う

## 1

午前2時47分。

ダッシュボードの上では、12 秒のグラフがまだ崖を描いている。僕の手元には、さっき覚えたばかりの EXPLAIN があった。

「桐生さん。じゃあ本番のクエリに EXPLAIN を打てば、犯人が分かるんですね」

「分からない」

「えっ」

「EXPLAIN が見せるのは*計画*だと言ったはずだ。『こう実行するつもりだ』というプランナの供述であって、現場で実際に何が起きたかじゃない。供述と現実がズレていたら？ プランナ自身が騙されていたら？」

騙されている。データベースの頭脳が。想像したことのない絵だった。

「だから両方を並べる。**計画と、実測を**。それが `EXPLAIN ANALYZE` だ。クエリを本当に実行して、プランの各行に『実際はこうだった』という数字を書き添えて返してくる。現行犯の記録だ」

「本当に実行される……さっき、刃物だって言ってましたよね」

「そうだ。だから先に鞘の使い方を教える。UPDATE や DELETE のプランを見たくなったら、必ずこれで包め」

```sql
BEGIN;
EXPLAIN ANALYZE UPDATE ...;  -- 本当に実行される
ROLLBACK;                    -- が、なかったことにできる
```

「トランザクションで囲んで、最後にロールバックする。実行の記録だけ手に入れて、変更は巻き戻す。公式ドキュメントもこの手順を推奨してる。──ただし今夜は SELECT しか触らない。まずはサンドボックスで、さっきと同じクエリの頭に ANALYZE を足せ」

```sql
EXPLAIN ANALYZE SELECT * FROM articles;
```

今度は、一瞬では返ってこなかった。体感で 0.5 秒弱。**実行されている**。さっきまで地図を眺めていただけの場所を、今、本当に歩いた。

```
                                                     QUERY PLAN
--------------------------------------------------------------------------------------------------------------------
 Seq Scan on articles  (cost=0.00..12181.00 rows=100000 width=852) (actual time=0.033..386.444 rows=100000 loops=1)
 Planning Time: 0.125 ms
 Execution Time: 390.524 ms
(3 rows)
```

「増えた場所が 3 つある。言ってみろ」

「ノードの行の右側に `(actual time=0.033..386.444 rows=100000 loops=1)`。それと最後に `Planning Time` と `Execution Time` が 2 行」

「そうだ。左半分の `cost / rows / width` は第 1 話と同じ**推定値**。右半分の `actual ...` が**実測値**だ。供述の隣に、監視カメラの映像が並んだと思え」

## 2

「actual time=0.033..386.444。この A..B の読み方は、cost と同じだ」

「スタートアップと、トータル……あ、でも単位が」

「そこだ。cost の 12181 は無次元の相対値だったが、**actual time は本物のミリ秒**だ。最初の 1 行が 0.033 ms で返り、全行返し切るのに 386.444 ms かかった。やっと時間の話ができる」

「Seq Scan だと A がほぼゼロなんですね。先頭ページを開けばすぐ 1 行目が返せるから……第 1 話と同じ理屈だ」

「そうだ。そして A と B の差は、ノードの*性格*を表す。A ≪ B なら『1 行目は早いが全部は長い』タイプ。逆に Sort や集計みたいに**全行読み終わらないと 1 行目すら返せない**ノードは、A が B に張り付く。`LIMIT 1` で速くなるかどうかも、ここを見れば見当が付く」

「末尾の 2 行は？」

「`Planning Time` はプランナが計画を練った時間。0.125 ms──計画立案は一瞬だ。`Execution Time` が実際に走った時間で 390.524 ms。クエリ全体の実時間を知りたければ、各ノードの actual time じゃなくこっちを見ろ」

「なるほど。……あの、もう一回同じの打ったら、actual time が 386 じゃなくて 250 くらいになったんですけど。壊れました？」

「壊れてない。2 回目はデータが OS のキャッシュに乗ってるからだ。実測値は毎回ブレる。ブレの理由を数字で暴く道具は後で教える──BUFFERS という。今は『実測は生き物だ』とだけ覚えておけ」

## 3

「次が今夜いちばん大事な罠だ。これを知らずに本番の出力を読むと、**間違った容疑者を逮捕する**」

桐生さんは、JOIN のあるクエリを打たせた。著者 1〜5 の記事に付いたコメントを引く、小さな結合だ。

```sql
EXPLAIN ANALYZE
SELECT a.title, c.body
FROM articles a
JOIN comments c ON c.article_id = a.id
WHERE a.author_id BETWEEN 1 AND 5;
```

```
 Nested Loop  (cost=7.11..11296.29 rows=2330 width=62) (actual time=0.625..29.985 rows=2469 loops=1)
   ->  Bitmap Heap Scan on articles a  (cost=6.68..754.94 rows=233 width=36) (actual time=0.562..3.537 rows=249 loops=1)
         Recheck Cond: ((author_id >= 1) AND (author_id <= 5))
         Heap Blocks: exact=241
         ->  Bitmap Index Scan on index_articles_on_author_id  (cost=0.00..6.62 rows=233 width=0) (actual time=0.497..0.497 rows=249 loops=1)
               Index Cond: ((author_id >= 1) AND (author_id <= 5))
   ->  Index Scan using index_comments_on_article_id on comments c  (cost=0.42..45.13 rows=11 width=58) (actual time=0.020..0.104 rows=10 loops=249)
         Index Cond: (article_id = a.id)
 Planning Time: 1.787 ms
 Execution Time: 30.337 ms
```

初めて見る、複数行のプランだった。木のようにインデントしている。

「Bitmap なんとかは第 3 話で教える。今夜見るのは 1 か所だけだ。いちばん下の Index Scan の行。読め」

「`actual time=0.020..0.104 rows=10 loops=249`……1 回 0.1 ms で 10 行。めちゃくちゃ速くて軽いノードですね。こいつはシロだ」

「**そう読んだ人間が、毎年何人も間違った index を張る**」

ぐ、と言葉に詰まった。

「`loops=249` を見ろ。このノードは 249 回呼ばれてる。外側の Bitmap Heap Scan が article を 249 行返して、**その 1 行ごとに** comments への Index Scan が 1 回走る。それが Nested Loop という結合のやり方だ。そして──ここが罠だ──**`actual rows` と `actual time` は、1 ループあたりの平均値**なんだ。公式ドキュメントにも明記されてる」

> `loops` 値はそのノードを実行する総回数を報告し、表示される実際の時間と行数は 1 実行当たりの平均です。
> ─ [PostgreSQL 17 文書 14.1.2 EXPLAIN ANALYZE](https://www.postgresql.jp/document/17/html/using-explain.html)

「つまり、このノードの本当の仕事量は掛け算しないと出ない」

```
総処理行数 = actual rows × loops = 10 × 249 = 2,490 行
総処理時間 ≈ actual time(B) × loops = 0.104 ms × 249 ≈ 26 ms
```

「2,490 行……Nested Loop の一番上の `actual rows=2469` とほぼ同じだ」

「平均値の四捨五入ぶんだけ端数が出る。近ければいい。──分かったか。『1 回 0.1 ms の速いノード』は、249 回呼ばれれば 26 ms の**主犯**になる。Execution Time 30 ms のうち 26 ms がこいつだ。loops を見ずに actual rows だけ読むのは、防犯カメラの静止画 1 枚で無罪にするのと同じだ」

背筋が冷えた。さっき僕は、確かに「シロだ」と言った。

## 4

「最後だ。推定と実測が*左右に並ぶ*意味を教える」

```
 Seq Scan on articles  (cost=0.00..12181.00 rows=100000 width=852)
                                            ^^^^^^^^^^^ 推定: 100,000 行
                       (actual time=...     rows=100000 loops=1)
                                            ^^^^^^^^^^^ 実測: 100,000 行
```

「このケースは推定と実測がぴったり一致してる。プランナの見立てが当たってるということだ。問題は**ズレたとき**。目安を置いておく」

| 乖離の比率 | 状態 | やること |
| --- | --- | --- |
| 2 倍以内 | 普通の誤差 | 何もしない |
| 10 倍以上 | プラン選択を間違えていそう | 統計情報・相関カラム・LIKE を疑う |
| 100 倍以上 | プランが暴走している疑い | 統計を取り直す。`pg_stats` で内訳を見る |

「なんでズレがそんなに問題なんですか？ 推定が外れても、結果は正しく返るんですよね」

「結果は正しい。**速度が死ぬ**。プランナは rows の見積もりを使って、次にどの結合方式を選ぶかを決めてる。『10 行のはずだ』と思って Nested Loop を選んだら実は 100 万行あった──そういう事故の現場を、俺は何度も見てきた」

「プランナが騙される原因って、何があるんですか」

「典型は 4 つだ。**統計情報が古い**──大量 INSERT / UPDATE の直後に ANALYZE が走っていない。これが一番多い。**相関カラムの AND**──`country='JP' AND lang='ja'` みたいに連動する条件を、プランナは独立とみなして掛け算で過小評価する。**LIKE 検索**、特に中間一致。**カラムを関数で包む**──`lower(title) = 'foo'` とか。プランナは関数の中の統計を持ってない」

僕はメモを取った。動機のプロファイリングだ、と思った。

## 5

「──さて。道具は揃った。本番だ」

心拍数が上がるのが分かった。時計は午前3時9分。

「リードレプリカに繋げ。SELECT のスロークエリなら、レプリカで EXPLAIN ANALYZE を打っても本番の書き込みには触れない。それでも作法として BEGIN と ROLLBACK で包め。癖にしろ」

言われたとおり、レプリカに繋いだ。スロークエリログからコピーした 12 秒のクエリを、震える手で貼り付ける。`EXPLAIN ANALYZE` を頭に付けて、実行。

──長い。体感でたっぷり十数秒。これが、実行されるということだ。

返ってきたのは、30 行を超えるプランの木だった。第 1 話の 1 行とは別世界だ。それでも、もう読めないわけじゃない。左に推定、右に実測。上から下へ、ノードごとに。

先に沈黙を破ったのは桐生さんだった。

「……見つけたか」

「……はい。たぶん。真ん中あたりの Nested Loop の内側です。推定は `rows=131`。でも実測が──**4 桁違う**。それに loops が、すごい数になってます」

「それが足跡だ」

キーボードの音が、また聞こえ始めた。

「プランナは騙されてる。少ない行数のつもりで Nested Loop を選んで、現場で暴走した。──だがな、湊。『どこがズレたか』が分かるのと『なぜそのプランが選ばれたか』が分かるのは別の話だ。プランナがどうやってスキャンの方法を選んでいるのか、その頭の中を知らないと、直したつもりがもっと遅くなる」

「頭の中……」

「Seq Scan、Index Scan、Bitmap Scan。プランナは毎回、全員のコストを見積もって**入札**させてる。次はその入札のしくみだ。──先に言っておくが、湊。『インデックスがあるのに使われない』のは、たいていバグじゃない。**正しい判断**だ」

インデックスを使わないことが、正しい。

午前3時16分。事件は、まだ終わらない。

（第3話「入札 ─ スキャンはこうして選ばれる」につづく）

## 今夜の捜査メモ

- `EXPLAIN ANALYZE` は**クエリを実際に実行して**、推定値の右隣に実測値を並べる。UPDATE / DELETE に打つなら `BEGIN; ... ROLLBACK;` で包む
- `actual time=A..B` は**実測ミリ秒**。A ≪ B は「1 行目は早い」型（Seq Scan）、A ≒ B は「全行揃うまで返せない」型（Sort・集計）
- 末尾の `Planning Time` は計画立案、`Execution Time` がクエリ全体の実時間
- **`actual rows` と `actual time` は 1 ループあたりの平均**。ノードの本当の仕事量は `× loops`:
  `rows=10 loops=249` → 総処理 2,490 行、`0.104 ms × 249 ≈ 26 ms`
- 推定 `rows` と実測 `actual rows` の乖離が捜査の起点: 2 倍以内は誤差、10 倍で疑い、100 倍で暴走
- プランナが騙される典型 4 パターン: **古い統計情報 / 相関カラムの AND / LIKE / カラムを包む関数**
- 実測値は毎回ブレる（キャッシュの温まり具合）。ブレの正体は BUFFERS の回で

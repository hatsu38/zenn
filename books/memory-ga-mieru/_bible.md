## ログライン

「メモリが視える」ようになった 3 年目エンジニアが、毎晩の再起動でしのぐ太り続ける Rails プロセスを救う。ただし視えたものは、コードで証明しない限り誰も信じてくれない。

### 能力のルール（絶対に破らない）
1. 視えるのは灯だけ。スクリーンショットも撮れない
2. 視えるのは「今この瞬間の風景」だけ。過去は視えない、原因は視えない
3. **視えたものを他人に伝えるには、コード（GC.stat / ObjectSpace / ヒープダンプ）で同じ事実を観測してみせるしかない** ← 読者が打つコードが物語の必然になる装置。「視えたから解決」は禁止
4. 能力は章が進むごとに解像度が上がる（下記レベル制）。物語上は「視かたを覚えた」として描く

### 能力レベル = 観測ツールの習得階段
- Lv.1 プロセスの「体格」が視える = `ps -o rss`（第1話）
- Lv.2 中に住む光の群れ（オブジェクト）が視える = `ObjectSpace.count_objects`（第2話）
- Lv.3 光と光を繋ぐ糸（参照）が視える = 到達可能性、後にヒープダンプ（第3話〜）
- Lv.4 光の色の違い（世代）が視える = RGenGC（第5話）

## 視える世界のビジュアル語彙（全話で統一）
- プロセス = ドーム状の街（ガラスの温室のイメージ）
- オブジェクト = 大きさの違う光の粒。String は白、Array は青、Hash は緑（色は T_STRING 等と対応）
- 参照 = 光同士を繋ぐ細い糸。糸を辿れば根（ルート）に着く
- GC のマーク&スイープ = 夜ごとに街を洗う「波」。糸で根に繋がっている光だけが波に耐える
- 断片化 = 街の空き地が虫食いになって、更地が作れない状態

## 実測環境

- rbenv の Ruby 4.0.5 (arm64-darwin25)。システム Ruby (2.6) は使わない
- 実験スクリプトの原本: セッション scratchpad の exp1〜exp5_*.rb（数値は 2026-07-04 実測）
- **重要: 物語で irb で打つ場面の数値は、必ず irb セッションで採り直す**（スクリプト実行と数値が大きく違う。例: 素のスクリプト RSS 17.3MB / irb 25.9MB、live_slots はスクリプト 20901 / irb 45045）
- irb は 1 行入力するたびに自身がオブジェクトを生むので、count 系の実験は (1) 先に GC.start (2) 1 万個以上生成でノイズを圧倒 (3) 差分に数十個の誤差を許す、が作法。文字列補間 `"x-#{n}"` は一時オブジェクトを余分に生む（1 万個生成で +16214 になった）ので、クリーンに見せたい場面ではリテラル `"light"` を使う
- 途中で GC が勝手に走って count が「減る」ことがある（実際に踏んだ罠。第 4 話のネタに使える）

### スクリプト実測値・第 4 話用（2026-07-04、exp6_wave.rb）
- GC.start 直後: GC回数=4, live_slots=17901 → `300_000.times { "ephemeral" }`（変数に入れない）後: GC回数=60, live_slots=21057
- つまり: 波は自動で 56 回来た。30 万個生んでも生き残りは 2 万強のまま（短命の光は勝手に回収される）

### スクリプト実測値・第 6 話用（2026-07-04、scratchpad/ch6/ の mini_app.rb / birthplace.rb で確定済み）
- 手法: `require "objspace"` + `ObjectSpace.trace_object_allocations_start`（出生記録）→ DEBUG_LOG（消し忘れのグローバル配列）に 3 万件 → GC.start → `ObjectSpace.dump_all` で heap.json へ。**外部 gem 不要、標準ライブラリのみ**（memory_profiler は使わない方針に変更）
- mini_app.rb 出力: `ダンプ完了: 20.2MB / 48182 行`（1 行 = 1 オブジェクトの JSON）
- birthplace.rb（file:line 集計上位 5 件）出力: 1 位 `30001	9.2MB	mini_app.rb:9`、2 位 `9	0.0MB	.../objspace.rb:100`、3 位 `8	0.0MB	mini_app.rb:18`、4 位 `3	0.0MB	mini_app.rb:17`、5 位 `2	0.0MB	mini_app.rb:15`
- ダンプ生 1 行（実物・キー抜粋）: `{"address":"0x104150018","type":"STRING","value":"GET /articles?page=599 200 12ms...(5回繰り返し)","memsize":320,"file":"mini_app.rb","line":9,"generation":4}`
- 注意: 出生地は「握った行（DEBUG_LOG <<）」ではなく「文字列が生まれた行」を指す。この差は本文で丁寧に扱う

### スクリプト実測値・第 7 話用（2026-07-04、scratchpad/diet.rb。exp5 を pages 付きで採り直した確定版）
- 開始時: RSS=17.3MB, live_slots=17934, pages=31
- 200万個抱えた状態: RSS=384.6MB, live_slots=2230096, pages=5392
- 全部手放して GC 後: RSS=354.2MB, live_slots=17954, **pages=4906**
- 読み方: 住民（live_slots）は完全に元に戻ったのに、RSS は 30MB しか下がらず 354MB で高止まり。Ruby 自身のページ（区画）も 5392→4906 とほぼ残ったまま＝空き家だらけの街。「痩せない」のではなく「土地を返していない」

### スクリプト実測値・第 8 話用（2026-07-04、scratchpad/compact.rb）
- 開始時: live_slots=17913, pages=31, RSS=17.1MB
- 200万個: live_slots=2413137, pages=1491, RSS=138.5MB
- 100人に1人だけ歯抜けに残して GC 後: live_slots=37939, **pages=1491（1 区画も返らない）**, RSS=138.7MB
- GC.compact（区画整理）後: live_slots=37962, **pages=998（493 区画返却）**, RSS=**107.9MB（31MB を OS に返却）**
- 読み方: 生き残り約 2 万人が 1491 区画にまばらに散っていると、区画は 1 つも返せない。compact で住民を詰めて引っ越しさせると、空になった区画が返る。live_slots が compact 後に微増（37939→37962）するのは観測用オブジェクトのノイズ
- 注: この実験の文字列は短い（スロット内に収まる）ので diet.rb よりページ効率が良い。数値の混用に注意

### スクリプト実測値・第 9 話用（2026-07-04、scratchpad/slope.rb。同一スクリプトを引数で健康/バグ入りに切替）
- 健康版（`ruby slope.rb`）: 観測1 old=17879 → 観測2〜5 old=17929 で完全フラット（live=17943, RSS=17.3MB も不動）＝**傾きゼロ**
- バグ入り版（`ruby slope.rb leak`）: old=17879→18930→19930→20930→21930 と**毎周 +1000 の階段**（живе も同様に上昇）＝傾き正
- 使い方: 最終話「見張りを立てる」の教材。同じ仕事量でも、握るバグの有無は old_objects の傾きだけで判別できる。監視すべきは RSS の絶対値ではなく old_objects / RSS の傾き
- 方針変更: derailed_benchmarks は使わない（外部 gem 不使用の方針を最終話まで貫く）
- 素の irb: RSS 25.9MB / GC.stat {count: 7, minor: 5, major: 2, heap_live_slots: 45045, heap_free_slots: 8665, old_objects: 24371}
- GC.start 直後の count_objects: {TOTAL: 53404, T_STRING: 15691, T_ARRAY: 3505, T_HASH: 272}
- `lights = Array.new(10_000) { "light" }` 後: T_STRING 25753（**+10062**。+62 は irb 自身のノイズ）
- `lights = nil; GC.start` 後: T_STRING 15744（+53 まで戻る）

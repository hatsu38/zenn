## 1

木曜の朝。通勤電車の中で、私は昨夜の疑問をもう一度反芻していた。

`lights = nil` で 1 万個の光が消えた。手放した瞬間に。

なぜ？

オブジェクトは、メモリの中に「在る」。その「在る」という状態が、なぜ `nil` で変わるのか。変数に `nil` を入れるという行為は、コンピュータの目からすると、何をしているのか。

昼休みに調べていくと、ヒントになりそうな言葉に行き当たった。

「参照」

オブジェクトと変数の関係は、「所有」ではなく「参照」だという。変数はオブジェクトそのものを持っているわけではなく、オブジェクトへの「参照」を持っているだけだと。

── 変数は、オブジェクトへの『糸』を持っているだけなのか。

頭の中で、何かが繋がりかけていた。早く帰って、確かめたかった。

## 2

帰宅して、いつもの Rails アプリを起動した。ドームが、机の端に生まれる。

いつもより、ゆっくり視た。眉をひそめて、目を凝らして。何か新しい見え方があるんじゃないか、そういう期待をもって。

──ッ

光。それと、もう一つ。

「……糸？」

ドームの床──底部──に、太い杭のようなものが立っていた。複数。変数とか定数の『根っこ』のような。そして、その杭から細い糸が伸びている。無数の糸が、空中を走って、光の粒に繋がっている。

糸は枝分かれしている。根から伸びた太い糸が、途中で光に触れて、そこからまた別の糸が分岐して、別の光へ向かっていく。そのネットワークは、複雑で、それでいて美しく見えた。

「視えた……」

視かたが、変わっていた。

昨晩までは、光と光とが独立していると思っていた。点としての粒。でも、実際は違う。光は、根の杭から伸びた糸で繋がれていて、その繋がりが、オブジェクトが「生きている」という状態なんだ。

糸をたどれば、根（ルート）に着く。根に繋がっている限り、光は波に呑まれない。

── 昨夜、光が消えたのは。

`lights = nil` と打った瞬間、その光を握っていた糸が、根から切れたんだ。根に繋がらなくなったオブジェクトは、『ゴミ』として回収される。──それをやっているのが、GC（ガベージコレクション）と呼ばれる何か、らしい。どうやって、は、まだ分からないけれど。

## 3

確かめたい。

その勢いのまま、テキストエディタを開いた。スクリプトファイルを作る。

```ruby
# reachability.rb
def live
  GC.stat[:heap_live_slots]
end

GC.start
puts "開始時: live_slots=#{live}"

data = Array.new(100_000) { "temp-#{rand(1000)}" }
GC.start
puts "10万個生成後: live_slots=#{live}"

data = nil  # 参照を切る
GC.start
puts "参照を切って GC 後: live_slots=#{live}"
```

```bash
ruby reachability.rb
```

出力:

```
開始時: live_slots=17909
10万個生成後: live_slots=117924
参照を切って GC 後: live_slots=17924
```

光が、視えた。

スクリプト実行の瞬間、小さなドームが生まれて、最初は少数の粒で満たされていた。その粒たちを繋ぐ細い糸が、根から伸びている。

次の瞬間、10 万個の光が、一気に生まれた。根から伸びた太い糸が、枝分かれして、無数の粒に繋がっていく。ドームが、ぱっと明るくなる。

そして、`GC.start` のコマンドが走った時。

根に繋がっていない 10 万個の光を、ぼんやりした波が覆った。光は次々と暗くなって、視界から消えていく。

live_slots の数字。17909 から 117924 へ。そして 17924 へ。

10 万個の光が、手放した瞬間に消えた。数字と視界が、完璧に一致した。

「参照と到達可能性」

その言葉が、初めて、言葉として自分の中に着地した。根の杭から参照の糸をたどって届くオブジェクトは生きていて、届かなくなったオブジェクトが、ゴミとして回収される。その『届く』『届かない』が、生死を分ける。

## 5

その夜。帰宅して、わざと「リーク」を再現する実験を始めた。

意図的に、参照を手放さないオブジェクトを作る。削除されないキャッシュみたいな。その中にどんどんデータを積み重ねて、GC が走ってもそれが消えない。そういう状態が「メモリリーク」だ。

```ruby
# leak.rb
def rss_mb
  (`ps -o rss= -p #{Process.pid}`.to_i / 1024.0).round(1)
end

CACHE = []  # 削除されないキャッシュ

5.times do |i|
  10_000.times { CACHE << "request-log-#{rand(100_000)}" * 10 }
  GC.start
  puts "回#{i + 1}: T_STRING=#{ObjectSpace.count_objects[:T_STRING]}, live_slots=#{GC.stat[:heap_live_slots]}, RSS=#{rss_mb}MB"
end
```

```bash
ruby leak.rb
```

出力:

```
回1: T_STRING=18612, live_slots=27955, RSS=20.4MB
回2: T_STRING=28641, live_slots=37973, RSS=23.8MB
回3: T_STRING=38641, live_slots=47973, RSS=27.2MB
回4: T_STRING=48641, live_slots=57973, RSS=30.5MB
回5: T_STRING=58641, live_slots=67973, RSS=34.4MB
```

毎回、GC.start している。掃除をしている。それなのに──

どれも、減らない。

T_STRING は 18612 から 58641。増え続ける。live_slots も、27955 から 67973。階段状に上昇し続ける。RSS も、20.4MB から 34.4MB。いっぺんも下がらない。

ドームの中で、何が起きているのか。

視界に映った。CACHE という定数から伸びた、太い、太い根の糸。その糸が、無数の白い光（T_STRING）を鎖のように繋いで、離さない。毎回、新しい光が生まれて、その根から伸びた糸に吸収されていく。

根に繋がった光は、GC が何度走ろうとも、波に呑まれない。糸で根に繋がっている限り、光は消えない。

「これが……リークの形」

その言葉が、喉から出た。

## 今夜の観測メモ

- **変数はオブジェクトそのものではなく、オブジェクトへの『参照』を持っているだけ**。その参照という糸を辿って根（ルート）に着くオブジェクトが生きていて、どこからも到達できなくなったオブジェクトがゴミとして回収される。これを「到達可能性」という
- `GC.stat[:heap_live_slots]` は「いま生きているスロット数」を返す。オブジェクトを手放して GC.start すれば、この数は減る。実測: 10 万個生成で 17909→117924、参照を切ると 117924→17924（ほぼ元通り）
- **メモリリーク** = 使わないのに参照され続けて回収できないオブジェクトが積もること。leak.rb で CACHE という定数がオブジェクトを握り続けると、T_STRING は 18612→58641、live_slots は 27955→67973、RSS は 20.4→34.4MB と、GC しても一度も減らない階段状上昇
- 夜ごとに街を洗う「波」の正体（どうやって根との繋がりを判定するのか）は、次話へ

## 1

水曜日の夜。自宅のノート PC で、GC の情報ページを読み進めていた。

ようやく腑に落ちたはずの「マーク&スイープ」は、その先に、もうひとつ深い工夫を隠していた。

「若い世代」と「古い世代」。

検索結果に、何度も出てくる言葉だった。

ページをスクロールしていくと、世代別 GC（Generational GC）という概念に行き当たった。

Ruby の GC は、手放されたオブジェクトを一律に消しているわけではない。オブジェクトの「年齢」に応じて、扱いを分けているのだという。

読み進めていくと、核心に辿り着いた。

**世代仮説。**

ほとんどのオブジェクトは、若くして死ぬ。

HTTP リクエストが一つ来るたびに、解析用の一時文字列が生まれる。計算用の配列が生まれる。でも、そのリクエストが処理され終わった瞬間、その一時オブジェクトは用済みになる。次のリクエストが来たら、また別の一時オブジェクトが生まれて、またすぐに死ぬ。

短命。そのオブジェクトたちは、生まれてから、次の波までの間に、ほぼ全員、糸が切れる。

だったら。

毎回、街全体をくまなくスキャンして、全員に印をつけるのは、効率が悪い。**若い区画だけを徹底的に洗えば、ゴミの大半は拾える。**

そういう最適化の話だった。

Ruby の波は、普段は若いオブジェクトだけを対象に走る。それが「小さい波」（minor GC）。一方、波を何度も生き延びたオブジェクト——もう当分死なないだろうと見なされたオブジェクト——は、「旧世代」（old）に移される。旧世代のオブジェクトは、普段の小さい波には含まれない。たまに来る大きな波（major GC）だけが、旧世代まで洗う。

「……それだ」

声が、喉から出ていた。

あの波の謎。何で毎回全部洗わないのか。その疑問が、ようやく解けた。

## 2

木曜の夜。帰宅してすぐ、実験スクリプトを書いた。

長生きするオブジェクトと短命なオブジェクト。二種類を並べて、同じプロセスの中で走らせてみる。

```ruby
# generational.rb
GC.start
puts "開始: minor=#{GC.stat[:minor_gc_count]}, major=#{GC.stat[:major_gc_count]}, old=#{GC.stat[:old_objects]}"

KEEP = Array.new(50_000) { "veteran-#{rand(1000)}" }  # 長生き組

500_000.times { "ephemeral-#{rand(1000)}" }  # 短命組

puts "短命50万個の後: minor=#{GC.stat[:minor_gc_count]}, major=#{GC.stat[:major_gc_count]}, old=#{GC.stat[:old_objects]}"

GC.start
puts "major GC 後: minor=#{GC.stat[:minor_gc_count]}, major=#{GC.stat[:major_gc_count]}, old=#{GC.stat[:old_objects]}"
```

実行した。

```bash
ruby generational.rb
```

出力:

```
開始: minor=2, major=2, old=13208
短命50万個の後: minor=60, major=4, old=67916
major GC 後: minor=60, major=5, old=67914
```

目の前の数字が、すべてを語っていた。

短命の 50 万個を生む間に、小さい波（minor GC）は 58 回も走った。でも大きい波（major GC）は 2 回だけ。

old_objects——旧世代に沈んだ光の数は、13208 から 67916 へ。55 万個ではなく、5 万個強だけ増えている。

その 5 万個は、何度も波に洗われながらも、生き延びたオブジェクトたちだ。KEEP という定数に握られた、古参の光。何度もの小さい波に耐えて、やがて旧世代に移された。一度 old に入ったら、小さい波の対象から外れる。だから、いくら短命オブジェクトが生まれて消えようとも、veteran の光たちは、そこで静かに沈んだままだ。

最後に `GC.start` で大きい波を呼んでも、old はほぼ変わらない（67916→67914）。KEEP は、まだ握られているから。

「……そういうことか」

スクリプトを眺めながら、つぶやいた。

あの夜から数え続けてきた「小さい波」「大きい波」の二つの違いは、実はこれだ。若い光だけを洗う小さい波と、街全体を洗う大きい波。その分け方は、不公平ではなく、むしろ何度も何度も短命なオブジェクトが生まれて消える日常に、最適化された仕組みだったのだ。

## 3

数字は、揃った。今度は、視たい。

ターミナルを閉じて、Rails アプリを起動した。

ドームが机の端に生まれて、いつものように光の群れが満ちている。その光たちを、これまでよりもゆっくり観察した。

眉をひそめて。目を凝らして。

あ。

光の色が、違って見える。

いや、色ではない。輪郭だ。

最近生まれたばかりの光は、輪郭が揺れている。瞬いている。まるで、ろうそくの炎のように。あたかも「まだ確定していない」みたいな、ゆらゆらした質感。

でも。

波を何度も生き延びた光は、違う。輪郭が固い。静かに光る。その深さに——まるで年輪みたいな——沈みがある。一目見て、「もう当分ここに留まるんだろう」という諦念が漂っている。

「……視えた」

ドームの奥のほう。街の奥底に、そういう深い光だけが、静かに集まっている一角がある。それが、旧市街——old 世代なんだ。若い光たちは表層で生まれては波に消えていくけれど、旧市街の光たちは、もう動かない。

ドームの構図が、初めて「二層構造」を持って視えた。

表層の新市街。奥底の旧市街。

その視え方は、世代仮説と完璧に一致していた。

## 4

翌日の昼。蓮見の席に向かった。

昨夜の気づきと、generational.rb の実行結果を見せた。

蓮見は、スクリーンに映った数字を眺めていた。コーヒーをひと口。また見る。

「……minor が 58 回で、major が 2 回か」

「はい。短命の 50 万個を生む間に、小さい波は何度も来たけど、大きい波はほぼ来なかった」

蓮見が頷いた。

「世代仮説だな。短命のオブジェクトは若い世代で次々と消える。だから、普段の GC は若い世代だけを対象にすればいい」

「はい。で、波を生き延びたオブジェクトが old 世代に移ると」

「そいつらは、小さい波の対象から外れる」

正確だった。

蓮見は、ダッシュボードのウィンドウを開いた。本番環境のメトリクス画面。階段状に上昇し続けるグラフの数々。

蓮見は、新しいグラフを追加した。

old_objects の推移。

グラフは、本来の「メモリ使用量」のグラフとは別に、「老い世代に沈んだオブジェクト数」の推移を映し出していた。

その形が、まったく同じだった。

毎日朝 4 時に再起動で一気に落ちて、そしてまた右肩上がり。階段状に上昇し続ける形。

leak.rb で見た「リークの形」と。

本番の数字が、完璧に一致していた。

「決まりだな」

蓮見が呟いた。

「リークしたオブジェクトは、何度もの小さい波に耐えて、結局、old 世代に溜まっていく。その old オブジェクト数が、毎日、朝 4 時の再起動まで階段状に上がり続けている」

「つまり」蓮見は立ち上がった。「old 世代の中身を見る。そこに、犯人がいる」

蓮見は、スケジュールを開いた。

「週明けの月曜。本番相当の環境でヒープダンプを採る。その中身をさらう。何のクラスが、どうやって、old オブジェクトを握り続けているのか。その正体を見つける」

ダンプ。いつか蓮見が呟いていた言葉に、ようやく輪郭がついた。

ドームの光を、一粒残らず数字に変える方法。それが、来週、目の前で動く。

## 5

その夜の帰り道。電車の中で、頭がずっと回っていた。

旧市街に沈んだ無数の光。

あの一つ一つに、名前があるはずだ。

どのクラスの、どのコードが生んだ、何の文字列なのか。

String だとしたら、その中身は何か。どの URL なのか。どのリクエスト ID なのか。

波に耐えて、耐えて、何度も何度も耐えて、ついに旧市街に辿り着いた光たち。

その一つ一つの「正体」が、週明け、ヒープダンプの中に、名前つきで現れるんだ。

数字に変わって。

「……今度こそ、顔が見たい」

そう、つぶやいていた。

## 今夜の観測メモ

- **世代仮説** = ほとんどのオブジェクトは、生まれてすぐに死ぬ。HTTP リクエストのたびに生まれる一時オブジェクトは、処理終了時に糸が切れて消える。だったら、毎回街全体を洗うより、若い区画だけを集中的に洗うほうが効率的
- **Generational GC** = Ruby は、普段は若い世代だけを対象に波を走らせる（minor GC）。波を何度も生き延びたオブジェクトは「旧世代」（old）に移り、小さい波の対象から外れる。たまに来る大きな波（major GC）だけが、旧世代まで含めてスイープする。実測: 短命 50 万個と長生き 5 万個を並べると、minor は 58 回走るが major は 2 回のみ。old_objects は 13208→67916（+54708、長生き組が old に昇格）。GC.start で major を強制しても old はほぼ変わらない（握られているから）
- **GC.stat の読み方** = `GC.stat[:minor_gc_count]` は小さい波の回数、`GC.stat[:major_gc_count]` は大きい波の回数。第4話の「GC 回数 60」（`GC.stat[:count]`）は、実はこの二つを足した値だった
- **リークしたオブジェクトは旧世代に溜まる** = 握られ続けたオブジェクトは、小さい波を何度も生き延びて、やがて旧世代に移される。そして major GC にも耐えながら、old_objects 数を増やし続ける。本番ダッシュボードの階段状上昇は、old_objects が膨張し続けている姿そのもの。次話で、ヒープダンプによって、その正体を特定する

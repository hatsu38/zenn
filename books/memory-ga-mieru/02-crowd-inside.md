## 1

月曜の朝。出社した私は、一日中、蓮見のことを考えていた。

昨夜見えた、あの光の群れ。ドームの中で呼吸する無数の粒。素の Ruby プロセスでも、irb でも、いつも同じパターンで現れるそれは、幻覚ではなく、何かの実在する現象のはずだった。

でも数字とどう繋がるのか、まだ全く見当がついていない。

GC.stat の計器盤は、相変わらず読めないままだ。heap。slots。old_objects。英語としては読める。その背後にある意味が、脳に届かない。

──だったら、誰かに話そう。

この会社で、証拠主義の鬼と言えば、蓮見だ。何かあるとすぐ「で、数字は？」と言って、根拠を求める。逆に言えば、数字があれば、話を聞いてくれる。

昼休みを避けて、午後 3 時。蓮見の席に近づいた。

チーム内では、蓮見は SRE（Site Reliability Engineering）の先輩だ。30 代前半で、ネットワーク・ストレージ・メモリ・システムコール──コンピュータの裏側のことなら何でも知っている。対照的に、私は経済学部から独学でこの業界に飛び込んだ 3 年目。コンピュータの裏側のことは、正直、ずっと避けてきた。

「蓮見さん、相談があります」

「どうした」

「あの……プロセスの中身が、視えるんです」

蓮見は、スクリーンから顔を上げた。瞬間、私は言ったことを後悔した。「視える」。そんな言葉を使ったら、この人は 100% 信じない。

予想通り、蓮見の表情は変わらなかった。まるで、何もない空気を相手にしたみたいに。

「で、数字は？」

一言。一蹴。

その一言が、全てだった。私は何も言い返せない。蓮見は画面に戻った。

私は引き返そうとしたが、その瞬間、蓮見が言った。

「待て。視える、って何だ」

「あ……」

「深夜の障害のせいか、何か幻覚でも見てる？」

「いや、そうじゃなくて……」

「仮説の出どころは問わない」蓮見は再び顔を上げた。「夢でも占いでもいい。検証できるなら何でもいい。だから、数字を持ってこい。再現手順つきでな」

その目は、完全に真面目だった。

## 2

帰宅後。蓮見の言葉が胸に刺さったまま、私は調べ物を始めた。

ドームの中の光の粒は、何なのか。

検索すると、「Ruby ではすべてがオブジェクト」という文句が何度も出てきた。文字列も、配列も、ハッシュも、数字も。ユーザーが書いたクラスのインスタンスも。すべてがオブジェクトで、すべてが同じように扱われるのだという。

そして、Ruby が作ったオブジェクトは、メモリの中でも決まった場所に置かれるらしい。

──メモリ。第1話で調べた「作業机」だ。

そのメモリの中に、いくつかの領域があって、オブジェクトはその中の「ヒープ」という領域に置かれるのだという。

ヒープ。

名前を見た瞬間、あのドームを思い出した。広い、広い空間。その中に無数の光。まるで、まるで──

「ヒープが、あのドームなのか」

声に出してそう言ってみた。

調べ物をもっと進めると、メモリは大きく二つの領域に分かれているのだと書いてあった。

一つは「スタック」。これは、いま実行中の関数だとか、ローカル変数だとか──今この瞬間に必要な、手元のメモ用紙みたいなもの。小さくて、素早くて、厳密な順序で積み上がる、手元のメモ用紙。

もう一つが「ヒープ」。大きな、自由な領域。Ruby が作るオブジェクトは、みんなここに置かれる。文字列も、配列も、ハッシュも。作業机のほとんどを占める、いちばん広い作業スペースだ。

「じゃあ、あの光の粒は……」

オブジェクトだ。

Ruby が作ったオブジェクトが、光の粒として見えているのか。そしてドームは、ヒープそのものか。

証明したい。数えたい。

## 3

翌日の夜。irb を起動して、実験を始めることにした。

蓮見に「数字を持ってこい」と言われたら、証明できる形で見せる。感覚の話じゃなくて、計測できる、再現できる話として。

まず、ドームの中の光を「数える」方法がないか調べた。

すると、Ruby には `ObjectSpace.count_objects` というメソッドがあるのだという。プロセス内のすべてのオブジェクトの数を、型ごとに数えてくれるらしい。

試してみる。

```bash
irb
```

視界に、いつもの小さなドームが生まれた。

調べた記事には、「数える前に一度 `GC.start` を実行して、不要なオブジェクトを掃除しておくのが作法」とあった。仕組みはまだ分からないけれど、お作法には従う。

```ruby
GC.start  # ガベージコレクションを走らせて、不要なオブジェクトを掃除
counts = ObjectSpace.count_objects
pp({TOTAL: counts[:TOTAL], T_STRING: counts[:T_STRING], T_ARRAY: counts[:T_ARRAY], T_HASH: counts[:T_HASH]})
```

出力:

```
{TOTAL: 53404, T_STRING: 15691, T_ARRAY: 3505, T_HASH: 272}
```

`GC.start` を打った瞬間、ドームの底から天辺へ、薄い波のようなものが、ざっと走った気がした。……気のせいだろうか。いや、いまは数えるほうが先だ。

5 万個。

何もしていない irb の中に、すでに 5 万個以上のオブジェクトが住んでいる。TOTAL = 合計。T_STRING = 文字列。T_ARRAY = 配列。T_HASH = ハッシュ。

視界のドームを見る。粒の色は、白がいちばん多い。次に青。緑は、まばら。数字の側も、文字列が 15691 個で圧倒的に多くて、配列が 3505、ハッシュが 272。

──もしかして、白い粒が、文字列？

まだ確かめようがない。でも、確かめる方法なら、思いつく。

## 4

次は、新しいオブジェクトを作ってみる。1 万個、一度に。そしてその前後で、オブジェクト数がどう変わるかを見る。

```ruby
lights = Array.new(10_000) { "light" }
counts = ObjectSpace.count_objects
puts "T_STRING: #{counts[:T_STRING]}"
```

出力: `T_STRING: 25753`

さっきは 15691 だった。今は 25753。

差分は──

**+10062**

1 万個作った。ほぼぴったり 1 万個増えた。62 個の余分は、irb 自身が私の 1 行コマンドを実行するたびに内部で生むオブジェクトだろう。irb も Ruby プロセスなのだから、当然だ。

視界を見た。

光が、一斉に生まれていた。星が一度に灯るみたいに、ドームの中が、ぱっと明るくなる。

生まれた 1 万個は、ぜんぶ白だった。

──白は、文字列だ。

作ったのは文字列 1 万個。増えた数字は T_STRING。生まれた光は白。三つが、一本の線で繋がった。

「……視えてるのに、説明できない」

それが、あの夜からの私の口癖だった。今夜、初めてその先に行けた。視えているものを、数字で言えた。

1 万個の光。数字でも、眼でも、同じものを指している。

## 5

翌日の昼。蓮見の席に、もう一度向かった。

今度は、「視えた」ではなく「数えた」という話として。

「あの……プロセスの中のオブジェクト数、数えられるんですね」

蓮見はコーヒーを飲んでいた。

「1 万個作ったら、1 万個増えました」

ノート PC を開いて、ターミナルをそのまま見せた。count_objects の出力。15691 から、25753 へ。

蓮見は画面をしばらく見た。秒数にして、5 秒くらい。長く感じた。

「……ふうん」

初めてまともな反応が返ってきた。

「count_objects か。で、再現手順は」

「irb で GC.start してから、Array.new で 1 万個の文字列を作るだけです」

蓮見は頷いた。「やってみるか。あとで」

手応え。小さい、小さい手応えだが、確かにあった。

蓮見はまだ「あのアプリの調査」までは言い出さないだろう。でも、扉が、少しだけ開いた気がした。

## 6

その夜。帰宅すると、一昨日から開きっぱなしの irb が、まだ生きていた。ドームも、机の端に変わらずそこにある。1 万個の白い光を抱えたまま。

ふと、思い立った。この光を「手放したら」、どうなるんだろう。

```ruby
lights = nil
GC.start
counts = ObjectSpace.count_objects
puts "T_STRING: #{counts[:T_STRING]}"
```

出力: `T_STRING: 15744`

25753 だった数が、15744 に戻った。

**ほぼ元通り。+53。**

1 万個の光が消えた。`lights = nil` と打っただけなのに。

視界では、何かが一斉に波に洗われるようにして消えた。白い光が次々と闇に呑まれていく。光の粒がぼんやり暗くなって、やがて見えなくなる。あたかも、夜の波が砂浜を洗い流すみたいに。

「手放す」。

その言葉の意味が、分からない。

`lights = nil` というのは、変数 lights に、「何ものでもない」という意味の nil を入れるコードだ。つまり、「ここに何も置かない」という宣言だ。それで、オブジェクトが消える？

なぜ？

そして──

本番のあのドームで、毎晩増え続けていた光。手放せば消えるのなら、あの光たちは──**誰かが、握り続けている**ということにならないか。

消える光と、消えない光。その違いは、なんだ。

「……視えてるのに、説明できない」

懐かしい、その苦しさが戻ってきた。

（第3話「増え続ける光」につづく）

## 今夜の観測メモ

- **オブジェクトはヒープに住んでいる**。Ruby が作るすべてのオブジェクト（文字列・配列・ハッシュ・その他もろもろ）は、メモリの「ヒープ」という広い領域に置かれる。スタックは「いま実行中の関数の手元メモ」程度。ほぼ気にしなくてよい
- `ObjectSpace.count_objects` で、プロセス内のオブジェクト数を型ごとに数えられる。T_STRING = 文字列、T_ARRAY = 配列、T_HASH = ハッシュ。TOTAL = 合計
- **irb で正確に数えるなら、先に `GC.start` を実行する**。irb 自身が補完・履歴などの内部オブジェクトを絶えず生み出しているので、ノイズを掃除してから観測するのが作法
- 何もしていない irb セッションには、すでに**53404 個**のオブジェクトが住んでいる（T_STRING 15691、T_ARRAY 3505、T_HASH 272）
- 1 万個のオブジェクトを作ると、数えた数も 1 万個増える。証明できた。だが、**手放したオブジェクトはなぜ消えるのか**。その答え（と、なぜ本番では消えない粒が残るのか）は、次話へ

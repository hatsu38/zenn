## 1

月曜の朝。

オフィスに着いて、すぐに蓮見を探した。

「監視、用意できました」

蓮見が椅子を回した。

「何を見張る」

「old_objects と RSS の傾きです。同じ仕事量なのに、握るバグがあると old が増え続けます。それを、アラートで捕まえたいんです」

蓮見が頷いた。

「いいだろう。どうやる」

私たちが作ったのは、ダッシュボードへの追加だった。いつもの画面に、2 本のグラフを足す。

一本目：old_objects と heap_live_slots の時系列。

二本目：RSS の時系列。

そこに、シンプルなアラートを乗せた。「old_objects が数時間、上がり続けたら通知」「RSS が一日を通して上がり続けたら通知」。

具体的な数値は出さない。傾き——「上がり続ける」「下がり続ける」——その方向を見張るだけだ。

## 2

火曜の午後。

オフィスのミーティングスペースに、チームが集まった。

蓮見からの指名で、私が小さな勉強会をすることになっていた。対象は Rails チーム 7 人。「メモリリーク判定の目」を、全員に持たせる話だ。

スライドは用意せず、コードだけを見せることにした。

手元の MacBook を、プロジェクタに接続する。

```ruby
# slope.rb ── 健康診断は「傾き」で見る（leak を付けると、握るバグ入りで動く）
HOLD = ARGV[0] == "leak"

CACHE = []
5.times do |i|
  100_000.times { "work-#{rand(100)}" }                 # ふつうの仕事（短命の光）
  1_000.times { CACHE << "sticky-#{rand(100)}" } if HOLD  # こっそり握る
  GC.start
  puts "観測#{i + 1}: old=#{GC.stat[:old_objects]}, live=#{GC.stat[:heap_live_slots]}, RSS=#{(`ps -o rss= -p #{Process.pid}`.to_i / 1024.0).round(1)}MB"
end
```

「このスクリプト。同じ仕事を繰り返してるんです。ただし、引数を付けるかどうかで、握るバグがあるかないかが決まります」

まず、健康版。

```bash
ruby slope.rb
```

出力：

```
観測1: old=17879, live=17926, RSS=17.3MB
観測2: old=17929, live=17943, RSS=17.3MB
観測3: old=17929, live=17943, RSS=17.3MB
観測4: old=17929, live=17943, RSS=17.3MB
観測5: old=17929, live=17943, RSS=17.3MB
```

「ここを見てください」と指す。「観測 2 以降、old が 17929 で微動だにしません。傾き、ゼロです」

次に、バグ入り版。

```bash
ruby slope.rb leak
```

出力：

```
観測1: old=17879, live=18927, RSS=17.2MB
観測2: old=18930, live=19943, RSS=17.5MB
観測3: old=19930, live=20943, RSS=17.5MB
観測4: old=20930, live=21943, RSS=17.5MB
観測5: old=21930, live=22943, RSS=17.5MB
```

「今度は。old が毎周ほぼ +1000 の階段です。同じ仕事なのに。握るバグがあるかないかは、この old_objects の傾きだけで判別できるんです」

後ろの席から、素朴な質問が飛んだ。

「GC が自動で掃除してくれるんですよね？　なのに、なんで増え続けるんですか」

頷いた。いい質問だ。

「GC が掃除できるのは、どこからも参照されていないオブジェクトだけなんです。どこかの配列やキャッシュに握られている限り、掃除が何度走っても消えません。それで、掃除を何度も生き延びたオブジェクトは old——『古株』として数えられます。つまり古株が増え続けているのは、『誰かが握り続けている』動かぬ証拠なんです」

その言葉がすらすら出てくる自分に、少し驚いていた。

あの夜、「メモリってなんですか」すら口に出せなかった自分が、いま、みんなの前で説明している。

「ほかに、質問ありますか」

いくつも手が上がった。勉強会は、予定を三十分超えた。DB チームからも、ひとり聞きに来ていたらしい。

## 3

水曜の朝。

蓮見に報告した。

「勉強会、やりました。old の傾きで判別できることと、それを見張ることの話です」

「反応は」

「盛り上がりました。監視のグラフ、もうみんな見てくれてます」

蓮見が頷いた。

「では、行くか」

その日の夜。決行の日だ。

PR を開いた。タイトルは「朝の体操: 再起動の自動化を廃止」。

内容は、cron 設定の削除。

```diff
-# 朝の体操: メモリが増え続けるため毎晩リセット（恒久対応まで）
-0 4 * * * /usr/local/bin/restart_puma.sh
```

それだけ。

蓮見がレビューを見て、一言。

「いいだろう。マージ」

本当に、それだけだった。

しばらく待つ。デプロイパイプラインが通る。本番に反映される。

これで、今夜の 4 時にあのスクリプトが走ることは、もうない。

## 4

その夜、私は目覚ましをかけて寝た。

午前 4 時。

かつて、あの障害アラートで何度も叩き起こされた側だった自分が、今度は自分の意思で目を覚ます。

MacBook を開いて、ダッシュボードの RSS グラフを見た。

いつもなら、このグラフには毎晩、同じ形が刻まれてきた。日中に太って、午前 4 時ちょうどに、ドカンと落ちる。のこぎりの歯。その周期が、何か月も繰り返されてきた。

午前 4 時。

——落ちない。

グラフは、昨日の夕方からずっと、静かな横ばいのまま、まっすぐに夜を越えていく。

のこぎりの歯が、初めて、途切れた。

午前 4 時 30 分。まだ、まっすぐ。

窓の外が、少しずつ白んでいく。

私はしばらく、ただの水平な線を、飽きずに眺めていた。世界でいちばん退屈で、世界でいちばん誇らしいグラフだった。

朝、Slack に蓮見からのメッセージが届いていた。

「傾き、ゼロ」

## 5

その後の数週間。

ドームは、今も視える。

オフィスで MacBook を開くたび、目の隅に、あの街が在る。光の粒たちが、呼吸している。

でも、もう能力に頼る必要はない。

チームの全員が、数字で同じものを見られるから。

——あの夜、寝転がったベッドで、私が一番知りたかった問い。

「メモリが足りないって、どういうこと？」

いまなら、言葉で説明できる。

メモリは、プロセスの作業机。その上にヒープという街があって、オブジェクトという住民が暮らしている。住民のほとんどは若くして役目を終え、夜ごとの波が、糸の切れた者を静かに連れていく。

でも、誰かに握られたままの住民は、波が何度来ても連れていってもらえない。生き延びて、老いて、旧市街に沈んで、街はじわじわ太っていく。それが、リーク。

そして、住民がいなくなっても、街は借りた土地をすぐには返さない。だから体格は、すぐには痩せない。でもそれは、病気じゃない。

見るべきは、大きさじゃなくて、傾き。

懐かしい言葉たちが、頭の中で、それぞれの場所を持つようになっていた。

作業机。住民。糸。波。旧市街。土地。

あの夜、見えたドームの隅々が、いまは、コードになっている。

GC.stat の計器盤。かつては読めない針の並びだったそれが、いまは——

「old_objects」「heap_live_slots」「heap_allocated_pages」「heap_free_slots」

どの針も、読める。

## 6

平和な午後。

Slack にアラートが鳴った。

「たまに、同じ記事が二重に投稿される」

そんな報告が上がってきた。投稿フォームの送信ボタン。どうやら、クリック一回で、記事が 2 個生まれるときがあるらしい。

「再現しない。ログにも異常がない」

ドームを視てみた。

メモリは、どこも健康だ。old_objects は傾きゼロ。RSS も、穏やかな横ばい。

リークではない。

蓮見に伝えると、蓮見は報告のログをしばらく眺めて、ぼそっと言った。

「……メモリじゃないな。スレッドだ」

「スレッド……？」

蓮見は、その先を続けなかった。ただ、椅子を引いた。

「また今度」

（おわり）

## 今夜の観測メモ

### メモリの健康診断チェックリスト

1. **見るのは絶対値ではなく傾き** ── RSS と old_objects は「増えているか」「止まっているか」を判断材料にする
2. **増え続けたら、最小再現か出生地特定** ── まず CACHE リークの再現スクリプトを組む。次に ObjectSpace.dump_all で heap.json を吐いて、出生地の集計
3. **出生地から 1 行読み下げて「握っている手」を探す** ── 重いオブジェクトが何に握られているのか、参照を辿って原因を特定
4. **直したら、傾きゼロを確認** ── リーク修正後は old_objects と RSS の傾きが落ちることで治療完了を確認
5. **RSS の高止まりは病気ではない** ── 三層構造（Ruby / malloc / OS）では、malloc の空き区画は返されず高止まりする。必要に応じて温まったあとに GC.compact で土地の返却
6. **定期再起動は鎮痛剤であって治療ではない** ── リークを直すまでは、朝の体操が続く。リークの根を除きながら、監視で傾きを見守る

### slope.rb の実測値の要約

- 健康版（`ruby slope.rb`）：old_objects は観測 2 以降 **17929 で完全フラット**（傾きゼロ）
- バグ入り版（`ruby slope.rb leak`）：old_objects は毎周 **+1000 の階段**（傾き正）
- **判定基準**：同じ仕事量でも、握るバグの有無は old_objects の傾きだけで判別できる

### 本書で使った観測コマンド一覧

- `ps -o rss= -p <PID>` ── プロセスの RSS（物理メモリ使用量）を確認
- `GC.stat` ── ガベージコレクタの詳細統計。old_objects, heap_live_slots, heap_allocated_pages など
- `ObjectSpace.count_objects` ── オブジェクト型別の個数集計（T_STRING, T_ARRAY など）
- `GC.stat[:heap_live_slots]` ── 生きているオブジェクトの総数
- `ObjectSpace.dump_all` + 出生地集計 ── heap.json を吐いて file:line ごとにメモリ消費を調査
- `GC.compact` ── 生きているオブジェクトを前方に詰めて、空いたページを OS に返す

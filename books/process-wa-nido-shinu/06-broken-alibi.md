## 1

月曜、朝 9 時。会議室のモニターに、氷室さんがミニチュア世界を映した。

「本番の再演よ。購読者 2 万人を **200 人**に、送信時間を **1 ミリ秒の sleep** に縮めた。構造は PR #1247 の送信ループと同じ。`deliver` が sleep なのは手抜きじゃない。メール送信の正体は**ネットワーク IO**──SMTP サーバーの返事を待つ時間。それを 1 ミリ秒に凝縮しただけ」

```ruby:race.rb
users = (1..200).map { |n| "user_#{n}" }
sent  = []
i     = 0

threads = 4.times.map do
  Thread.new do
    while i < users.size
      user = users[i]   # (1) 次のユーザーを取る
      sleep 0.001       # (2) メールを送る（IO 待ち）
      i += 1            # (3) 「次へ」を進める
      sent << user
    end
  end
end

threads.each(&:join)

dup = sent.tally.select { |_, c| c > 1 }
puts "送信件数:       #{sent.size}"
puts "ユーザー数:     #{sent.uniq.size}"
puts "二重送信の例:   #{dup.first(3).inspect}"
puts "二重送信の人数: #{dup.size}"
```

「実行して。これで何も起きなければ、スレッドもシロよ」

祈るような気持ちで Enter を押した。

```
$ ruby race.rb
送信件数:       203
ユーザー数:     200
二重送信の例:   [["user_1", 4]]
二重送信の人数: 1
```

「……**出た**。200 人しかいないのに 203 通。user_1 に、**4 通**」

「現行犯逮捕ね」氷室さんは静かに言った。「怪事件①の再現に成功。じゃあ次は、**目撃者を増やす**。誰が user_1 に送ったのか、ワーカーに名札をつけて記録させるの」

```ruby:witness.rb
users   = (1..200).map { |n| "user_#{n}" }
witness = []   # [ワーカー名, ユーザー] の記録
i       = 0

threads = 4.times.map do |w|
  Thread.new do
    name = "worker_#{w}"
    while i < users.size
      user = users[i]
      sleep 0.001
      i += 1
      witness << [name, user]
    end
  end
end
threads.each(&:join)

puts "user_1 を送ったワーカー:"
witness.select { |_, u| u == "user_1" }.each { |w, u| puts "  #{w} が #{u} を送信" }
```

```
$ ruby witness.rb
user_1 を送ったワーカー:
  worker_0 が user_1 を送信
  worker_1 が user_1 を送信
  worker_2 が user_1 を送信
  worker_3 が user_1 を送信
```

「**全員だ**。4 人のワーカー全員が、それぞれ user_1 に送ってる。1 人が 4 回送ったんじゃなくて、**4 人が 1 回ずつ**……」

「証言が揃ったわね。全員に悪意はない。全員が『自分が取った次のユーザー』に真面目に送っただけ。なのに宛先が重なった。──じゃあ、犯行の瞬間をスローモーションで見ましょう」

## 2

「まず前提。`user = users[i]` と `i += 1` は、あなたの目には 2 行に見える。でも働き手の動きとしては、少なくともこう分かれる」

1. `users[i]` を**読む**（いまの i は 0。user_1 を手に取る）
2. メールを**送る**（IO 待ち。1 ミリ秒〜数百ミリ秒）
3. `i += 1` を**書く**（i を 1 にする）

「ポイントは (2)。**IO の待ち時間、その働き手は何もしてない**。Ruby は律儀だから、待ってる間、他のスレッドに仕事を回すの。つまり──」

「(1) と (3) の間に、**別のスレッドが割り込める**」

「そう。バッチ開始直後の 4 人の動きを追うわよ」

| 時刻 | worker_0 | worker_1 | worker_2 | worker_3 | i |
| --- | --- | --- | --- | --- | --- |
| 0ms | `users[0]` を読む → user_1 | | | | 0 |
| 0ms | 送信中（待ち） | `users[0]` を読む → **user_1** | | | 0 |
| 0ms | 送信中 | 送信中 | `users[0]` → **user_1** | | 0 |
| 0ms | 送信中 | 送信中 | 送信中 | `users[0]` → **user_1** | 0 |
| 1ms | `i += 1` | `i += 1` | `i += 1` | `i += 1` | 4 |

「4 人全員が、i がまだ 0 のうちに `users[0]` を読んだ。だから **user_1 に 4 通**。さっきの証言（4 人が 1 回ずつ）ともぴったり一致する。そのあと全員が i を進めたから、i は一気に 4 になって、user_2 から user_4 は**誰にも送られないまま飛ばされた**……あれ、でもユーザー数は 200 になってますよ？」

「4 人が i += 1 する*タイミング*も毎回ずれるからよ。読み込みと書き込みが偶然きれいに交互になれば正常に進む。今回はたまたま抜けが出なかった。**出力が毎回ちょっと違う──それこそが、この犯人の最大の特徴**。名前を教えるわね」

氷室さんはホワイトボードに書いた。

**レースコンディション（競合状態）** ── 複数の働き手が同じデータに同時に手を伸ばし、**実行順序という偶然**によって結果が変わってしまう状態。

「本番との突き合わせもしておくわよ。本番で被害が『十数人・飛び飛び』だった理由──本物の送信は 1 ミリ秒じゃなくて数十〜数百ミリ秒で*揺れる*。割り込みの起きやすさはタイミング次第で毎回違う。**運が悪い瞬間にだけ、二重取りが起きる**。staging が 3 日間無事だった理由も同じ。staging の送信先はダミーの SMTP で、応答が速くて安定してた。**環境が健全なほど、この犯人は尻尾を出さない**の。性能問題は負荷をかければ再現するけど、レースは『再現しないこと』が何の証明にもならない。確率の犯罪、って言ったでしょ」

## 3

「じゃあ、帳簿の 257 不足──怪事件④も再現するわよ。今度は共有カウンタだけを抜き出したミニチュア」

```ruby:race_counter.rb
def deliver(user)
  sleep 0.001  # メール送信 = ネットワーク IO（1ms かかると仮定）
end

counter = 0

threads = 4.times.map do
  Thread.new do
    100.times do
      current = counter     # (1) 今の値を読む
      deliver(current)      # (2) 送信する（IO 待ち）
      counter = current + 1 # (3) +1 して書き戻す
    end
  end
end

threads.each(&:join)
puts "送ったはずの数: 400"
puts "counter の値:   #{counter}"
```

「4 人が 100 通ずつ。合計 400 通送って、1 通ごとに帳簿を +1。さあ、帳簿はいくつになる？」

「レースが起きるんですよね……380 とか？」

```
$ ruby race_counter.rb
送ったはずの数: 400
counter の値:   100
```

「**100！？** 4 分の 1 しかない」

「ふふ。読み解いてごらんなさい。4 人とも `counter = 0` を読む → 4 人とも送信で 1ms 待つ → 4 人とも `0 + 1 = 1` を書く。**4 通送って帳簿は 1**。それが 100 周続いた。カウントの 4 分の 3 は、上書きで**蒸発**したの」

「`counter += 1` って、1 個の操作じゃないんだ……」

「そこが今日いちばん大事な一文よ。`counter += 1` は糖衣で、実体は `counter = counter + 1`。**読む → 足す → 書く、の 3 手**。3 手の間に他人が同じ値を読めば、その人の +1 とあなたの +1 は**同じ 1**になる。本番の『送信 19,743 / 対象 20,000』は、二重送信で*増えた*ぶんすら飲み込んで、蒸発のほうが上回った帳簿の姿。不足数が毎日違ったのも、割り込みの回数が毎日違うから」

## 4

「ひとつ引っかかってることがあるんですけど、いいですか。前に別のチームの人から『Ruby には GVL があるから、スレッドは実は同時に動かない。だからそんなに気にしなくていい』って聞いたことがあって」

「──いい機会だから、その伝説はいまここで殺しましょう。実験で」

```ruby:pure_cpu.rb
counter = 0
threads = 4.times.map do
  Thread.new { 250_000.times { counter += 1 } }
end
threads.each(&:join)
puts "実測値: #{counter}"
```

```
$ ruby pure_cpu.rb
実測値: 1000000
```

「あれ。100 万ぴったり。……race_counter.rb と同じ 3 手のはずなのに、今度は 1 個も蒸発してない」

「3 回流しても 3 回ぴったりよ。これが伝説の出どころ。GVL の詳細は第 8 話でやるけど、先に結論だけ言うと──CRuby では、**IO 待ちのような『隙』がないと、スレッドの割り込みが 3 手の急所に入りにくい**の。純粋な計算ループは隙が小さいから、こういう素朴な実験では壊れて見えない。それで『Ruby は大丈夫』という都市伝説が生まれる」

「でも、うちの本番は壊れた」

「そう。**本物の仕事には必ず IO が挟まる**からよ。メール送信、DB クエリ、API 呼び出し、ログ書き込み。race_counter.rb と pure_cpu.rb の違いは `deliver` の 1 行だけ。つまり『いまは壊れないコード』は、**誰かが間に 1 行 IO を足した瞬間に壊れるコード**なの。半年後の同僚が、悪気なくログ出力を 1 行足すところを想像しなさい。──『たまたま壊れない』を『安全』と呼んではいけない。これがこの実験の教訓」

「レースって、この 2 パターンだけですか？　`i` の取り合いと、カウンタの蒸発と」

「形としては大きく 2 つ。今日の (1)〜(3) で言うと、**読んで、確認して、行動する**間に世界が変わる型（check-then-act ── `users[i]` の二重取りがこれ）と、**読んで、変えて、書き戻す**間に割り込まれる型（read-modify-write ── カウンタの蒸発がこれ）。実務の現場は、だいたいこのどっちかの変奏よ。『在庫が 1 個あるのを確認してから引き当てる』『残高を読んでから引き落とす』『キャッシュになければ作って入れる（`||=`）』──全部、複数の働き手が同時にやれば同じ事件が起きる」

僕は、二つの実行結果を並べて眺めた。二重送信と、カウント消失。まったく別の顔をした 2 つの怪事件が、**「読みと書きの間に割り込まれる」というたった 1 つの手口**で説明された。

「これで全部の謎が解けました。……あとは直すだけですよね。読みと書きの間に割り込まれるのが原因なら、**割り込ませなければいい**。調べたんです、Mutex っていうのがあるって。鍵をかけるんですよね」

「あら、予習してるじゃない」

氷室さんは感心した顔をして──それから、少しだけ声を落とした。

「いいわ、修正はあなたがやりなさい。ただしひとつだけ忠告。**鍵はね、閉じ込めるためにも使えてしまう**の。かけ方を間違えると、レースより静かで、レースよりたちの悪い事件が起きる。──直したら、staging で一晩流してから見せて」

その忠告の意味を、僕は水曜の朝に思い知ることになる。

（第7話「密室 ── Mutex とデッドロック」につづく）

## 捜査メモ

- **レースコンディション**: 複数スレッドが共有データに同時に手を伸ばし、**実行順序の偶然**で結果が変わる状態。「毎回ちょっと違う結果」「ごく低確率で発生」「健全な環境ほど再現しない」が典型的な症状
- `users[i]` を読んでから `i += 1` するまでの間に IO 待ちがあると、**他のスレッドが同じ `i` を読む** → 二重送信（実測: 200 人に 203 通、user_1 に 4 通。目撃証言: **4 ワーカー全員が 1 回ずつ**送信）
- `counter += 1` は 1 手ではなく「**読む → 足す → 書く**」の 3 手。割り込まれると +1 同士が重なって**蒸発**する（実測: 400 通送って counter は 100）── 増える事件と減る事件が同じ 1 つの手口から生まれる（怪事件①④の真相）
- **「GVL があるから大丈夫」は都市伝説**。純 CPU ループは 100 万ぴったりになる（実測）が、間に IO が 1 行入った瞬間に壊れる。「たまたま壊れない」≠「安全」
- レースの 2 大類型: **check-then-act**（確認と行動の間に世界が変わる）と **read-modify-write**（読みと書き戻しの間に割り込まれる）。在庫引き当て・残高更新・`||=` キャッシュも同型
- IO（メール送信・DB・API・ログ）の待ち時間に Ruby はスレッドを切り替える。**「1 人なら正しいコード」は、共有データがある限り 4 人では正しくない**

## 1

「スレッドを一言で言うと、**プロセスという家の中で働く、複数の働き手**よ」

金曜の朝。氷室さんはホワイトボードに家の絵を描いた。

「プロセスは家。メモリ──変数も配列もハッシュも──は家の中の家具。fork は**家ごと複製**して隣に建てる。スレッドは**同じ家に働き手をもう一人増やす**。家具は買い足さない。同じ冷蔵庫、同じ本棚、同じ帳簿を、全員で使う」

「どんなプロセスにもスレッドはいるんですか？」

「いるわ。あなたが普通に `ruby foo.rb` と打ったときも、**メインスレッドという働き手が 1 人**で全部やってる。プロセスとスレッドは別々の概念というより、**プロセス = 家 + 最低 1 人の働き手**というセットなの。`Thread.new` は 2 人目を雇う操作。──昨日と同じ実験を、fork をスレッドに変えてやってみて。答え合わせよ」

```ruby:thread_memory.rb
sent_flags = []

t = Thread.new do
  sent_flags << "user_1"
end

t.join
puts "メイン: sent_flags = #{sent_flags.inspect}"
```

```
$ ruby thread_memory.rb
メイン: sent_flags = ["user_1"]
```

「**見えた**。昨日の fork では親から `[]` だったのに、スレッドだと `["user_1"]` が入ってる」

「これが決定的な違い。並べて刻みなさい」

|  | fork（別の家の双子） | Thread（同じ家の同居人） |
| --- | --- | --- |
| メモリ（変数） | **コピー**。以後は独立 | **共有**。全員が同じ実体を読み書き |
| 協力の手段 | パイプ等の「手紙」が必要 | 変数に代入するだけ（速い） |
| 作るコスト | 家 1 軒ぶん（CoW で軽減） | 働き手 1 人ぶん。ずっと軽い |
| 片方が異常終了したら | もう片方は無事 | 家ごと巻き込まれうる |
| 変更の伝わり方 | 伝わらない | **即座に全員に伝わる** |

「昨日のパイプ、あんなに手続きが要ったのに。スレッドだと `<<` 一発……」

「そう、**協力のコストが桁違いに安い**の。それがスレッドの光の面。全員が同じ家具を見てるんだから、受け渡しという概念そのものが要らない。──ちなみに『全部共有』は正確じゃなくて、ひとつだけ**各自の私物**がある。**スタック**──いま実行中のメソッドのローカル変数や呼び出し履歴を積む作業机ね。働き手ごとに 1 台ずつ。だから同じメソッドを 2 人が同時に実行しても、それぞれの机の上の作業は混ざらない。混ざるのは**ヒープ**──`sent_flags` みたいに机の外に置かれた共有の家具のほう。今回の事件で問題になるのは、全部ヒープ側よ」

## 2

「`t.join` は？」

「『その働き手の仕事が終わるまで待つ』。fork における `Process.waitpid` の同居人版ね。そして join には、waitpid の死亡届と同じくらい大事な役目がもうひとつある。──同居人が**仕事中に事故を起こしたとき**の話。実験するわよ」

```ruby:thread_crash.rb
t = Thread.new do
  raise "SMTP接続エラー"
end

sleep 0.1
puts "メイン: 私はまだ生きている"
puts "t の容態: #{t.status.inspect}"   # nil = 例外死

t.join   # ここで初めて例外が飛んでくる
```

```
$ ruby thread_crash.rb
#<Thread:0x00000001030c2f48 thread_crash.rb:1 run> terminated with exception (report_on_exception is true):
thread_crash.rb:2:in 'block in <main>': SMTP接続エラー (RuntimeError)
メイン: 私はまだ生きている
t の容態: nil
thread_crash.rb:2:in 'block in <main>': SMTP接続エラー (RuntimeError)
```

終了コードは 1。だが出力の順番が奇妙だった。

「スレッドの中で raise してるのに……メインは『まだ生きている』って言えてる。エラーで死んだのは最後、join のところ？」

「そこがスレッドの例外の不気味さよ。時系列で読むわね。まず t が例外で死ぬ。このとき**プロセスは死なない**。Ruby は親切だから『こんな例外で死にましたよ』という報告だけ標準エラーに流してくれる（それが 1〜2 行目。`report_on_exception` という仕組み）。メインは何事もなく続行。`t.status` が `nil` なのは『例外死』の印。──そして `t.join` した瞬間、**死んだ本人の例外が、join した側に投げ直される**。waitpid が死亡届を受け取るように、join は例外ごと受け取るの」

「じゃあ、join しなかったら？」

「例外は**どこにも届かない**。標準エラーに 2 行流れるだけで、プロセスは正常終了する。──想像して。送信ワーカーのスレッドが 4 本、うち 1 本が開始 3 分で SMTP エラーで死んだ。でも誰も join しない。残り 3 本は元気に働いて、バッチは『正常終了』する。**4 分の 1 のユーザーにメールが届いてないのに、exit code は 0**」

「怖……。落ちてくれた方がマシですね」

「そういう思想の設定もある。`Thread.abort_on_exception = true` にすると、どのスレッドの例外でも即プロセス全体を落とす。潔癖だけど、分かりやすい。いずれにせよ鉄則はこれ──**作ったスレッドは必ず join する。join は「待つ」ためだけじゃなく「エラーを受け取る」ために要る**」

## 3

「じゃあ、PR #1247 の送信本体を見ましょう。ようやく本丸の diff よ」

```ruby
# 送信のスレッド化（4 並列）。users は購読者リスト
i = 0
sent_count = 0

threads = 4.times.map do
  Thread.new do
    while i < users.size
      user = users[i]       # 次のユーザーを取る
      deliver(user)         # メールを送る
      i += 1                # 「次へ」を進める
      sent_count += 1       # 帳簿をつける
    end
  end
end

threads.each(&:join)
notify_slack("daily_digest 完了: 送信 #{sent_count} / 対象 #{users.size}")
```

「4 人の同居人が、**同じ `i`**（次に送る番号）と**同じ `sent_count`**（帳簿）を見ながら働いてる。join はちゃんと全員分ある。えらい」

「素直に読めば、仕事を 4 人で山分けして 4 倍速……ですよね。実際、送信時間は 3 時間から 40 分に縮んでます。ちゃんと動いてるように見える」

「そうね。このコードは一見、何も悪くない。`users[i]` を取って、送って、`i` を進めて、帳簿をつける。1 人でやるなら**完璧に正しいコード**よ。レビューであなたが通したのも責められない。動くもの、実際」

氷室さんはマーカーのキャップを閉めて、僕をまっすぐ見た。

「でもね、深町くん。同居人が 4 人いる家で『冷蔵庫のプリンを取って、食べて、*食べた*ってメモを貼る』を全員が同時にやったら、何が起きると思う？」

「……メモを貼る前に、別の人が冷蔵庫を開けたら」

「そう。**同じプリンに、2 人の手が伸びる**」

背中がぞわりとした。同じユーザーに、2 通のメール。

「気づいた顔ね。でも、まだ逮捕はできないわよ。『起きうる』と『起きた』の間には距離がある。それに疑問も残るでしょう。staging では 3 日間無事だった。本番でも 2 万人中の十数人。**なぜ毎回・全員に起きないのか**。凶器の正体だけじゃなく、**発生条件**まで説明できて初めて解決よ。──月曜に、全部再現する。本番の 2 万人は使わない。**200 人のミニチュア世界**を作って、そこで犯行を再演させるの」

「月曜……週明けまで待つんですか？」

「実験には作法があるの。急いで雑にやると、再現しなかったときに『シロなのか、実験が下手なのか』が区別できなくなる。それにね」

氷室さんは、帰り支度をしながら付け加えた。

「今夜のバッチも、明日の朝のバッチも、たぶん*ほぼ正常に*動くわ。二重送信は確率の犯罪よ。**再現条件を正確に握るまでは、犯人は毎朝、涼しい顔で通常業務をこなし続ける**。──焦らず、確実に。月曜は逮捕の日よ」

## 捜査メモ

- **スレッド = 同じプロセス（家）の中の複数の働き手**。どのプロセスにも最低 1 人、メインスレッドがいる（プロセス = 家 + 働き手のセット）
- fork はメモリを**コピー**（以後独立・協力は手紙）、Thread はメモリを**共有**（変更が即座に全員へ・協力は代入一発）── 実測: 同じ実験で fork は `[]`、Thread は `["user_1"]`
- 共有にも例外がひとつ: **スタック**（実行中メソッドのローカル変数・呼び出し履歴）だけは働き手ごとの私物。共有されるのは**ヒープ**（それ以外のオブジェクト）で、事故が起きるのはヒープ側
- スレッド内の例外は**そのスレッドだけを殺し、プロセスは続行する**（実測: メインは生存、`t.status` は nil）。例外は **join した瞬間に投げ直される**。join しなければ握り潰されて exit 0 ──「静かな欠損」の温床
- 鉄則: **作ったスレッドは必ず join する**（待つためではなく、エラーを受け取るため）。全滅即死型にしたいなら `Thread.abort_on_exception = true`
- PR #1247 の送信本体: 4 スレッドが**共有の `i` と `sent_count`** を読み書きしている。1 人なら完璧、4 人だと──（次話で再現実験）

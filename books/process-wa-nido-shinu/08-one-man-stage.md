## 1

「岸田さんの質問を、まず正確に言い直すわね。──**Ruby（CRuby）には GVL がある。Global VM Lock。Ruby のコードを実行できるスレッドは、1 プロセスにつき常に 1 本だけ**。4 スレッド作っても、CPU の上で Ruby コードを動かしているのは、どの瞬間も 1 人」

木曜の会議室。氷室さんはホワイトボードに舞台の絵を描いた。

「劇場だと思って。スレッドは役者。何人雇ってもいいけど、**舞台には一度に 1 人しか立てない**。これが GVL。だから岸田さんの疑問はもっともなの。『舞台に 1 人しか立てないのに、なんで 4 人雇ったら 4 倍速くなるんだ？』」

「そもそも、なんでそんな制限があるんですか。舞台を広くすればいいのに」

「Ruby 処理系の内部データ──オブジェクトの管理台帳や GC の状態──を、複数スレッドの同時アクセスから守るためよ。皮肉な話でしょう。**GVL 自体が、Ruby 処理系を守るための巨大な 1 本の鍵**なの。昨日あなたが学んだ Mutex の、いちばん大きいやつ。処理系の中身が常に 1 人ずつしか触られないなら、内部の整合性は壊れない。設計として単純で頑丈──その代償が『舞台は 1 人』」

「実測しましょう」と氷室さんは 2 つのファイルを作らせた。

「まず、**計算だけ**の仕事。フィボナッチをわざと重く計算させる。CPU バウンド──CPU を回し続ける仕事の代表よ」

```ruby:cpu_bound.rb
require "benchmark"

def fib(n) = n < 2 ? n : fib(n - 1) + fib(n - 2)

N = 32

t1 = Benchmark.realtime { 4.times { fib(N) } }
puts "直列 4 回:          %.2f 秒" % t1

t2 = Benchmark.realtime do
  4.times.map { Thread.new { fib(N) } }.each(&:join)
end
puts "スレッド 4 本:      %.2f 秒" % t2

t3 = Benchmark.realtime do
  4.times.map { fork { fib(N) } }.each { |pid| Process.waitpid(pid) }
end
puts "プロセス 4 個:      %.2f 秒" % t3
```

```
$ ruby cpu_bound.rb
直列 4 回:          0.67 秒
スレッド 4 本:      0.67 秒
プロセス 4 個:      0.17 秒
```

「スレッド 4 本、**まったく速くなってない**。0.67 秒のまま。……プロセスは 4 分の 1 になってるのに」

「これが GVL の姿よ。計算の仕事では、4 人の役者が舞台の袖で順番待ちしてるだけ。交代の手間すらあるから、下手すると直列より遅くなることもある。一方 fork は**劇場ごと 4 つ建てる**。GVL は 1 プロセスに 1 個──劇場ごとの決まりだから、劇場が 4 つあれば 4 人が同時に演じられる。うちのマシンは 12 コアだから、OS が 4 プロセスを別々のコアに割り当てて、きれいに 4 倍速」

「12 コアあるなら、劇場 12 個ならもっと速い？」

「試しなさい」

```ruby:full_power.rb
require "benchmark"

def fib(n) = n < 2 ? n : fib(n - 1) + fib(n - 2)

t = Benchmark.realtime do
  12.times.map { fork { fib(32) } }.each { |pid| Process.waitpid(pid) }
end
puts "プロセス 12 個で fib(32) × 12: %.2f 秒" % t
```

```
$ ruby full_power.rb
プロセス 12 個で fib(32) × 12: 0.25 秒
```

「12 回分の計算が 0.25 秒。直列なら 0.17 × 12 で 2 秒コースのはずだから……ほぼ 12 倍。**全コアが埋まった**」

「これが物理の限界までスケールする、ということ。**CPU バウンドの仕事を並列にしたければ、スレッドではなくプロセス**。Ruby の鉄則その 1 よ」

## 2

「じゃあ、**待つだけ**の仕事ならどうか。IO バウンド。メール送信の正体はこっちだったわね」

```ruby:io_bound.rb
require "benchmark"

def deliver(user) = sleep(0.5)   # 送信 = 0.5 秒の IO 待ち

t1 = Benchmark.realtime { 4.times { |i| deliver(i) } }
puts "直列 4 件:     %.2f 秒" % t1

t2 = Benchmark.realtime do
  4.times.map { |i| Thread.new { deliver(i) } }.each(&:join)
end
puts "スレッド 4 本: %.2f 秒" % t2
```

```
$ ruby io_bound.rb
直列 4 件:     2.02 秒
スレッド 4 本: 0.50 秒
```

「今度は**4 倍速い**。0.5 秒×4 件が、0.5 秒で終わってる」

「種明かし。GVL は『**Ruby のコードを実行する権利**』の鍵なの。IO 待ち──ネットワークの返事待ち、ディスク待ち、sleep──の間、そのスレッドは Ruby のコードを実行してない。ただ待ってるだけ。だから Ruby は、**IO に入るとき GVL を手放す**。舞台に立ってても台詞がないなら、袖にはけて次の役者に舞台を譲るの」

| 時刻 | Thread A | Thread B | Thread C | Thread D |
| --- | --- | --- | --- | --- |
| 0.0 秒 | 送信開始→**待ち** | 送信開始→**待ち** | 送信開始→**待ち** | 送信開始→**待ち** |
| 0.5 秒 | 完了 | 完了 | 完了 | 完了 |

「待ち時間は重ねられる。しかも役者を増やすのは安い。20 人ならどうなると思う？」

```ruby:io_20.rb
require "benchmark"

t = Benchmark.realtime do
  20.times.map { Thread.new { sleep 0.5 } }.each(&:join)
end
puts "20 件の送信(0.5秒)をスレッド 20 本で: %.2f 秒" % t
```

```
$ ruby io_20.rb
20 件の送信(0.5秒)をスレッド 20 本で: 0.51 秒
```

「20 件でも 0.51 秒。直列なら 10 秒かかるのに」

「舞台が 1 つでも、**楽屋で待つのは何人いてもタダ同然**だから。実際、雇うコスト自体も測っておきましょう」

```ruby:hiring_cost.rb
require "benchmark"

t1 = Benchmark.realtime do
  100.times.map { Thread.new {} }.each(&:join)
end
puts "スレッド 100 本の起動と終了: %.4f 秒" % t1

t2 = Benchmark.realtime do
  100.times.map { fork {} }.each { |pid| Process.waitpid(pid) }
end
puts "プロセス 100 個の起動と終了: %.4f 秒" % t2
```

```
$ ruby hiring_cost.rb
スレッド 100 本の起動と終了: 0.0035 秒
プロセス 100 個の起動と終了: 0.0644 秒
```

「約 18 倍。スレッドの起用は圧倒的に軽い。──だから岸田さんの設計判断は、実は筋が良かったのよ。メール送信はほぼ全部が待ち時間。IO バウンドの仕事に、軽いスレッドを 4 本。教科書どおり。3 時間が 40 分になったのは、GVL の**例外規定**──IO 待ちでは鍵を手放す──のおかげ。**鉄則その 2: IO バウンドの仕事はスレッドで重ねる**」

「じゃあ、岸田さんは何も間違ってなかった……」

「並列化の*方式*はね。間違ってたのは共有データの扱いだけ。──そして皮肉なことに、深町くん。**その同じ例外規定が、レースコンディションの舞台装置でもあった**の。第 6 話の宿題を回収するわよ。純 CPU の `counter += 1` ループは、なぜ 100 万ぴったりだった？」

「……GVL を手放すのは IO のとき。純 CPU ループには IO がないから、読む→足す→書くの 3 手の途中で舞台を降りることが、ほとんどない」

「そう（正確には CPU だけでも一定時間で強制交代はあるけど、急所に入る確率が低い）。そして `deliver`──IO──を挟んだ瞬間、**3 手のど真ん中で必ず舞台を降りる**ようになった。だから毎回確実に壊れた。GVL を手放す瞬間こそが、スレッドが速くなる瞬間であり、割り込まれる瞬間。**性能の源泉と事件の温床が、同じ 1 つの仕組み**だった。──これが今回の事件の、いちばん深い構図よ」

## 3

「整理しましょう。使い分けの地図はこう」

| 仕事の種類 | 正体 | 効く並列化 | 理由 |
| --- | --- | --- | --- |
| CPU バウンド | 計算し続ける（画像処理、集計、暗号化…） | **プロセス**（fork） | GVL は劇場に 1 つ。劇場ごと増やすしかない |
| IO バウンド | 待ち続ける（API、DB、メール、ファイル…） | **スレッド**で十分 | 待ち時間は GVL を手放すので重ねられる。しかも軽い |

「Puma が『worker（プロセス）× threads（スレッド）』の 2 段構えなのも、この地図の通りよ。リクエスト処理は DB 待ちが多いからスレッドで重ね、CPU をコア数ぶん使い切るためにプロセスを並べる。`WEB_CONCURRENCY`（worker 数）と `RAILS_MAX_THREADS`、あの 2 つの環境変数はまさに『劇場をいくつ建てるか』と『役者を何人雇うか』。Sidekiq の concurrency も同じ発想。──今日の地図があれば、もうあの設定値を雰囲気で決めなくて済むわね」

「ひとつ気になります。『1 プロセス内では計算は並列にならない』って、Ruby は永遠にそうなんですか？」

「いい質問。実は抜け道がもう作られてる。**Ractor**。劇場の中に、**防音壁で仕切った小劇場**をいくつも作る仕組みよ。GVL は Ractor ごとに 1 つになるから、1 プロセスの中で本当に並列に計算できる。見てみる？」

```ruby:ractor_stage.rb
require "benchmark"

def fib(n) = n < 2 ? n : fib(n - 1) + fib(n - 2)

t = Benchmark.realtime do
  ractors = 4.times.map { Ractor.new { fib(32) } }
  ractors.each(&:value)
end
puts "Ractor 4 つで fib(32) × 4: %.2f 秒" % t
```

```
$ ruby ractor_stage.rb
ractor_stage.rb:6: warning: Ractor API is experimental and may change in future versions of Ruby.
Ractor 4 つで fib(32) × 4: 0.17 秒
```

「0.17 秒！　プロセス 4 個と同じだ。**fork してないのに並列計算できてる**」

「ね。ただし警告が出てるとおり、まだ experimental──API が変わりうる段階。それに防音壁の代償として、**Ractor 同士は普通のオブジェクトを共有できない**。共有できないからこそ鍵なしで並列にできる、という思想なの。気づいた？　この二週間で学んだ構図そのものよ。**共有するから事件が起きる。共有を断てば、鍵も事件も要らない**」

「……それ、明日の『鍵が要らない設計』の話につながってます？」

「察しが良くなってきたじゃない」氷室さんはマーカーを置いた。「Ractor は共有を*言語の壁*で断つ。明日やるのは、もっと身近な道具で共有を*設計*で断つ話。金曜の最終報告、修正版のコードで締めるわよ」

## 捜査メモ

- **GVL（Global VM Lock）**: CRuby では、Ruby コードを実行できるスレッドは 1 プロセスに常に 1 本。正体は**処理系の内部を守る巨大な 1 本の Mutex**。スレッドを何本作っても**計算は並列にならない**（実測: fib×4 が直列 0.67 秒 → 4 スレッド 0.67 秒）
- **プロセスなら計算も並列になる**。GVL は劇場（プロセス）ごとに 1 つ（実測: 4 プロセスで 0.17 秒、12 プロセスで 12 回分が 0.25 秒 ── 12 コアを使い切った）
- **IO 待ちに入るスレッドは GVL を手放す**。IO バウンドはスレッドで重ねられる（実測: 0.5 秒×4 件が 0.50 秒、20 件でも 0.51 秒）。スレッドは作るのも軽い（実測: 100 本 0.0035 秒 vs 100 プロセス 0.0644 秒 ≒ 18 倍差）
- 使い分けの鉄則: **CPU バウンド → プロセス、IO バウンド → スレッド**。Puma の worker × threads（`WEB_CONCURRENCY` × `RAILS_MAX_THREADS`）はこの合わせ技
- GVL を手放す瞬間 = スレッドが速くなる瞬間 = **割り込まれてレースが起きる瞬間**。性能と事件は同じ仕組みの表裏（第 6 話の「純 CPU では壊れなかった」の答え）
- **Ractor**: GVL を Ractor 単位に分割し、1 プロセス内で真の並列計算をする仕組み（実測: 4 Ractor で 0.17 秒 = プロセス並列と同速。ただし experimental 警告付き）。代償は「普通のオブジェクトを共有できない」──**共有を断てば鍵なしで並列にできる**という思想

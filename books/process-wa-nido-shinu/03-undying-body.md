title: "第3話 消えない死体 ── ゾンビプロセスと wait"
---

:::message
コードと出力は Ruby 4.0.5（rbenv / macOS arm64）での実測です。PID は毎回変わります。
:::

## 1

「今日は検死からよ。まず、死体を 1 体作ります」

水曜の朝一番に言う台詞ではないと思う。氷室さんは構わず続けた。

「条件は 2 つ。**子が先に死ぬこと。親がそれを看取らないこと。** 打って」

```ruby:zombie.rb
pid = fork { exit }   # 子は生まれてすぐ死ぬ

sleep 1               # 親は看取らずに放置

puts "子の PID: #{pid}"
system("ps -o pid,stat,command -p #{pid}")
```

```
$ ruby zombie.rb
子の PID: 93246
  PID STAT COMMAND
93246 Z    <defunct>
```

「出た……STAT が `Z`。`<defunct>`。本番で見たのと同じだ。でも変ですよ。子は `exit` してるんです。**死んだのに、なんで ps に写るんですか**」

「じゃあ聞くけど、深町くん。人が亡くなったら、その瞬間に戸籍も消える？」

「……消えないです。死亡届を出すまで残る」

「プロセスも同じなの。子が exit した瞬間、大きな持ち物は OS がすぐ回収する。メモリも、開いてたファイルも、CPU を使う権利も、全部。だから死んだ子は 1 バイトも仕事をしないし、ほとんど場所も取らない。でもね、**1 行だけ消せない記録がある**。『この子は、どう死んだか』──終了ステータスよ。正常終了の 0 だったのか、異常終了だったのか」

「誰のために残すんですか」

「**親のため**。子の死に方を知る権利と義務があるのは親だけ。だから OS は、親が確認しに来るまで、プロセス表にその 1 行を残し続ける。この『死んだけど、死亡確認待ちの 1 行』が──**ゾンビプロセス**」

「じゃあゾンビの正体って、動き回る怪物じゃなくて……」

「ただの、**読まれていない死亡届**。1 体なら無害よ。──1 体ならね。第 1 話で見たでしょ、`kern.maxproc: 12000`。プロセス表には定員がある。そしてゾンビは、死んでいるくせに**律儀に 1 席使い続ける**の。wait しない fork をループで回すデーモンがいたら、席は静かに埋まっていって、ある日突然、サーバー全体が新しいプロセスを作れなくなる。`fork failed: Resource temporarily unavailable`──そうなってからでは、ssh のログインさえ怪しくなる。死体は無害。**死体の山は事故**よ」

## 2

「で、親はどうやって死亡確認するんですか」

「`Process.wait`。昨日の fork2.rb にも書いた `waitpid` の仲間ね。今度は看取るバージョンを打って」

```ruby:wait.rb
pid = fork { exit }

Process.waitpid(pid)  # 子の死亡を確認する（死亡届を受け取る）

puts "子の PID: #{pid}"
system("ps -o pid,stat,command -p #{pid}")
puts "ps の終了コード: #{$?.exitstatus}"
```

```
$ ruby wait.rb
子の PID: 93274
  PID STAT COMMAND
ps の終了コード: 1
```

「ps に……何も出ない。終了コード 1 は『そんなプロセスはいない』。**今度は完全に消えてる**」

「`waitpid` が死亡届を受け取った瞬間、OS は最後の 1 行を消せる。これで成仏。ちなみに死亡届の中身も読めるわよ」

```ruby:wait2.rb
pid = fork { exit 42 }          # わざと 42 で死ぬ

Process.waitpid(pid)
puts "子の終わり方: #{$?}"
puts "終了コード:   #{$?.exitstatus}"
```

```
$ ruby wait2.rb
子の終わり方: pid 97898 exit 42
終了コード:   42
```

「`$?` に、どの子がどう死んだかが入る。子が失敗してたらリトライする、通知する、諦める──親の責任はこの死亡届を読むところから始まるの。実はあなた、これも昨日から使ってるのよ。`system("ps ...")` のあとの `$?.exitstatus`。system は fork + exec + wait だって言ったでしょ。**あの終了コードは、ps に変身した子の死亡届**だったの」

伏線が、また一本つながった。

## 3

「もうひとつ、第 1 話の宿題を回収するわよ。**PID 1 の大事な役目**。──親が子より先に死んだら、子はどうなると思う？」

「死亡届の受け取り手がいなくなる……永遠にゾンビ？」

「実験しましょう。今度は逆パターン。**親が先に死んで、子が残る**」

```ruby:orphan.rb
fork do
  puts "子: 生まれた直後の親は #{Process.ppid}"
  sleep 1                              # その間に親が死ぬ
  puts "子: いまの親は #{Process.ppid}"
end

puts "親: #{Process.pid} は先に退場する"
# 親は wait せずにそのまま終了
```

```
$ ruby orphan.rb
親: 26288 は先に退場する
子: 生まれた直後の親は 26288
子: いまの親は 1
```

「親が 26288 から……**1 に変わってる**。launchd だ。家系図の頂点の」

「そう。親を失った子──**孤児プロセス**は、放置されない。**PID 1 が自動的に里親になる**の。そして PID 1 は職業的な看取り屋でもある。引き取った子が死ねば、即座に死亡届を処理してくれる。これが第 1 話で予告した役目。ご先祖様は、身寄りのない子たちの世話係でもあるのよ」

「あ……それ、本番の数字の説明になりませんか。バッチは毎朝 20 回 fork するのに、ゾンビは 17 体。それにプロローグで見た監視グラフだと、**バッチが終わる頃にゾンビが消えてる**」

「つながったわね。バッチ本体（親）が 40 分の仕事を終えて exit した瞬間、残ってたゾンビたちは孤児になって PID 1 に引き取られ、まとめて成仏する。だから翌朝まで持ち越さない。17 という数字も『あなたが ps を打った瞬間に、たまたま溜まってた数』というだけ。20 回 fork の途中経過よ。──つまりうちの事件は、幸運にも**毎朝リセットのかかる死体の山**だった。これがもし常駐デーモンだったら、と考えると背筋が冷えるでしょう」

## 4

「直し方の話をしましょう。PR #1247 のレポート処理、もう一度見て」

```ruby
# Slack API が遅くても送信を止めないよう、レポートは別プロセスで投げる
fork do
  notify_slack("#{sent_count} 件送信済み")
end
```

「fork して、そのまま次の送信ループに戻ってる。**wait がどこにもない**」

「岸田さんの狙い自体は正しいのよ。Slack が 3 秒詰まっても、本体の送信は止めたくない。でも、**産んだ子を看取る処理を書き忘れた**。──さて、ここで単純に `fork` の直後に `Process.waitpid` を書くと？」

「親が Slack 投稿の完了を待っちゃう。『送信を止めない』という狙いが死にます」

「そう。**看取りたい、でも待ちたくない**。このジレンマには定番の道具が 3 つある。全部実測するわよ」

「1 つ目、**`WNOHANG`**。『死んでたら片付ける、まだなら待たずに帰る』という巡回モード」

```ruby:patrol.rb
pid = fork { sleep 0.5 }   # 0.5 秒働いて死ぬ子

3.times do |n|
  reaped = Process.waitpid(pid, Process::WNOHANG)  # 待たずに聞くだけ
  puts "巡回#{n + 1}回目: #{reaped ? "看取った" : "まだ働いている"}"
  sleep 0.3
end
```

```
$ ruby patrol.rb
巡回1回目: まだ働いている
巡回2回目: まだ働いている
巡回3回目: 看取った
```

「普通の `waitpid` は子が死ぬまでそこで止まるけど、`WNOHANG` 付きは即答してくれる。生きてれば nil、死んでれば片付けて PID を返す。送信ループの合間にこれを差し込めば、仕事を止めずに死亡届を回収できる」

「2 つ目、**`Process.detach`**。『看取り係』を雇って、自分は先へ進む」

```ruby:detach.rb
pid = fork { exit }
Process.detach(pid)   # 看取り係を立てて、自分は先へ進む

sleep 1
system("ps -o pid,stat -p #{pid}")
puts "ps の終了コード: #{$?.exitstatus}（1 = もういない）"
```

```
$ ruby detach.rb
  PID STAT
ps の終了コード: 1（1 = もういない）
```

「zombie.rb と同じ構図なのに、1 秒後にはもう死体がない。detach は裏で小さな見張りを立てて、子が死んだ瞬間に wait してくれるの。『この子の死に方には興味がない、でも死体は残すな』というときの道具。今回のレポート処理には、これが一番手軽ね」

「3 つ目は？」

「**そもそも fork をやめる**。`Thread.new { notify_slack(...) }`。Slack に文字列をひとつ投げるだけの仕事に、プロセス丸ごとの分身は大げさすぎるのよ。同じ家の中に、もうひとり働き手を増やせば足りる。……ただし」

氷室さんはそこで言葉を切って、僕を見た。

「スレッドには、fork にはない**別の危険**がある。それを知らずに『軽いから』でスレッドを選ぶと──たぶん、うちの本丸の事件みたいなことになる」

「本丸……怪事件①。二重送信」

「そう。ここまでで解けたのは怪事件②（ログ二重）と③（ゾンビ）。どっちも fork の犯行だった。でもね、実は fork には、**①と④だけは絶対に犯せない理由**があるの。明日はそのアリバイ調べ。fork という容疑者を、正式にシロにする回よ」

（第4話「双子のアリバイ ── fork とメモリの独立」につづく）

## 捜査メモ

- 子プロセスが exit しても、**終了ステータス（死亡届）だけはプロセス表に残る**。親が受け取るまで消えない ── これが**ゾンビプロセス**（STAT `Z` / `<defunct>`）
- ゾンビはメモリも CPU もほぼ使わないが、**プロセス表の 1 席を占有し続ける**（定員は実測 12,000）。wait しない fork の放置は、いつか `fork failed` でサーバー全体を巻き込む
- 死亡届の受け取り: `Process.wait` / `Process.waitpid(pid)`。内容は `$?`（`$?.exitstatus`）で読める。`system` の後の `$?` も同じもの（system = fork + exec + wait）
- **親が先に死ぬと子は孤児になり、PID 1 が里親になる**（実測: `Process.ppid` が 26288 → 1 に変化）。PID 1 は引き取った子の死亡届を自動処理する ──「毎朝 20 fork でゾンビ 17 体、バッチ終了で全部消える」の完全な説明
- **看取りたいが待ちたくない**ときの道具 3 つ（すべて実測済み）:
  1. `Process.waitpid(pid, Process::WNOHANG)` ── 待たずに聞く巡回（生きてれば nil）
  2. `Process.detach(pid)` ── 看取り係を立てて先へ進む（手軽さ最優先ならこれ）
  3. そもそも fork ではなく **Thread** にする（ただしスレッド固有の危険は第 5 話以降）
- 解決済み: 怪事件②③。未解決: ①二重送信、④送信数 257 不足

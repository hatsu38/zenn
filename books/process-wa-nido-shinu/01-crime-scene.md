コードと出力はすべて Ruby 4.0.5（rbenv / macOS arm64）での実測です。PID は OS が採番するため、**実行するたびに・環境ごとに必ず違う数字**になります。数字そのものではなく、数字の*関係*に注目してください。

## 1

「まず言葉を揃えましょう。深町くん、**プロセス**って何？」

会議室のホワイトボードの前で、氷室さんが言った。捜査会議の第一声がこれだ。

「え……実行中のプログラム、ですか」

「70 点。じゃあ質問を変える。同じ Ruby のバッチを 2 回起動したら、プロセスは何個？」

「2 個……ですよね」

「そう。ここが大事。**プログラムはレシピ、プロセスは実際に作った料理**。レシピは 1 枚でも、料理は作った回数だけ皿に載る。そしてそれぞれの皿は独立している。片方に塩を足しても、もう片方の味は変わらない」

「レシピと料理……つまり、`daily_digest.rb` というファイルがレシピで、毎朝 6 時にそれが料理される、と」

「そういうこと。ディスクの上の `daily_digest.rb` は、ただの文字列よ。実行もされなければ、メモリも持たない。cron がそれを『起動』した瞬間に、OS が席をひとつ用意する──**メモリの割り当てと、CPU を使う権利と、開いたファイルの管理台帳**。そのひとそろいがプロセス。だから同じレシピから何皿でも作れるし、皿ごとに中身は別」

氷室さんはホワイトボードに書いた。

- プログラム = ディスク上のコード（レシピ）。それ自体は動かない
- プロセス = 実行中のプログラム 1 つぶん。**自分専用のメモリ**と、OS の管理台帳の 1 席を持つ
- OS はプロセスごとに背番号を振る = **PID**（Process ID）

「そして PID は、現場に残る指紋よ。打ってみて。3 行目の `ps` は現場検証の道具──Process Status の略で、プロセスの一覧や状態を見せてくれるコマンド。自分で自分を検証させてるの」

```ruby:pid.rb
puts "PID: #{Process.pid}"
puts "親PID: #{Process.ppid}"
system("ps -o pid,ppid,stat,command -p #{Process.pid}")
```

```
$ ruby pid.rb
PID: 3930
親PID: 3927
  PID  PPID STAT COMMAND
 3930  3927 S    ruby pid.rb
```

「PID が 3930。……この『親 PID』っていうのは？」

「いい質問。プロセスは無から湧かない。**必ず別のプロセスから生まれる**の。あなたがいまターミナルで `ruby` と打ったから、シェルというプロセスが Ruby のプロセスを産んだ。3927 は、そのシェルの PID。試しにもう一回打ってみて」

```
$ ruby pid.rb
PID: 3943
親PID: 3927
  PID  PPID STAT COMMAND
 3943  3927 S    ruby pid.rb
```

「PID は変わった。親は同じ……あ、そうか。同じシェルから起動してるから」

「そういうこと。同じ親から、別の子が生まれた。1 回の実行は 1 個のプロセスで、**プロセスは使い捨て**。前回の 3930 という席はもう空席で、いずれ別の誰かに使い回される。──指紋と言ったけど、正確には『在任中の背番号』ね。永久欠番じゃない」

## 2

「プロセスに家系図があるなら、辿れるんですか？　親の親、そのまた親、みたいに」

「辿れるわよ。やってみましょう。`ppid` を繰り返し引くだけ」

```ruby:family_tree.rb
pid = Process.pid

while pid > 0
  info = `ps -o pid=,ppid=,comm= -p #{pid}`.strip
  break if info.empty?
  puts info
  pid = info.split[1].to_i
end
```

```
$ ruby family_tree.rb
26092 26089 /Users/.../.rbenv/versions/4.0.5/bin/ruby
26089 18783 /bin/zsh
（…あなたのターミナル環境の家系がしばらく続く…）
1     0 /sbin/launchd
```

「ruby の親が zsh。そこから上は、ターミナルアプリの系譜が続いて……最後、PID **1** の launchd？」

「そこが家系図の頂点。OS が起動して最初に生まれるプロセスが PID 1（macOS では launchd、Linux では init や systemd）。**この星のすべてのプロセスは、先祖を辿ると必ず PID 1 に行き着く**。ちなみにこの PID 1、ただのご先祖様じゃなくて、ある大事な役目を持ってる。それは第 3 話でね」

「プロセスって、いくらでも作れるんですか？」

「いい着眼。有限よ。OS の管理台帳には定員がある」

```
$ sysctl kern.maxproc
kern.maxproc: 12000
```

「このマシンだと 12,000 席。潤沢に見えるでしょう。でもね、**席は『生きてるプロセス』だけが使うとは限らない**の。これも第 3 話の伏線。覚えておいて」

## 3

「`ps` の出力の読み方も揃えておくわよ。PID・PPID・COMMAND はいいわね。残る **STAT がプロセスの容態**」

| STAT | 意味 |
| --- | --- |
| `R` | 実行中 / 実行待ち。CPU の上で元気に働いている |
| `S` | スリープ。何かを待っている（IO の完了、タイマー、誰かの終了…） |
| `T` | 停止中（一時停止させられている） |
| `Z` | **ゾンビ**。死んでいるのに、消えていない |

「さっきの実測で `S` だったのは、`system` で ps の結果を待ってたから。サーバーのプロセスは大半が `S` よ。Web サーバーもワーカーも、実はほとんどの時間を『待ち』で過ごしてる。この『プロセスは待つのが仕事』という感覚、後半でものすごく効いてくるから、頭の隅に置いておいて」

「で──`Z`。プロローグであなたが見たやつね。あれは容態『死亡。ただし埋葬待ち』。詳しくは第 3 話でやる」

「もうひとつ質問いいですか。プロセスが『自分専用のメモリを持つ』って言いましたよね。それって具体的には……」

「変数よ。あなたが Ruby で作る変数も配列もハッシュも、ぜんぶそのプロセス専用のメモリの中にいる。**他のプロセスからは見えないし、触れない**。さっき pid.rb を 2 回動かしたでしょ。もし 1 回目の実行で作った変数が 2 回目から見えたら、世界中の Ruby スクリプトが大惨事よ。隣の皿の料理には手を出せない。──これ、いま普通のことに聞こえるでしょ。この『当たり前』が、第 4 話で事件の鍵になる」

## 4

「じゃあ現場に戻りましょう。うちのバッチのログには、各行に PID が入る。二重になってた `[START]` の行、PID は何だった？」

僕は月曜のメモを開いた。ここに、今朝からずっと引っかかっている謎がある。

```
[41001] [START] daily_digest 開始 対象: 20000 件
[41001] [INFO]  worker 起動
[41001] [START] daily_digest 開始 対象: 20000 件
[41001] [INFO]  1000 件送信済み
```

「**両方とも 41001**。同じプロセスなんです。それが変で……。START を書くコードは 1 箇所しかないし、ループの外です。起動記録も 1 回。**同じプロセスが、1 回しか通らないはずの行を 2 回書いてる**。ありえなくないですか？」

「ふうん。『同じ指紋が、ありえない場所に 2 つ』ね。……いい証拠だけど、それ単体ではまだ解けない。もうひとつの現場と突き合わせるわよ。あなたが見つけた `<defunct>` の死体たち。**あの子たちの家系図**は調べた？」

「家系図……PPID ですか。いえ、STAT しか見てませんでした」

「現場検証の基本よ。死体が 17 も転がってたら、まず身元と血縁を洗う」

サーバーの監視ログから、今朝 6 時台の `ps` のスナップショットを引っ張り出した。PPID の列を足して眺める。

```
  PID  PPID STAT COMMAND
41283 41001 Z    <defunct>
41156 41001 Z    <defunct>
41022 41001 Z    <defunct>
（以下、14 行。PPID はすべて 41001）
```

血の気が引いた。

「親が……**全員 41001**。ゾンビ 17 体の親、全部うちのバッチ本体です」

「はい、家系図が吐いたわね。整理するわよ。cron が産んだのは 41001 ただ 1 人。これは起動記録とロックファイルが保証してる。でもその 41001 が、**自分の中で子プロセスを 17 回以上産んでる**。監視の起動記録に残らないのは当然よ。外から誰も起動してないんだから。──2 人目以降の登場人物は、外から来たんじゃない。**中で生まれたの**」

「プログラムが実行中に自分の分身を作るなんて、そんなことできるんですか」

「できるわ。たった 1 行で。むしろね、深町くん。**あなたは今日、もうそれを 3 回見てる**」

「え？」

「`system("ps ...")` よ。Ruby が ps コマンドを実行できたのはなぜ？　ps は Ruby じゃない、別のプログラムよ。あれはね、Ruby のプロセスが**子プロセスを産んで**、その子に ps を実行させて、終わるのを待ってたの。あなたはさっきから、プロセスがプロセスを産む現場を眺めながら『そんなことできるんですか』って言ってる」

言われて、鳥肌が立った。`ruby pid.rb` の出力の中の `ps` は、たしかに──誰が動かしたんだ？

「その『産む』の正体が、あなたが approve した PR #1247 の diff にも入ってる」

氷室さんはホワイトボードに、4 文字だけ書いた。

`fork`

「明日はこれの取り調べ。分身がどう生まれるかが分かれば、『同じ指紋の START が 2 つ』の謎も、たぶん一緒に解ける。──言ったでしょ、プロセスは嘘をつかない。**産ませてみれば分かる**」

## 捜査メモ

- **プログラムはレシピ、プロセスは料理**。同じプログラムを 2 回動かせばプロセスは 2 個。1 回の実行 = 1 個の使い捨てのプロセス
- プロセス = 実行中のプログラム 1 つぶん。**自分専用のメモリ** + OS の管理台帳の 1 席。台帳には定員がある（実測: `kern.maxproc` = 12,000）
- プロセスの指紋 = **PID**（`Process.pid`）。プロセスは必ず**親から生まれる**（親は `Process.ppid`）。家系図を遡ると必ず **PID 1**（launchd / init）に行き着く
- 現場検証の道具は `ps`。`ps -o pid,ppid,stat,command` で指紋・親・容態・正体が見える
- STAT の読み方: `R` 実行中 / `S` 待機中（サーバーのプロセスは大半これ）/ `Z` **ゾンビ（死亡・埋葬待ち）**
- `system("...")` は、実は**子プロセスを産んで**コマンドを実行させている（何気なく使っていた「分身」の実例）
- 証拠 A: 二重の `[START]` は**どちらも同じ PID 41001**。1 回しか通らないはずのコードを、同じプロセスが 2 回書いた……ように見える
- 証拠 B: ゾンビ 17 体の **PPID はすべて 41001**（バッチ本体）→ バッチは実行中に**自分で子プロセスを産んでいた**。手段は `fork`（次話）

## 1

木曜の捜査会議は、僕の推理の発表から始まった。

「怪事件①、二重送信の犯人も fork だと思うんです。筋はこうです。──送信ループの途中でレポート用の fork が走る。子は親の完全なコピーだから、**送信ループごとコピーされる**。それで親と子が同じユーザーに送った。どうですか」

我ながら筋が通っていると思った。ログ二重と同じ理屈だ。バッファがコピーされるなら、送信リストだってコピーされる。実行位置までコピーされるなら、ループの続きだって実行するかもしれない。

氷室さんは、否定も肯定もしなかった。

「仮説としては上出来。じゃあ、証拠は？」

「これから集めます」

「よろしい。じゃあその仮説、**実験で潰しにいくわよ**。いい、深町くん。推理はね、正しいことを確かめるんじゃない。**間違える方法を全部試して、それでも生き残ったものだけを信じる**の」

## 2

「あなたの仮説が正しいなら、こういうことが起きてるはず。『子プロセスでの変更や動きが、親の世界に影響する』。あるいはその逆。──ちょうどいい題材があるじゃない。**送信済みリスト**。fork をまたいで共有されるのか、実験して」

```ruby:fork_memory.rb
sent_flags = []

pid = fork do
  sent_flags << "user_1"
  puts "子: sent_flags = #{sent_flags.inspect}"
end

Process.waitpid(pid)
puts "親: sent_flags = #{sent_flags.inspect}"
```

子で「user_1 に送信済み」の印をつけて、親からその印が見えるかを確かめる。実行する。

```
$ ruby fork_memory.rb
子: sent_flags = ["user_1"]
親: sent_flags = []
```

「……空だ。子はたしかに `["user_1"]` になってるのに、**親は `[]` のまま**」

「これが第 1 話で予告した『当たり前』の正体よ。**プロセスは自分専用のメモリを持つ**。fork の瞬間、子は親のメモリの*コピー*をもらう。コピーよ。共有じゃない。生まれたあとは完全に赤の他人。子が自分のコピーに何を書き込もうと、親のメモリは 1 ビットも変わらない」

「昨日までは『生まれる前の記憶は持ち出す』って話でしたよね。ログのバッファみたいに」

「そう。**生まれる前の記憶は持ち出す。生まれたあとの記憶は共有しない。** この 2 つはセットで覚えて。第 2 話のログ二重は前者、今日のこれは後者。──もっと不気味な証拠も見せてあげる。`object_id` って知ってる？」

「オブジェクトの識別番号ですよね。同じオブジェクトなら同じ番号」

「その『同じなら同じ』が、プロセスの壁を越えると崩れるの」

```ruby:same_id.rb
sent_flags = []
puts "親: object_id = #{sent_flags.object_id}"

pid = fork do
  sent_flags << "user_1"
  puts "子: object_id = #{sent_flags.object_id}, 中身 = #{sent_flags.inspect}"
end

Process.waitpid(pid)
puts "親: object_id = #{sent_flags.object_id}, 中身 = #{sent_flags.inspect}"
```

```
$ ruby same_id.rb
親: object_id = 16
子: object_id = 16, 中身 = ["user_1"]
親: object_id = 16, 中身 = []
```

しばらく出力を睨んだ。

「object_id は……**両方 16**。完全に同じ。なのに中身が違う。『同じオブジェクト』が、違う中身……？」

「不気味でしょう。種はこう。object_id も、その元になる管理情報も、**全部メモリの中の値**なの。だから fork で丸ごとコピーされる。子の世界にも『object_id 16 の配列』が生まれる。でも、それは親の配列と**同じ番地に住む別人**。プロセスにはそれぞれ自分専用のメモリ空間があって、**番地は世界ごとのローカルな住所**なのよ。A 市の中央通り 1 丁目と B 市の中央通り 1 丁目が別の家であるように、親の 16 番と子の 16 番は別の実体。──この『プロセスごとに独立した住所の世界』を、OS の言葉で**仮想メモリ**という。深入りは別の事件簿に譲るけど、fork の独立性はこの仕組みが土台になってる」

「じゃあ、『メモリ丸ごとコピー』って、2GB の Rails を fork したら 2GB 余分に食うんですか」

「実際の OS はもっと賢い。**Copy on Write（書き込み時コピー）**という遅延戦術を使う。fork 直後は、親子で同じメモリの実体をこっそり共有しておいて、**どちらかが書き込もうとした瞬間に、そのページだけ本当にコピーする**。だから fork 自体は一瞬だし、読むだけのメモリは倍にならない。Puma や Unicorn が『アプリを 1 回ロードしてから worker を fork で増やす』のは、この性質のおかげ。──ただし論理的な意味は変わらないわよ。書いた瞬間に自分専用になるんだから、**『子の変更は親から見えない』というルールは 1 ミリも揺らがない**」

## 3

「さて、実験結果をあなたの仮説に突きつけましょう。子が送信ループを回したとして──親の世界の送信済み情報には、何も反映されない。ここまではいい？」

「はい。……あれ、でもそれって、僕の仮説だとむしろ好都合じゃないですか？　子が勝手に送った分を親は知らないんだから、二重に送れる」

「いい粘りね。じゃあ物証と突き合わせる。実際に本番で 2 通届いたメール、送信時刻を見た？」

CS の問い合わせに添付されていたヘッダを開いた。

「1 通目が 6:14:03、2 通目が 6:14:05。……送信は 6 時 3 分から始まってます。**fork される前じゃない。送信ループが始まってだいぶ経ってから**」

「そこで考えて。レポートの fork は 1,000 通ごと。仮に 6:14 直前の fork の子が、コピーされたループの続きを回したとしましょう。子は親と**同じ位置**から、**同じ順番**でループを再開する。そうしたら二重に届くのは 3 人どころじゃない。**そこから先の何千人、全員に 2 通届く**はずよ。子は途中で止まる理由がないもの。でも実際の被害は？」

「……確認できてるのは 3 人。サポートに来てないだけかもと思って送信ログを突き合わせたんですけど、二重になってたのは全部で**十数人**でした。しかも user ID が飛び飛び」

「はい、ここで犯行の**手口の形**が合わなくなった。もし『ループごと分身』なら被害は『ある時点から先、連続で全員』という形になる。実際の被害は『ごく少数が、飛び飛びに』。連続犯の手口と、散発犯の手口。まるで違う」

「それに、そもそも……」自分で言いながら気づいた。「PR のレポート fork は `notify_slack` を呼ぶだけの `fork do ... end` です。ブロックが終わったら子は死ぬ。**送信ループには戻れない**んだ。ブロック付き fork って、そういう意味でも安全なんですね」

「そういうこと。仮説は二重に死んだ。**fork 犯人説、正式に棄却**。──おめでとう、深町くん。あなたはいま、自分の推理を自分の実験で殺した。それができる人から順に、障害対応は上手くなるのよ」

悔しさより、すっきりした感覚のほうが強かった。潰れた仮説のぶんだけ、真相の輪郭が細くなる。

## 4

「ところで氷室さん、ひとつ引っかかってることがあって。プロセスのメモリが完全に独立してるなら……世の中のプログラムって、どうやって協力してるんですか？　親子で結果の受け渡しくらい、したいですよね」

「いい質問。独立には**公式の抜け道**が用意されてるの。代表が**パイプ**。プロセスとプロセスの間に渡す、一方通行の土管よ」

```ruby:pipe.rb
reader, writer = IO.pipe

pid = fork do
  reader.close
  writer.puts "user_1 に送信完了"   # 手紙を書く
  writer.close
end

writer.close
puts "親が受け取った手紙: #{reader.gets.chomp}"
reader.close
Process.waitpid(pid)
```

```
$ ruby pipe.rb
親が受け取った手紙: user_1 に送信完了
```

「子の言葉が、親に届いた。メモリは独立してるのに」

「パイプは『メモリの共有』じゃなくて『**手紙のやりとり**』なの。書き手が書いたバイト列を、OS が預かって、読み手に渡す。受け渡しの仲介を OS がやるから、お互いのメモリには一切触らない。ターミナルの `ps aux | grep digest` の `|`──あれもこのパイプ。ps の出力が grep への手紙になってる」

「独立したまま、協力できる……」

「そう。そしてここからが明日への橋よ。プロセスの協力は**手紙**。手間はかかるけど、相手の部屋を荒らす事故は絶対に起きない。──でも世の中には、もっと手っ取り早い協力の形がある。**同じ部屋に住んで、同じ黒板に直接書き合う**の。速いわよ。手紙なんて目じゃない。その代わり」

氷室さんはマーカーを置いた。

「黒板の前で、**手がぶつかる**」

「……それが、スレッド」

「そう。PR #1247 に入ってた、もうひとつの並列化。`Thread.new`。二重送信とカウント消失──**メモリが独立した fork には絶対にできない犯行**が、同じメモリを触れる同居人になら、できる。明日はその生態から。本丸に入るわよ」

（第5話「同居人 ── スレッドと共有メモリ」につづく）

## 捜査メモ

- fork の子は親のメモリの**コピー**をもらう。**共有ではない**。生まれたあとは、子の変更は親に一切見えない（実測: 子 `["user_1"]` / 親 `[]`）
- セットで記憶: 「**生まれる前の記憶は持ち出す**（ログバッファ＝第 2 話）/ **生まれたあとの記憶は共有しない**（今回）」
- `object_id` まで含めてコピーされる ── 親子で**同じ 16 を名乗る別人**が生まれる（実測）。プロセスごとにメモリの住所空間が独立しているため（**仮想メモリ**）。番地は世界ごとのローカルな住所
- 実際のコピーは **Copy on Write**: fork 直後は実体を共有し、書き込まれたページだけ本当にコピー。fork が一瞬で済み、Puma/Unicorn の worker 方式が成立する理由。ただし「見え方の独立」は完全に保たれる
- 推理の作法: 仮説は**実験で潰しにいく**。被害の「形」（連続 vs 飛び飛び）が手口を絞り込む。ブロック付き fork の子はブロックの外へ出られない、という構造上のアリバイもあった
- プロセス間の公式な協力手段 = **パイプ**（`IO.pipe`）。メモリを共有せず、OS 仲介の「手紙」で受け渡す。シェルの `|` も同じ仕組み
- 二重送信 + **カウント消失**は、メモリが独立した fork には不可能な犯行 → 容疑は**同じメモリを共有する Thread** へ

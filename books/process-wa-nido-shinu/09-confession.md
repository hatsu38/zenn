## 1

金曜、最終報告の前に、氷室さんと最後のコードレビューをした。

「第 7 話の Mutex 版でも事件は防げてる。でも、あれには構造的な弱点が残ってるの。分かる？」

「……鍵のかけ忘れ、ですか。今回は `i` を守った。でも半年後に誰かが送信ループに『共有の再送リスト』とか足したら、その人がまた鍵をかけ忘れるかもしれない」

「そう。Mutex は**規律**なの。全員が、未来永劫、守り続けなければいけない規律。規律で守るより、**間違えようがない構造**にするほうが強い。──そこで、これが最終形」

```ruby:queue_version.rb
queue = Queue.new
sent  = []

workers = 4.times.map do
  Thread.new do
    while (user = queue.pop)   # pop は仕事が来るまで待つ。nil で退勤
      sleep 0.001              # 送信（IO）
      sent << user
    end
  end
end

(1..200).each { |n| queue.push("user_#{n}") }
4.times { queue.push(nil) }    # 「今日はここまで」の合図を人数分

workers.each(&:join)
puts "送信件数:   #{sent.size}"
puts "ユーザー数: #{sent.uniq.size}"
puts "二重送信:   #{sent.tally.count { |_, c| c > 1 }} 人"
```

```
$ ruby queue_version.rb
送信件数:   200
ユーザー数: 200
二重送信:   0 人
```

「**Queue**。Ruby 標準の、スレッドセーフな仕事の窓口よ。仕組みは 3 つだけ」

- `queue.push(仕事)` ── 窓口に仕事を積む
- `queue.pop` ── 仕事を 1 件取る。**同じ 1 件が 2 人に渡ることは絶対にない**（Queue の内部が保証する）。仕事がなければ来るまで待つ
- 終わらせ方 ── 働き手の人数ぶん `nil` を積む（pop した nil が「退勤」の合図）

「気づいた？　このコード、**Mutex が 1 個もない**。共有インデックス `i` も消えた。『次のユーザーは誰か』という危険な帳簿そのものが、設計から消えたの。取り出しの正しさは Queue が内部で保証してくれるから、未来の誰かが規律を破る余地がない」

「レースの温床だった『読む→送る→進める』の 3 手が、`pop` の 1 手になってる……。あ、でも待ってください。Queue の*内部*は誰が守ってるんですか？　複数スレッドが同時に pop したら、内部でレースが起きそうな」

「鋭い。Queue の内部では、結局ロック相当の仕組みが働いてる。つまりね、**鍵が消えたんじゃなくて、鍵の管理を専門家に移譲した**の。あなたが毎回手で書いて毎回間違えうる synchronize を、Ruby 本体が実装してテストし尽くした部品の中に閉じ込めた。『鍵を正しく使う』より『鍵を使う場所を 1 箇所に集めて、その 1 箇所をプロの部品にする』。これが並行設計の基本姿勢よ」

「nil を人数分、っていうのが少し不格好な気もします」

「同じことをもう少し上品にやる道具もある。`close`」

```ruby:close_version.rb（変更部分）
(1..200).each { |n| queue.push("user_#{n}") }
queue.close                    # 「今日の受付は終了」

workers.each(&:join)
```

```
$ ruby close_version.rb
送信件数: 200 / 二重送信: 0 人
```

「`close` された Queue は、残りの仕事を全部さばいたあと、pop に nil を返すようになる。nil の数を数えなくていいぶん安全ね」

## 2

「この設計には名前がついてる。**プロデューサー・コンシューマー**パターン。積む人（プロデューサー）と、こなす人（コンシューマー）を、窓口で完全に切り離す。両者はお互いを知らない。知ってるのは窓口だけ。──そして、この形には隠れた美点がもうひとつあるの。積む側が速すぎたら、どうなると思う？」

「Queue に仕事が積み上がっていく……メモリを食いますね。2 万件ならいいけど、これが 1000 万件だったら」

「そのための **SizedQueue**。窓口に定員をつける」

```ruby:backpressure.rb
queue    = SizedQueue.new(10)   # 窓口には 10 件しか積めない
max_seen = 0

worker = Thread.new do
  while (user = queue.pop)
    max_seen = [max_seen, queue.size].max
    sleep 0.001                 # 処理はゆっくり
  end
end

(1..100).each { |n| queue.push("user_#{n}") }  # 積む側は全力
queue.close
worker.join

puts "積み上がりの最大: #{max_seen} 件（上限 10 を超えない）"
```

```
$ ruby backpressure.rb
積み上がりの最大: 9 件（上限 10 を超えない）
```

「積む側は 100 件を全力で push してるのに、積み上がりは最大 9 件。……push が待たされてるんだ」

「そう。満杯の窓口への push は、**空きが出るまで自動的に待つ**の。これを**背圧**（バックプレッシャー）という。処理が遅いなら、入り口で受け取りも遅くする。速い側が遅い側を押し潰さないための、流量の安全弁。ジョブキューやストリーム処理では必須の考え方よ」

「……Sidekiq って、これの大きいやつですよね。Redis が窓口で、ワーカーがコンシューマーで」

「その通り。Sidekiq も、GitHub Actions のジョブも、レジ待ちの行列も、全部この形。**窓口 1 つと、それを取り囲む働き手**。今日 200 人のミニチュアで覚えた構造は、そのまま世界の大規模システムの構造よ」

## 3

「仕上げに、修正 PR の最終形をミニチュアで通しで動かすわよ。送信と、あなたが第 7 話でこじらせた**帳簿**の、両方の正解を入れて」

```ruby:digest_fixed.rb
require "logger"

logger  = Logger.new("digest.log")
users   = (1..200).map { |n| "user_#{n}" }
jobs    = Queue.new
results = Queue.new

# コンシューマー: 送信ワーカー 4 人
workers = 4.times.map do |w|
  Thread.new do
    while (user = jobs.pop)
      sleep 0.001                       # 送信（IO）
      results.push(user)                # 結果も Queue に積む
    end
  end
end

# 集計係: 帳簿は 1 人だけが書く
tally = Thread.new do
  count = 0
  while (user = results.pop)
    count += 1
    logger.info("#{count} 件送信済み") if count % 50 == 0
  end
  count
end

logger.info("daily_digest 開始 対象: #{users.size} 件")
users.each { |u| jobs.push(u) }
jobs.close
workers.each(&:join)

results.close
sent_count = tally.value                # 集計係の最終回答

logger.info("完了: 送信 #{sent_count} / 対象 #{users.size}")
puts "レポート: 送信 #{sent_count} / 対象 #{users.size}"
```

```
$ ruby digest_fixed.rb
レポート: 送信 200 / 対象 200
$ tail -2 digest.log
I, [2026-07-04T10:08:43.768841 #27222]  INFO -- : 200 件送信済み
I, [2026-07-04T10:08:43.768931 #27222]  INFO -- : 完了: 送信 200 / 対象 200
```

「読み解いて」

「Queue が 2 本。`jobs` は仕事の窓口で、`results` は**結果の窓口**。ワーカーは帳簿に触らず、送れた結果を results に積むだけ。カウントするのは集計係のスレッド 1 人だけ──あ、そうか。**`count += 1` してるのが 1 人なら、3 手に割り込む他人がいない**。Mutex なしで帳簿が正しい」

「そういうこと。『共有カウンタを鍵で守る』んじゃなくて、『**カウンタを触る人を 1 人にする**』。第 7 話の counter_lock も log_lock も、この形なら最初から生まれない。ロガーが Logger なのも意図的よ──第 2 話でやったとおり、自前 File のバッファ事故を構造で避ける。`tally.value` は join して戻り値を受け取る便利メソッド。第 5 話の教訓（join は例外の受け取り口）もここに含まれてる」

「fork は……もうどこにもいないんですね」

「今回の仕事には要らなかった、というだけよ。fork が悪いんじゃない。**分ける道具と共有する道具を、仕事の性質で選べるようになった**──それがこの二週間の成果。さ、報告会よ」

「犯人は、PR #1247 です。──ただし」

僕は続けた。ここからが、氷室さんと決めた本当の結論だ。

「この PR は、岸田さんひとりの過失ではありません。IO バウンドだからスレッド化する、という方針は正しかった。速度目標も達成した。問題は共有状態の扱いという 1 点だけです。そしてその 1 点を、レビューで通したのは僕です。**並行処理のコードは、書いた人とレビューした人の共犯でしか壊れません**。今後は再発防止として──」

修正 PR の要点と、レビューのチェックリストを映した。

1. 送信ワーカーは **Queue ベース**に書き換え（共有インデックスの廃止）。帳簿は**結果 Queue + 集計係 1 人**方式に
2. Slack レポートは fork をやめて **Thread** に（ゾンビの根絶）。どうしても fork する箇所は `wait` / `Process.detach` をレビュー必須項目に
3. ログは自前 File をやめ **Logger** に統一（バッファ事故の構造的排除）
4. CI に**並行処理の再現テスト**を追加──200 人のミニチュア世界を 100 回回して、重複・欠落・帳簿ズレがないことを検証する
5. レビュー観点の追加: 並行コードの diff では「**複数スレッドから触られる変数はどれか**」を最初に列挙する。`+=`・`||=`・「確認してから行動」の 3 パターンは要注意マーク。複数 Mutex は取得順序をコメントで明記

## 捜査メモ（最終）

- **Queue** = Ruby 標準のスレッドセーフな仕事の窓口。`push` / `pop` だけで、同じ仕事が 2 人に渡らないことを**構造で**保証する（実測: 200 / 200 / 二重 0 人、Mutex 不使用）
- 鍵が消えたのではなく、**鍵の管理をプロの部品に移譲した**。手書きの synchronize を減らし、検証済みの部品に共有を閉じ込めるのが基本姿勢
- 終業の合図: 働き手の人数ぶん `nil` を push、またはより安全な **`queue.close`**（実測: どちらも 200 / 0）
- **SizedQueue** は窓口に定員をつけ、満杯なら push を自動で待たせる = **背圧**（実測: 全力 push でも積み上がり最大 9 件）。速い側が遅い側を押し潰さない安全弁
- 帳簿の正解は「鍵で守る」より「**触る人を 1 人にする**」── 結果 Queue + 集計係 1 スレッド方式なら `count += 1` に割り込む他人がいない（実測: 送信 200 / 対象 200 で帳簿一致）
- プロデューサー・コンシューマーパターン: 積む人とこなす人を窓口で分離。Sidekiq などのジョブキューはこの構造の拡張
- レースコンディションは「手元で何万回通った」が無罪の証明にならない。**並行処理のテストは回数と揺らぎで攻め、レビューでは「複数スレッドから触られる変数」をまず列挙する**
- 総括: **fork = メモリを分ける**（安全・重い・要 wait）/ **Thread = メモリを共有する**（軽い・速い・全読み書きが現場になりうる）/ 共有するなら**共有の形を設計する**（Queue・単一書き手）。それでも残る共有にだけ、正しい鍵を

## 事件簿・完全解決一覧

| 怪事件 | 真相 | 学んだ概念 |
| --- | --- | --- |
| ② 開始ログが 2 行（同じ PID で） | fork がログバッファごと分身をコピー、親子が各々 flush。PID は fork 前に文字列に固定済み | fork とバッファ（第 2 話） |
| ③ ゾンビ 17 体（バッチ終了で消える） | レポート用 fork を誰も wait していない。親の死後は PID 1 が里親として処理 | ゾンビと wait（第 3 話） |
| ① メールが 2 通（十数人・飛び飛び） | IO 待ちで GVL を手放した隙に、別スレッドが同じ `i` を読んだ | レースコンディション（第 6 話） |
| ④ 帳簿が 257 不足（毎日違う数） | `sent_count += 1` の 3 手の間に割り込まれ、+1 が上書きで蒸発 | 同上（第 6 話） |
| （新)staging でバッチが固まる | 2 本の Mutex を逆順に取る密室。全スレッド停止時のみ Ruby が検出 | デッドロック（第 7 話） |

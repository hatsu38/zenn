## 1

「推理小説にはね、禁じ手すれすれの古典トリックがあるの。**双子トリック**。目撃者が見た犯人と、アリバイのある容疑者。実は双子でした、っていうあれ」

火曜の朝。氷室さんは僕の隣にノート PC を置いて言った。

「`fork` はそれを合法的にやる仕組みよ。打って」

```ruby:fork1.rb
puts "分裂前: PID #{Process.pid}"

child_pid = fork

if child_pid.nil?
  puts "子: 私の PID は #{Process.pid}、親は #{Process.ppid}"
else
  puts "親: 私の PID は #{Process.pid}、子は #{child_pid}"
end
```

```
$ ruby fork1.rb
分裂前: PID 92919
親: 私の PID は 92919、子は 92920
子: 私の PID は 92920、親は 92919
```

出力を三度見した。

「……あの、変です。`puts` が 3 回実行されてます。コードには if の片側しか通らないはずなのに、**親と子、両方の行が出てる**」

「そこが fork の全てよ。`fork` を呼んだ瞬間、OS はそのプロセスの**完全なコピー**を作る。メモリの中身も、変数も、開いてるファイルも、**いま何行目を実行中か**という情報さえも同じ分身。そして親と子の**両方**が、fork の次の行から実行を続ける」

「1 つのプログラムなのに、途中から 2 人になる……。呼び出しは 1 回なのに、**帰ってくるのは 2 回**、ってことですか」

「その表現、いいわね。fork は『1 回呼ぶと 2 回返ってくる関数』とよく言われる。そして 2 回の戻り値は違う値になる。ここは初見だと必ず混乱するから、ゆっくりね」

- **親**にとっての `fork` の戻り値 = 生まれた**子の PID**（92920）
- **子**にとっての戻り値 = **nil**

「双子が生まれた瞬間、それぞれの手に紙が握らされるの。親の紙には子の名前が書いてある。子の紙は白紙。**自分の紙を見れば、自分が親か子か分かる**。だからさっきの if 文が定番の書き方になる」

「なんで子は白紙なんですか？　親の PID を書いてくれればいいのに」

「子は `Process.ppid` でいつでも親を調べられるからよ。逆に親は、産んだ子の PID をその場で受け取らないと、あとから『さっき産んだ子はどれ？』を知る確実な手段がない。必要な情報だけ渡す、合理的な設計なの。──で、その子の親 PID を見て。92919。つまり fork した本人」

「昨日の証拠 B と同じ形だ。ゾンビ 17 体の PPID が、ぜんぶバッチ本体の 41001 でした」

「そう。**バッチの中に fork がいる**ことは、家系図がもう証明してる。PR #1247 の diff にも `fork` の文字がある。ここまでは固い。──残ってるのは証拠 A。『**同じ指紋の START が 2 つ**』。fork で分身ができるのは分かった。でも、START を書いたのは fork より*前*。しかも 2 行とも親の PID、41001 だった。分身が書いたなら、分身の PID が残るはずでしょう？」

「そうなんですよ。だから僕、fork が犯人だとしても、あの 2 行だけは説明がつかない気がして」

「──それが今日の本題。**双子トリックの核心**よ」

## 2

氷室さんは新しいファイルを作らせた。

「本番と同じ構造のミニチュアね。ログファイルを開いて、START を書いて、それから fork する」

```ruby:fork2.rb
log = File.open("batch.log", "w")
log.puts "[START] daily_digest 開始"   # fork の前に 1 回だけ書いた行

pid = fork do
  log.puts "[child] 送信ワーカー起動"
end

Process.waitpid(pid)
log.puts "[parent] 全件完了"
log.close
```

「今度はブロック付きの fork ね。`fork do ... end` と書くと、**子はブロックの中だけを実行して自動で死ぬ**。親はブロックに入らず次の行へ進む。さっきの if 分岐より安全で読みやすいから、実務はだいたいこっち。`Process.waitpid` は『子の終了を待つ』──詳しくは明日やる。いいから実行して、ログを見て」

```
$ ruby fork2.rb
$ cat batch.log
[START] daily_digest 開始
[child] 送信ワーカー起動
[START] daily_digest 開始
[parent] 全件完了
```

鳥肌が立った。

「**出た。`[START]` が 2 行**。……うちの本番とまったく同じだ。でも、なんで？　`log.puts` は fork の前に 1 回しか呼んでないのに」

「呼んだのは 1 回よ。でも、**書かれた場所**を考えて。`log.puts` した瞬間、その文字列はディスクに直行しない。プロセスのメモリの中にある**バッファ**という荷物置き場に、いったん積まれるの。ディスクへの書き込みは 1 回 1 回が遅いから、ある程度溜めてからまとめて書く。これは IO を速くするための、ごく普通の仕組み。Ruby が悪いんじゃなくて、世界中のプログラムがやってる最適化よ」

「バッファはメモリの中にある。メモリの中にあるということは──」

「そう。**fork はメモリを丸ごとコピーする**。荷物置き場も、積んであった `[START]` ごと」

頭の中で、事件が再生された。

1. 親、`[START]` を書く。ただし実際にはまだ**バッファの中**
2. 親、fork する。子は「バッファに `[START]` が積まれた状態」ごとコピーされて生まれる
3. 子が終了するとき、自分のバッファをディスクに flush する → **`[START]` + 子のログ**が書かれる
4. 親も終了時に自分のバッファを flush する → **もう 1 回 `[START]`** + 親のログが書かれる

「書いたのは 1 回。でも flush したのは 2 人。だから 2 行。──出力の*順番*も見て。子の START が先、親の START が後でしょ。子は fork のすぐ後に死んで flush したけど、親は全件送り終わるまで生きてたから、flush が最後になった。ログの並び順まで、この筋書きと一致する」

## 3

「じゃあ、指紋の謎に決着をつけるわよ。本番と同じように、**ログの各行に PID を刻む**バージョン」

```ruby:fork3.rb
log = File.open("batch.log", "w")
log.puts "[#{Process.pid}] [START] daily_digest 開始"

pid = fork do
  log.puts "[#{Process.pid}] [INFO] レポート送信"
end

Process.waitpid(pid)
log.puts "[#{Process.pid}] [INFO] 1000 件送信済み"
log.close
```

「予想して。2 回出るはずの START、それぞれの PID は？」

「1 回目は親の PID。2 回目は……子が flush するんだから、子の PID？」

```
$ ruby fork3.rb
$ cat batch.log
[1693] [START] daily_digest 開始
[1694] [INFO] レポート送信
[1693] [START] daily_digest 開始
[1693] [INFO] 1000 件送信済み
```

「──**両方 1693**。親の PID だ。子が書いた INFO の行はちゃんと 1694 なのに、START だけ 2 行とも親……あ。**そうか、そういうことか**」

声が出た。

「`"[#{Process.pid}] [START]..."` の文字列が組み立てられたのは、**fork の前**なんだ。その時点で PID の部分は 1693 という文字に*固まってる*。子がコピーで受け継いだのは、**完成品の文字列が入ったバッファ**。子は自分の PID を書いたんじゃなくて、**親の指紋が押された紙を、そのまま運んで捨てただけ**──」

「それが双子トリックの完成形よ」氷室さんは満足そうに頷いた。「双子はね、**生まれる前に押された親の指紋ごと、記憶を持ち出す**の。だから現場には『同じプロセスが 2 回開始した』という、ありえない証拠が残る。──証拠 A、これにて解決。うちの本番の 2 行が両方 41001 だったのは、fork 犯行説の*反証*じゃない。むしろ**動かぬ証拠**だったのよ」

「防ぐには？」

「2 通りある。ひとつは、fork の前に荷物置き場を空にしておく。`log.flush` を 1 行入れて、もう一度」

```ruby
log.puts "[START] daily_digest 開始"
log.flush                             # fork の前にバッファを吐き出す

pid = fork do
  # ...
```

```
$ ruby fork2.rb && cat batch.log
[START] daily_digest 開始
[child] 送信ワーカー起動
[parent] 全件完了
```

「消えた。1 行になりました」

「もうひとつは、そもそも溜めない。`sync = true` でバッファリング自体を切る方法」

```ruby
log = File.open("batch.log", "w")
log.sync = true                       # 書いた瞬間にディスクへ
log.puts "[START] daily_digest 開始"

pid = fork { log.puts "[child] 送信ワーカー起動" }
Process.waitpid(pid)
log.puts "[parent] 全件完了"
log.close
```

```
$ ruby sync.rb && cat batch.log
[START] daily_digest 開始
[child] 送信ワーカー起動
[parent] 全件完了
```

「これも 1 行。ログは事件の証言台なんだから、多少 IO 効率を落としてでも正直に書かせる価値がある。Ruby 標準の Logger も内部の IO は sync を有効にしてる。うちのバッチが素の `File.open` で自前ログを書いてたのは、そういう意味でも筋が悪かったの。──ちなみに `puts` で標準出力に書く分には、手元の Ruby では fork のタイミングで flush が済んでいて二重にならない（fork1.rb で分裂前の行が 1 回しか出てないでしょ）。**危ないのは自前で開いたファイル**」

## 4

「fork の使い道の話もしておくわよ。分身の使い道は 2 つある。ひとつは、うちのバッチみたいに**同じ仕事を手分けする**ため。もうひとつが──**別人に変身する**ため」

「変身？」

```ruby:henshin.rb
pid = fork do
  exec("ps", "-o", "pid,command", "-p", Process.pid.to_s)
end
Process.waitpid(pid)
puts "親: 子(#{pid})の変身を見届けた"
```

```
$ ruby henshin.rb
  PID COMMAND
26102 ps -o pid,command -p 26102
$ 親: 子(26102)の変身を見届けた
```

「`exec` は、**自分のプロセスの中身を、別のプログラムで丸ごと入れ替える**。出力を見て。PID 26102 の COMMAND が、Ruby じゃなくて `ps` そのものになってる。ruby として生まれた子が、PID はそのままに、ps に変身して、ps として死んだの」

「fork して、exec する……あ。これ、昨日の `system` の正体ですか」

「ご名答。`system("ps")` は中で **fork → 子が exec で ps に変身 → 親が wait** をやってる。そしてね、もっと言うと──あなたが毎日ターミナルで打ってるコマンド、あれ全部これよ。zsh が fork して、子が ruby に変身して、zsh は wait で待つ。昨日の家系図で『zsh の子が ruby』だったのは、この仕組みが動いた痕跡。**Unix の世界でプロセスが生まれる方法は、突き詰めればこの fork ただひとつ**。PID 1 から始まって、fork の連鎖が家系図を作ってる」

世界の見え方が少し変わった気がした。ターミナルに打つ一行一行が、fork だったのか。

## 5

「これで怪事件②は完全解決ですね。……で、そもそも PR #1247 は何のために fork してたんでしたっけ」

diff を開いた。該当箇所はすぐ見つかった。1,000 通送るごとに、Slack へ進捗レポートを投げる処理。コメントにはこうある。

```ruby
# Slack API が遅くても送信を止めないよう、レポートは別プロセスで投げる
fork do
  notify_slack("#{sent_count} 件送信済み")
end
```

「レポートのたびに fork……2 万通で 1,000 通ごとだから、毎朝 20 回」

「毎朝 20 回、双子の弟が生まれてる。で、深町くん。この子たち、**Slack に投稿し終わったあと、どうなると思う？**」

「ブロックが終わったら死ぬ、って今日教わりました」

「死ぬわね。じゃあ、死んだら消える？」

「……え？　消えるでしょう、普通」

氷室さんは、PPID がすべて 41001 だった、あの `<defunct>` 17 体のスナップショットを画面に出した。

「これが『普通に消えた』ように見える？──明日は**検死**よ。プロセスがどう死に、なぜ死体が残るのかをやる。fork2.rb にしれっと書いた `Process.waitpid` の意味も、そこで分かる。それと昨日予告した **PID 1 の『大事な役目』**もね」

（第3話「消えない死体 ── ゾンビプロセスと wait」につづく）

## 捜査メモ

- **fork = 実行中のプロセスの完全な分身を作る**。メモリ・変数・開いたファイル・実行位置ごとコピーされ、親子の両方が fork の次から実行を続ける（「1 回呼ぶと 2 回返ってくる関数」）
- 戻り値で自分の立場が分かる: **親には子の PID、子には nil**。`fork do ... end` なら子はブロックだけ実行して自動終了（実務はこちらが安全）
- ファイル書き込みは即ディスクに行かず、**プロセスのメモリ内バッファ**に溜まる。fork は**バッファごと**コピーする → 親子それぞれが flush して**同じ行が 2 回書かれる**（怪事件②の真相）
- 二重化した行の PID が**両方とも親のもの**になるのは、文字列が fork 前に組み立て済みだから（実測: `[1693]` が 2 回、子自身の行だけ `[1694]`）──「同じ指紋が 2 つ」の種明かし
- 対策は 2 つ: **fork の前に `flush`**、または **`sync = true`** でバッファリングを切る（ログは正直さ優先。標準の Logger は sync 済み）
- `exec` は自プロセスを**別のプログラムに変身**させる。`system` = fork + exec + wait。シェルがコマンドを実行する仕組みも同じ──**プロセスが生まれる方法は fork ただひとつ**
- PR #1247 は進捗レポートのために**毎朝 20 回 fork** していた → ゾンビ 17 体との関連へ（次話）
